# HRL (Halo Race Leaderboard) — Agent Handoff

Orientation doc for any AI agent (Claude Code, Codex, or otherwise) working in this repo. This file is intentionally short — it tells you what the project is and where to find the details. **This "Current state" section is the project's living handover doc — update it, and `docs/roadmap.md`, at the end of every task, not just at the end of a session.** Also keep everything else in `docs/` current: when you make an architectural decision, resolve an open item, hit and fix a real bug, or change the plan, update the relevant doc before finishing — don't just report it in chat and let the docs go stale. See [docs/decisions.md](docs/decisions.md) for the standing format (chronological, includes reversals and incidents, not just current-state facts).

## What this is

Rebuilding the frontend and API for a public Halo racing leaderboard (HRL). Existing app: https://github.com/EffakT/HRL-Web-Interface, live at https://hrl.effakt.info. The **database is preserved** (real prod data imported into dev), the **application is reinstalled fresh** on Laravel 13 + Livewire + Tailwind. Full detail: [docs/scope.md](docs/scope.md).

## Current state (at a glance)

**Phase 1 (frontend design) is complete. Phase 2 (backend/real data) is now complete too** — every page is wired to real data: Servers List (`ServerList`), Server Single (`ServerShow`: Maps, Stats Card, Latest Laps, and Top Players/Server Score), the nested Server Map Leaderboard (`ServerMapLeaderboard`), Maps List (`MapList`), the global Map Leaderboard (`MapLeaderboard`, real pagination for ranks 4+ via `HasRankedLeaderboardPagination`), Players List (`PlayerList`, a real Global Leaderboard), Player Single (`PlayerShow`, real Global Rank/Score/Points and real split comparison), and Home (every highlight block is real: Quick Stats, Live Stats Snapshot, New Maps/Servers, Fastest Improvements, Player Achievements, Most Active Server, and Latest/Current Records — none left mock or empty-by-necessity). All ranked leaderboards share real gaps, server/player identity, and split comparison via `LapTimeSplit::compare()`. **A public, read-only, versioned API now exists too** (`/api/v1/servers`, `/api/v1/maps/{map}/leaderboard`, `/api/v1/laps/{lapTime}` — no auth, rate-limited, see [docs/api.md](docs/api.md)), **the lap-submission webhook is rebuilt and live** (`POST /api/v1/laps`, now with SEC-01 HRL query verification — see below), **and server cards now show genuinely live current-map/online status** (roadmap item 19) rather than only lap-history-derived proxies. **Real-time updates are wired up too** (roadmap item 16) — Reverb/Echo, live-updating the nested/global map leaderboards on a new PB, and Servers List's header stats/"MOST ACTIVE" card and every Home highlight on *any* logged attempt. **A server-scoped player profile now exists too** (roadmap item 20, see below) — `/servers/{serverId}/players/{playerId}`, reached from Server Single's Top Players list. Test suite: 125 Pest tests / 308 assertions (2026-07-07), PHPStan clean, Rector configured (fixes not yet applied). See [docs/roadmap.md](docs/roadmap.md) for the full plan, and [docs/database.md](docs/database.md) for real data quirks the wiring surfaced: `lap_times`' timestamp precision, duplicate pivot rows (root cause now confirmed and fixed at the write site), archived-server exclusion, and sparse split coverage (~4%).

**Just done (2026-07-07)**: a site audit (`SITE_AUDIT.md`) flagged the open lap-submission webhook as critical (SEC-01) — anyone could POST fabricated leaderboard data. Rather than the audit's suggested TLS/HMAC relay (real infrastructure outside this repo, on every game-server host), `App\Helpers\LapSubmissionVerifier` cross-checks every submission against a live UDP `\query` response from the submitting ip:port, reusing the protocol already built for item 19 — the Lua script publishes `hrl_enabled`/`hrl_protocol`/`hrl_token` via SAPP's existing `query_add`, and verification requires that plus a matching map and an online player name. **`config('webhook.hrl_query.enforce')` defaults to `false`** (log-only) since real game servers' Lua scripts don't send these fields yet — flip it once every active server's script is confirmed updated, or legitimate submissions will be rejected.

**Same-day follow-up**: a second review of that implementation (findings recorded directly in `SITE_AUDIT.md`) caught 5 real gaps, all fixed the same day: (1) the webhook rate limiter's `ip:port` key was bypassable by rotating the caller-supplied port — now two limits apply together, the existing per-`ip:port` one plus a coarser per-IP ceiling; (2) the idempotency guard reserved its cache key *before* processing, so a pre-commit failure left retries permanently stuck and a legitimate retry got a bare `409` instead of its original result — reworked into a real `processing`/`done` state machine that replays the original response, plus an optional client-supplied `submission_id`; (3) `LapSubmissionVerifier` now accepts an `hrl_token_prev` field for token-rotation grace; (4) removed a duplicate UDP round-trip — `ProcessNewLap` now reuses the verifier's already-fetched query response instead of querying the same ip:port again; (5) tightened online-player matching from "any `player_`-prefixed key" to `/^player_\d+$/`. 117 → 125 tests, 286 → 308 assertions. Full detail in [docs/security.md](docs/security.md) and [docs/decisions.md](docs/decisions.md).

**Just done (2026-07-06)**: shipped a server-scoped player profile (roadmap item 20), on request — clicking a player from Server Single's Top Players list previously landed on the global Player Single page, losing server context. Extracted `App\Models\PlayerProfile::build()` from `PlayerShow::mount()` — the second real consumer of the same "assemble a player's per-map/recent-lap/stats breakdown" logic — parameterized by an optional `$serverId`. Caught a real regression from the extraction itself: `PlayerShow`'s original inline code tolerated `GlobalRanking::forPlayer()` returning `null` (PHP's forgiving `null['key'] ?? default`), but the extracted method's tightened `array $ranking` signature turned that into a real `TypeError` — caught immediately by the existing `RoutesTest.php` smoke suite, fixed by widening to `?array`. Added `ServerPlayerShowTest.php` (4 tests) and extended `ServerShowTest.php`/`RoutesTest.php`. Full detail in [docs/decisions.md](docs/decisions.md) and [docs/player-single.md](docs/player-single.md).

Earlier the same day: wired up Reverb/Echo for real-time leaderboard updates (roadmap item 16). `App\Events\LeaderboardUpdated` (firing since item 14, previously going nowhere) now broadcasts on two public channels — `servers.{serverId}.maps.{mapId}` and a new map-only `maps.{mapId}` — and `ServerMapLeaderboard`/`MapLeaderboard` both listen via Livewire's `#[On('echo-public:...')]` attribute, re-fetching their ranking on receipt (simpler and more obviously correct than patching one new lap into an already-ranked array). Verified end-to-end with a standalone `pusher-js` script that bypasses Echo/Livewire entirely, connecting directly to the local Reverb server and confirming the real WebSocket payload arrives on a real PB submission — this also surfaced two genuine operational requirements now documented in [deployment.md](docs/deployment.md): broadcasting is queued (needs a running queue worker, not just Reverb — a submitted PB silently went nowhere until one was started) and this dev environment's default Reverb port (8080) was already taken by something else on the shared host (moved to 8081). `ServerMapLeaderboard` had zero test coverage before this — added `ServerMapLeaderboardTest.php` alongside the item 16 test coverage.

**Immediate follow-up, same day**: user asked whether the "Most Active" card updates live, then — once confirmed it didn't (real data, but page-load-only) — asked to extend live updates "everywhere it's visible." New `App\Events\LapSubmitted` fires on *every* logged attempt (not just a PB) on a site-wide `activity` channel, since Servers List's header stats/"MOST ACTIVE" Activity Score and Home's highlights change on any attempt anywhere, not one map. `ServerList`/`Home`'s `mount()` bodies became `loadServers()`/`loadHighlights()`, reused by a matching `#[On('echo-public:activity,lap.submitted')]` listener each. Verified the same way (standalone `pusher-js` script + a real webhook submission). Also: made a real mistake mid-task — ran `git checkout --` on a file with the user's own uncommitted in-progress edit (`PLAYERS_PER_PAGE` 15→10) while isolating an unrelated test failure, silently discarding it; caught immediately and restored by hand. Earlier the same day: shipped live server info (roadmap item 19). Queried the actual production server (id 7, `hrl.effakt.info` — genuinely live and reachable) to empirically confirm the `QueryServer` protocol's real response fields rather than guessing: `mapname` matches `maps.name`'s format exactly, `numplayers` gives a live player count, and the `sapp` field confirms this is a Halo PC server running the SAPP mod. New scheduled command `App\Console\Commands\RefreshLiveServerInfo` (every minute) stores the result on new `servers` columns (`current_map_id`, `live_player_count`, `queried_at`, `query_successful`) via a real migration run against the dev DB. `ServerList` prefers fresh (≤5 min) live data for "online"/"now playing," treats a fresh *failed* query as authoritative for "offline" specifically, and falls back to the pre-existing lap-history proxies once data goes stale — a failed query never wipes previously-known good values. Never fabricates a `Map` row from an unrecognized live `mapname` (same principle as the webhook). Also fixed a real, pre-existing Larastan gap along the way: it doesn't resolve a custom `casts(): array` Carbon cast for non-standard column names in this project — fixed with `@property` docblock annotations on `Server` (the standard mechanism, not a suppression).

## Ground rules

- Nothing in this app is a like-for-like port of the old site by default — where the redesign's UX diverges (e.g. the server-scoped vs. global leaderboard split), the new design wins. Check [docs/decisions.md](docs/decisions.md) before assuming old-site behavior is the target.
- The whole frontend currently runs on mock data by deliberate sequencing choice, not oversight. Don't "fix" this unprompted — wiring real data is a planned, sequenced step (see [docs/roadmap.md](docs/roadmap.md)).
- Read [docs/livewire-guide.md](docs/livewire-guide.md) before writing or editing any Livewire component — several non-obvious framework gotchas are already solved there; don't rediscover them.
- Follow [docs/coding-standards.md](docs/coding-standards.md) for project-specific conventions not covered by the general Laravel Boost guidelines (see `CLAUDE.md`).
