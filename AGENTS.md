# HRL (Halo Race Leaderboard) — Agent Handoff

Orientation doc for any AI agent (Claude Code, Codex, or otherwise) working in this repo. This file is intentionally short — it tells you what the project is and where to find the details. **This "Current state" section is the project's living handover doc — update it, and `docs/roadmap.md`, at the end of every task, not just at the end of a session.** Also keep everything else in `docs/` current: when you make an architectural decision, resolve an open item, hit and fix a real bug, or change the plan, update the relevant doc before finishing — don't just report it in chat and let the docs go stale. See [docs/decisions.md](docs/decisions.md) for the standing format (chronological, includes reversals and incidents, not just current-state facts).

## What this is

Rebuilding the frontend and API for a public Halo racing leaderboard (HRL). Existing app: https://github.com/EffakT/HRL-Web-Interface, live at https://hrl.effakt.info. The **database is preserved** (real prod data imported into dev), the **application is reinstalled fresh** on Laravel 13 + Livewire + Tailwind. Full detail: [docs/scope.md](docs/scope.md).

## Current state (at a glance)

**Phase 1 (frontend design) is complete. Phase 2 (backend/real data) is now complete too** — every page is wired to real data: Servers List (`ServerList`), Server Single (`ServerShow`: Maps, Stats Card, Latest Laps, and Top Players/Server Score), the nested Server Map Leaderboard (`ServerMapLeaderboard`), Maps List (`MapList`), the global Map Leaderboard (`MapLeaderboard`, real pagination for ranks 4+ via `HasRankedLeaderboardPagination`), Players List (`PlayerList`, a real Global Leaderboard), Player Single (`PlayerShow`, real Global Rank/Score/Points and real split comparison), and Home (every highlight block is real: Quick Stats, Live Stats Snapshot, New Maps/Servers, Fastest Improvements, Player Achievements, Most Active Server, and Latest/Current Records — none left mock or empty-by-necessity). All ranked leaderboards share real gaps, server/player identity, and split comparison via `LapTimeSplit::compare()`. **A public, read-only, versioned API now exists too** (`/api/v1/servers`, `/api/v1/maps/{map}/leaderboard`, `/api/v1/laps/{lapTime}` — no auth, rate-limited, see [docs/api.md](docs/api.md)), **the lap-submission webhook is rebuilt and live** (`POST /api/v1/laps`), **and server cards now show genuinely live current-map/online status** (roadmap item 19, see below) rather than only lap-history-derived proxies. Test suite: 86 Pest tests / 229 assertions (2026-07-06), PHPStan clean, Rector configured (fixes not yet applied). See [docs/roadmap.md](docs/roadmap.md) for the full plan, and [docs/database.md](docs/database.md) for real data quirks the wiring surfaced: `lap_times`' timestamp precision, duplicate pivot rows (root cause now confirmed and fixed at the write site), archived-server exclusion, and sparse split coverage (~4%).

**Just done (2026-07-06)**: shipped live server info (roadmap item 19). Queried the actual production server (id 7, `hrl.effakt.info` — genuinely live and reachable) to empirically confirm the `QueryServer` protocol's real response fields rather than guessing: `mapname` matches `maps.name`'s format exactly, `numplayers` gives a live player count, and the `sapp` field confirms this is a Halo PC server running the SAPP mod. New scheduled command `App\Console\Commands\RefreshLiveServerInfo` (every minute) stores the result on new `servers` columns (`current_map_id`, `live_player_count`, `queried_at`, `query_successful`) via a real migration run against the dev DB. `ServerList` prefers fresh (≤5 min) live data for "online"/"now playing," treats a fresh *failed* query as authoritative for "offline" specifically, and falls back to the pre-existing lap-history proxies once data goes stale — a failed query never wipes previously-known good values. Never fabricates a `Map` row from an unrecognized live `mapname` (same principle as the webhook). Also fixed a real, pre-existing Larastan gap along the way: it doesn't resolve a custom `casts(): array` Carbon cast for non-standard column names in this project — fixed with `@property` docblock annotations on `Server` (the standard mechanism, not a suppression). Added `RefreshLiveServerInfoTest.php` (4 tests) and extended `ServerListTest.php` (+3). 79→86 tests, 208→229 assertions. Full detail in [docs/decisions.md](docs/decisions.md) and [docs/database.md](docs/database.md).

Earlier the same day: rebuilt the lap-submission webhook end-to-end (roadmap item 14) — `POST /api/v1/laps` → `LapSubmissionController` → `ProcessNewLap`. Recovered the previously-missing `app/Helpers/QueryServer.php-legacy` helper, confirming the UDP `QueryServer` protocol (a GameSpy-style `\query` request) that items 14 and 19 were both blocked on; rebuilt it with real key/value parsing instead of the legacy's fragile fixed-array-offset reads. Deliberate behavioral changes beyond the old app, all explicit decisions not silent ports: logs every lap attempt (not just PB improvements), a failed live query no longer discards the lap, the duplicate-pivot-row bug is fixed at the write site (`syncWithoutDetaching`), and a new `LeaderboardUpdated` broadcast event exists (ready for item 16, not yet consumed). **No auth** — surfaced to the user explicitly before building; decided to keep it open, matching the old app. Also caught and fixed a real SQLite/PDO bug along the way: a bound parameter compared against an aggregate in `WHERE`/`HAVING` is silently ignored by this environment's bundled SQLite 3.34.1 (confirmed via raw PDO outside Laravel) — fixed by computing the leaderboard-position comparison in PHP instead of SQL. ## Ground rules

- Nothing in this app is a like-for-like port of the old site by default — where the redesign's UX diverges (e.g. the server-scoped vs. global leaderboard split), the new design wins. Check [docs/decisions.md](docs/decisions.md) before assuming old-site behavior is the target.
- The whole frontend currently runs on mock data by deliberate sequencing choice, not oversight. Don't "fix" this unprompted — wiring real data is a planned, sequenced step (see [docs/roadmap.md](docs/roadmap.md)).
- Read [docs/livewire-guide.md](docs/livewire-guide.md) before writing or editing any Livewire component — several non-obvious framework gotchas are already solved there; don't rediscover them.
- Follow [docs/coding-standards.md](docs/coding-standards.md) for project-specific conventions not covered by the general Laravel Boost guidelines (see `CLAUDE.md`).
