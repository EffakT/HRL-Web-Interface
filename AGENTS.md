# HRL (Halo Race Leaderboard) — Agent Handoff

Orientation doc for any AI agent (Claude Code, Codex, or otherwise) working in this repo. This file is intentionally short — it tells you what the project is and where to find the details. **This "Current state" section is the project's living handover doc — update it, and `docs/roadmap.md`, at the end of every task, not just at the end of a session.** Also keep everything else in `docs/` current: when you make an architectural decision, resolve an open item, hit and fix a real bug, or change the plan, update the relevant doc before finishing — don't just report it in chat and let the docs go stale. See [docs/decisions.md](docs/decisions.md) for the standing format (chronological, includes reversals and incidents, not just current-state facts).

## What this is

Rebuilding the frontend and API for a public Halo racing leaderboard (HRL). Existing app: https://github.com/EffakT/HRL-Web-Interface, currently live in production. The **database is preserved** (real prod data imported into dev), the **application is reinstalled fresh** on Laravel 13 + Livewire + Tailwind. Full detail: [docs/scope.md](docs/scope.md).

## Current state (at a glance)

**Just done (2026-07-08)**: PERF-01 (Home page recomputation) fully closed, in three passes. Pass 1 measured and deduped: `\DB::listen()` confirmed **132 queries/~2.3s** at real scale; `fastestImprovements()`/`achievements()` each independently queried the same recent-laps window and both re-ran `GlobalRanking::scores()`'s full every-lap computation once per recent lap just to read one player's row. Fixed in `Home.php` by computing the shared recent-laps query and one "current" `scores()` snapshot once per request — 132 → 94 queries, ~2.3s → ~1.5s. Re-audited same day, correctly called "partially improved, not resolved" (DB time was only ~170ms of that remaining ~1.5s — every visitor between two lap submissions was still paying the same PHP-side cost independently). Pass 2: asked the user to pick caching scope (inside `GlobalRanking` app-wide vs. `Home` only, a real architectural decision) — **user chose Home-only** — then cached the entire `$highlights`/`$quickStats` output as one blob, invalidated by a new `App\Listeners\InvalidateHomeHighlightsCache` on the already-firing `LapSubmitted` event, with a 10-minute TTL as a safety net only. Pass 3 (code review, same day): the pass-2 caching itself had two real bugs — a cache **stampede** (`Cache::remember()` has no locking, so every visitor rebuilds at once right after invalidation) and a more serious **stale-write race** (a rebuild already in flight when a new lap invalidates the cache would finish afterward and silently write stale data back, undoing the invalidation for up to the full TTL) — plus a factual error in the listener's docblock (`ProcessNewLap` isn't queued at all; it deliberately runs synchronously in the webhook's own request). Fixed with a generation key (bumped on invalidation instead of deleting the data key directly, so a stale in-flight rebuild can only write to an abandoned key) plus a `Cache::lock()` with a cache recheck on acquisition. Caught two more real things along the way: `Cache::increment()` on the `database` cache store returns `false` and does nothing if the key was never set (needs an explicit `Cache::add(..., 0, null)` first); and Pest's `CACHE_STORE=array` persists across the whole test process, so one test's cached value could leak into another's assertions (fixed globally via `Cache::flush()` in `tests/Pest.php`'s `beforeEach()`). Result: cache hit = 1–2 queries/<1ms; live TTFB held at ~37–53ms throughout, clearing both the under-20-queries and under-500ms targets with large margin. Full detail in [docs/decisions.md](docs/decisions.md) and [docs/performance.md](docs/performance.md).

**Earlier (2026-07-08)**: SEC-02's "Not verified" gap turned out to be real, then the first fix for it shipped a real open redirect, caught the same day by the user re-testing the live site. Round one: `curl -sSI http://redesign.hrl.effakt.info/` returned a plain `200 OK` instead of a redirect — root-caused (temporary debug route, added/curled/removed immediately) to FastPanel's nginx unconditionally setting `$_SERVER['HTTPS']="on"` regardless of the real connection, so `RedirectIfNotSecure` never fired. Fixed by trusting the (accurate) `X-Forwarded-Proto` header from Nginx Proxy Manager via `$middleware->trustProxies(at: '*', headers: FOR|HOST|PORT|PROTO)`. Round two: the user immediately found that trusting `HOST`/`PORT` too made this a live, working **open redirect** — `X-Forwarded-Host: evil.example` + `X-Forwarded-Port: 444` produced `Location: https://evil.example:444/...`. Fixed by dropping `HOST`/`PORT` from the trusted mask (kept `FOR`/`PROTO` only — the edge overwrites forged versions of those two but passes `HOST`/`PORT` through as-is). Also removed `AddHstsHeader` entirely (deleted the class) per explicit instruction: it's host-wide, so it was quietly threatening to force HTTPS on `/api/v1/*` too, which must stay plain-HTTP for legacy Halo game-server clients — this should have been the call when SEC-02 first shipped. Verified live after both rounds: spoofed-header requests now redirect to the real domain, the normal redirect and `/api/v1/*` plain-HTTP access are unaffected, and HSTS is no longer sent. Full detail in [docs/decisions.md](docs/decisions.md) and [docs/security.md](docs/security.md)'s SEC-02 section.

**Just done (2026-07-07)**: a site audit (`SITE_AUDIT.md`) flagged the open lap-submission webhook as critical (SEC-01) — anyone could POST fabricated leaderboard data. Rather than the audit's suggested TLS/HMAC relay (real infrastructure outside this repo, on every game-server host), `App\Helpers\LapSubmissionVerifier` cross-checks every submission against a live UDP `\query` response from the submitting ip:port, reusing the protocol already built for item 19 — the Lua script publishes `hrl_enabled`/`hrl_protocol`/`hrl_token` via SAPP's existing `query_add`, and verification requires that plus a matching map and an online player name. **`config('webhook.hrl_query.enforce')` defaults to `false`** (log-only) since real game servers' Lua scripts don't send these fields yet — flip it once every active server's script is confirmed updated, or legitimate submissions will be rejected.

**Same-day follow-up**: a second review of that implementation (findings recorded directly in `SITE_AUDIT.md`) caught 5 real gaps, all fixed the same day: (1) the webhook rate limiter's `ip:port` key was bypassable by rotating the caller-supplied port — now two limits apply together, the existing per-`ip:port` one plus a coarser per-IP ceiling; (2) the idempotency guard reserved its cache key *before* processing, so a pre-commit failure left retries permanently stuck and a legitimate retry got a bare `409` instead of its original result — reworked into a real `processing`/`done` state machine that replays the original response, plus an optional client-supplied `submission_id`; (3) `LapSubmissionVerifier` now accepts an `hrl_token_prev` field for token-rotation grace; (4) removed a duplicate UDP round-trip — `ProcessNewLap` now reuses the verifier's already-fetched query response instead of querying the same ip:port again; (5) tightened online-player matching from "any `player_`-prefixed key" to `/^player_\d+$/`. 117 → 125 tests, 286 → 308 assertions. Full detail in [docs/security.md](docs/security.md) and [docs/decisions.md](docs/decisions.md).

**Second same-day follow-up**: a third review found 5 more gaps. Two involved genuinely open-ended design decisions (exact rate-limit numbers; how much idempotency infrastructure to build) — surfaced to the user via AskUserQuestion rather than decided solo, per a new standing rule (`feedback_audit_scope_discipline` in memory: ask before implementing anything an audit phrases open-endedly). **Third same-day follow-up**: a fourth review verified the staging deployment directly and found 4 more gaps. Two unambiguous, done directly: revoke the verified rate-limit marker immediately on any failed verification (was only expiring naturally after 5 min); a reused `submission_id` with genuinely different content now returns `409 idempotency_conflict` (stores a content hash alongside the cached response) and `submission_id` is `required` once enforcement is on. The other — no unique `(ip, port)` constraint on `servers`, so `Server::firstOrCreate()` itself wasn't race-proof — took three real attempts (recorded in the migration's docblock and [docs/decisions.md](docs/decisions.md)): a plain `unique(ip, port)` blocks reusing an archived server's ip:port forever; `unique(ip, port, deleted_at)` silently never fires because SQL treats `NULL != NULL`; a generated column referencing the row's own `id` is flatly rejected by MySQL — **which is this app's real dev/prod database, not just the SQLite the Pest suite uses** (a fact now saved to memory, discovered the hard way here). Shipped: a generated `active_since` column collapsing every active row to one sentinel value while soft-deleted rows keep their real `deleted_at`, so active duplicates still collide but archived ones and legitimate reuse don't; `ProcessNewLap` now retries the whole transaction once on a `UniqueConstraintViolationException` its `violatesSubmissionIdUniqueness()` helper doesn't attribute to the lap-level constraint (replacing an earlier, over-broad generic-SQLSTATE check). Given the constraint's complexity grew well past "just add a unique index," this was also confirmed with the user before building, per the same standing ask-first rule. 129 → 133 tests, 321 → 331 assertions. Full detail in [docs/security.md](docs/security.md) and [docs/decisions.md](docs/decisions.md).

**Fourth same-day follow-up**: a fifth review inspected the deployed MySQL schema directly (confirmed the generated `active_since` column, its unique index, no duplicate active identities, archived-address reuse working) and found 3 more gaps, all narrow enough to fix directly without going back to the user. ## Ground rules

- Nothing in this app is a like-for-like port of the old site by default — where the redesign's UX diverges (e.g. the server-scoped vs. global leaderboard split), the new design wins. Check [docs/decisions.md](docs/decisions.md) before assuming old-site behavior is the target.
- The whole frontend currently runs on mock data by deliberate sequencing choice, not oversight. Don't "fix" this unprompted — wiring real data is a planned, sequenced step (see [docs/roadmap.md](docs/roadmap.md)).
- Read [docs/livewire-guide.md](docs/livewire-guide.md) before writing or editing any Livewire component — several non-obvious framework gotchas are already solved there; don't rediscover them.
- Follow [docs/coding-standards.md](docs/coding-standards.md) for project-specific conventions not covered by the general Laravel Boost guidelines (see `CLAUDE.md`).
