# Coding Standards

For general Laravel/PHP/Pest conventions, defer to the Laravel Boost-provided guidelines (bottom of `CLAUDE.md`, owned by Boost — don't hand-edit that block) and the `laravel-best-practices`/`pest-testing` skills. This doc covers **project-specific** conventions established while building this app that aren't covered by those.

## Livewire components

- **Traditional two-file format only**: `app/Livewire/{Namespace}/{Name}.php` + `resources/views/livewire/{namespace}/{kebab-name}.blade.php`. Do **not** use `artisan make:livewire` as-is — it scaffolds Livewire v4's default single-file-component (`⚡`-prefixed) style, which this project deliberately doesn't use. Create files by hand instead.
- Full-page Livewire components (route-bound via `Route::get('/x', Component::class)`) must render **exactly one root element with no `<!DOCTYPE html>`**. Use the `#[Layout('components.layout', [...])]` attribute for the page shell, never embed `<x-layout>` manually inside the component's own view. See [livewire-guide.md](livewire-guide.md).
- Never name a route parameter the same as an unrelated public property on the bound component (e.g. `{map}` colliding with `public string $map`) — Livewire auto-hydrates matching property names from route params before `mount()` runs, silently overwriting them.
- Any modal/popup toggled by a Livewire property (not a route) must have a **stable, non-dynamic `wire:key`** if you want Alpine `x-transition` to actually animate — a key that changes per-render (e.g. keyed to the selected index) causes Livewire's morph to recreate the DOM node instead of patching it, which prevents Alpine from seeing the enter transition.

## Extracting shared logic

Extract a trait/partial **the moment a second genuinely-duplicate component exists**, not preemptively and not "eventually." `App\Livewire\Concerns\HasLapDetailModal` and the two `resources/views/livewire/partials/*.blade.php` files are the working example — extracted as soon as the server-scoped and global leaderboards needed identical open/close/comparison behavior, not before.

Conversely: don't force an abstraction for genuinely small, non-reused logic. `PlayerShow`'s lap-detail popup initially used its own tiny `openLap()`/`closeLap()` pair instead of the shared trait, on the reasonable assumption it'd stay simpler than the leaderboard modal. Once the scope actually converged (split comparison was added to it too), it was switched to reuse the trait. Don't over-extract in anticipation of convergence that hasn't happened yet.

**Lap Detail modal, second variant extracted**: `resources/views/livewire/partials/lap-vs-record-modal.blade.php` — shows a single lap in isolation against its map's course record ("vs MAP RECORD · {holder} {time}"), with a "VIEW MAP LEADERBOARD" link. Extracted from Player Single's inline copy once Server Single's Latest Laps needed the identical thing (second real consumer). Distinct from the older `lap-detail-modal.blade.php`, which shows a driver's **position within one map's ranked leaderboard** (rank, gap-to-leader, "VIEW PLAYER") — don't conflate the two; a page showing one ranked list of drivers on one map wants the leaderboard variant, a page showing laps that span multiple different maps/players (a feed) wants the vs-record variant. Both share the same `HasLapDetailModal` trait (`$selectedPlayerIndex`/`openLap()`/`closeLap()`/`$this->comparison`). `ServerShow`, `ServerMapLeaderboard`, and `MapLeaderboard` override the trait's `getComparisonProperty()` with a real one, all delegating to `LapTimeSplit::compare(int $lapId, int $referenceLapId): array`. `PlayerShow` keeps the mock fallback until its lap data is wired. Only ~4% of real laps have split rows recorded, so an empty comparison is normal and renders an explicit empty state.

**`LapTimeSplit::compare()` conventions, all load-bearing** (see docs/decisions.md for the bugs each one fixes):
- Returns `[]` only when the *selected* lap has no splits. If the selected lap has splits but the reference lap doesn't, it still returns rows — with `hasReference: false` and null delta/running/barW — so the caller can show raw split times without a comparison, rather than nothing. Both modal partials check `hasReference` per-row and show an explanatory note when every row lacks it.
- `barW` is scaled to a max of **50, never 100** — the delta bar is drawn from a center line (`left-1/2`/`right-1/2`) out to one edge, which is only half the track's width. A `barW` of 100 overflows the track by 50%. This applies to the mock fallback in `HasLapDetailModal` too — its values were rescaled to the corrected 0–50 range for the same reason.
- Every row carries both a formatted `delta` string (`"+0.041"`, display only) and a signed numeric `deltaValue` (float, for sorting/math) — **never sort or compare on the formatted string**, it's not numeric-safe (`"+0.041"` vs `"−0.028"` sort lexicographically, not by magnitude). The "biggest gain/loss sector" line in `lap-detail-modal.blade.php` sorts on `deltaValue`.
- Callers must not call `compare($lapId, $lapId)` (same id twice) — it returns `[]`. When the item being viewed *is* the reference point (e.g. viewing the #1 driver's own lap, or a lap that already is its map's course record), pick a different reference instead (e.g. #2's lap, or that map's runner-up lap) rather than showing nothing. Both `ServerMapLeaderboard` and `ServerShow` do this.
- **Whenever the reference lap can vary (the #1-vs-#2 fallback above), expose a `getComparisonReferenceProperty()` returning `['name', 'time', 'label']` and have the modal partial read it — never hardcode the displayed "vs X" name/label to a fixed driver.** `ServerMapLeaderboard`, `MapLeaderboard`, and `ServerShow` each share their reference resolver between comparison data and display metadata so the two cannot drift. `PlayerShow`'s mock vs-record partial falls back to its lap's mock record fields.

**Podium extracted**: `resources/views/livewire/partials/podium.blade.php` is the shared "3 clipped cards, green/cyan/gold, 1st bigger and centered" visual, used by Server Single's Top Players section and Players List's top-3 (see [server-single.md](server-single.md), [players-list.md](players-list.md)). It takes a generic `$podium` array (`title`/`subtitle`/`value`/`meta`/`badge`/`href` per position) with no click behavior or sparkline. Each card renders as an `<a>` when `href` is set, a plain `<div>` otherwise — **don't build this with a dynamic tag name via `{{ }}` interpolation** (e.g. `<{{ $isLink ? 'a href=...' : 'div' }}>`); Blade's `{{ }}` HTML-escapes its output, so the `<`/`>`/quotes in a hand-built tag string get escaped into literal text instead of a real tag. Use a real `@if/@else` with two full, separately-written blocks instead — more repetition, but the only version that actually renders as HTML. The original Map Leaderboard podium (`leaderboard-podium-and-table.blade.php`) intentionally still has its **own** podium markup rather than being rewired onto the shared partial — it needs lap-detail-modal click handling and the split-time sparkline on 1st place, which the generic partial doesn't support and shouldn't be forced to; the two are kept visually identical by convention (copy the class list, don't invent new styling), not by sharing one file. Homepage's Most Active Server block deliberately stays on its own compact list style, confirmed by the user — not a future candidate for the shared partial (see [decisions.md](decisions.md)).

## Stat displays

- **Simple, small stat counts** (a handful of standalone all-time totals) use a badge-pill style: `<span class="border border-hud-green/16 bg-hud-green/4 px-2.5 py-1.5 font-mono text-[10px] font-semibold tracking-[0.12em] text-hud-text-muted">{{ value }} LABEL</span>` in a `flex flex-wrap gap-2.5` row. **Two established variants, not interchangeable**:
  - **Homepage's Quick Stats**: under its own `// QUICK STATS` section header, positioned after the intro paragraph — left exactly as-is (confirmed explicitly: "Don't touch homepage, as was is correct"). Don't move or re-header it to match the other variant below.
  - **Server Single's Stats Card**: no section header, placed directly below the page's `<h1>` (above the tagline paragraph), badges without a fill background and `tracking-[0.14em]` — matching the Map Leaderboard's title badges exactly (`// ALL SERVERS · GLOBAL` → title → badges). Use **this** variant for new pages' simple stat rows unless told otherwise; it was the corrected/preferred styling once the user saw both side by side.

  Reserve the bigger `hud-clip` bordered-card-with-large-numbers treatment (e.g. Homepage's Live Stats Snapshot highlight block) for stats that are paired/windowed (30d/90d, today/week) rather than simple single totals — that's a different kind of display, not the same pattern at a different size.

## Tailwind v4

- **Never build class names via string interpolation** (`text-{{ $variable }}`) — Tailwind's JIT scanner can't detect them and silently produces no CSS. Pass whole literal class name strings from PHP instead (e.g. `'colorClass' => 'text-hud-red'`). This applies to `app/**/*.php`, not just Blade files — Tailwind v4's automatic content detection scans the whole non-gitignored project, so literal strings anywhere are fine, string concatenation anywhere is not.
- **Theme values that appear in more than one place must be theme tokens** (`@theme` block in `resources/css/app.css`), not independently hardcoded hex/gradient values in multiple Blade files. If two elements need "the same background," they should both reference the same `--color-hud-*` custom property, not two copies of the same hex string — otherwise they drift the moment one gets edited.
- Be aware `.hud-clip`/`.hud-clip-sm` (`clip-path`) **clip all descendant overflow**, not just literal content — this bit the sparkline hover tooltips (see [decisions.md](decisions.md)). Prefer Alpine's Anchor plugin (`x-anchor`) for anything that needs to escape a clipped or otherwise bounded container, over hand-rolled edge-case positioning classes.
- **Use the project's custom breakpoint scale** (`mm`/`ml`/`tp`/`tl`/`d`/`dlg`/`dxlg`/`dxxlg`) — Tailwind's defaults (`sm`/`md`/`lg`/`xl`/`2xl`) are fully disabled project-wide. Never write `sm:`/`md:`/etc.; they compile to nothing. See [architecture.md](architecture.md#breakpoints) for the full scale and current usage (`tp:` is the de facto desktop/mobile split point).
- **Never use arbitrary hardcoded pixel breakpoints** (`max-[550px]:`, `min-[900px]:`, etc.) — always snap to the nearest named breakpoint instead (e.g. `max-[550px]:` → `max-ml:` since `ml` is 500px, the closest defined value; `max-[360px]:` → `max-mm:` since `mm` is 375px). Tailwind auto-generates `max-{name}:` variants for every named breakpoint in the scale, so there's no reason to reach for an arbitrary value. If a genuinely new fine-tuning point keeps coming up, add it to the named scale in `resources/css/app.css` rather than inlining more one-off pixel values.

## Date/time display

- **Show relative time ("3 days ago"), not an absolute date/time, as the primary display for any real timestamp** — an absolute time is ambiguous to an international audience with no visible timezone context (is "01 Jul 2026, 14:00" local to the viewer, the server, or UTC?). Use Carbon's `diffForHumans()`. Established on Server Single's Latest Laps feed, per explicit user feedback; apply to any other real (non-mock) date display as those pages get wired up (Player Single's Recent Laps/Last Active, Players List's Active column, etc.).
- **Pair every relative-time display with a hover tooltip showing the exact date/time + timezone abbreviation** (e.g. "01 Jul 2026, 00:00 UTC" via `->format('d M Y, H:i').' '.->format('T')`), so the precise moment is still available for anyone who needs it. Use the same Alpine Anchor tooltip pattern already established for the split-pace sparkline hover (`x-anchor.top.offset.6="$el.parentElement"`, `group`/`group-hover:block` on a `pointer-events-none` positioned span) rather than inventing a new tooltip mechanism — see `server-show.blade.php`'s Latest Laps section for the reference implementation (both desktop and mobile variants).
  - **`x-anchor`'s `$el.parentElement` must be a tight-fitting wrapper around just the hovered text, not a wide layout container** — if the `group relative` element anchoring the tooltip is a direct grid/flex child (or otherwise stretches to fill its column), it gets sized to the full column width, and the tooltip centers over that whole width instead of the actual (often right-aligned) text. Fix: nest an `inline-block` `group relative` span around *just* the text inside the wider cell/container, and anchor to that inner span, not the outer one. This bit the Latest Laps date tooltip — it initially anchored to the full 170px grid column instead of the right-aligned date text sitting at its edge.

## Blade markup

- **`<button>` elements don't get a pointer cursor by default** (unlike `<a>` tags) — add `cursor-pointer` explicitly on any clickable `<button>`. Pair with `disabled:cursor-not-allowed` when the button can be disabled (e.g. pagination Prev/Next) — Tailwind's `disabled:` variant correctly overrides the base `cursor-pointer` when the `disabled` attribute is present.
- **Double-check the boolean direction on `@disabled(...)` for Prev/Next-style controls** — `@disabled($paginator->onFirstPage())` (disable Previous when already on page 1) is easy to accidentally write inverted as `@disabled(! $paginator->onFirstPage())` (disables Previous on every page *except* the first — backwards). This exact bug shipped once on Server Single's Latest Laps pagination; the underlying `WithPagination`/`previousPage()` wiring was never broken, only the button's disabled condition. If Prev/Next ever looks broken, verify with a direct `Livewire::test()->call('previousPage', 'page')` before assuming the pagination framework itself is at fault — check the disabled-condition boolean direction first.
- **Any "status dot" badge (a small `rounded-full` circle next to all-caps mono text, e.g. the nav's "LIVE" indicator, the featured server's "ONLINE · {region}" badge) needs `-mt-px` on the dot**, even though the wrapper is `inline-flex items-center`. Flexbox's `items-center` centers the dot geometrically against the text's line-height box, but all-caps text has no descenders, so its *visual* weight sits above true center — a geometrically-centered dot reads as slightly low next to it. `-mt-px` nudges the dot up by 1px to correct for this. Apply to any new dot-indicator badge from the start.
- **The `◆` glyph used to lead badge text** (e.g. "◆ MOST ACTIVE", "◆ COURSE RECORD", "◆ TOP SCORE", "◆ #1 GLOBAL") reads as slightly low next to the all-caps text next to it, for the same font-metrics reason as the status-dot badges above. Fix: wrap just the glyph in its own `<span class="-translate-y-px inline-block">◆</span>`, not the whole badge string. Because of this, **badge label strings in mock data/PHP should never bake the `◆` into the string itself** (e.g. `'badge' => 'TOP SCORE'`, not `'◆ TOP SCORE'`) — the glyph + alignment fix belongs in the Blade template that renders the badge (see `podium.blade.php`'s 1st-place badge), not in the data. This also means the fix only has to exist in one place (the template) rather than needing `{!! !!}`-unescaped HTML smuggled through data strings.
- **Any row using the `border-l-2 border-l-transparent ... hover:border-l-hud-green` hover-accent pattern must have horizontal padding (`px-3.5`) on the row itself**, not just the row's inner content. Without it, the row's text sits flush against the left edge with no gap from the hover accent line — this was missed on every mobile-list row across the app (Map List, Player List, Server List, Server Single's Maps table, the Map Leaderboard's rest-table, Player Single's Recent Laps) until caught on Player Single's Recent Laps and fixed everywhere at once. **Always do this** — when adding a new clickable/linked list row with this hover pattern, include `px-3.5` from the start; don't rely on catching it later.
- A bullet `<li>` using `flex` with mixed content (bullet + plain text + an inline `<a>`) breaks natural text wrapping — the `<a>` becomes its own flex item instead of flowing inline with the surrounding text. Fix: wrap the text+link portion in one nested `<span>` so the flex row only ever has two items (bullet, content-span). Applied on the opt-in and contact pages' bullet lists — use this pattern for any future list item containing an inline link or other element.

## Mock data (current phase)

The whole frontend currently runs on mock/sample data by explicit decision — no models, no real queries. Every component with mock data should:
- Comment it clearly as mock with a `// TODO: replace with real ... query once backend integration is wired up` note.
- Prefer numbers/shapes that plausibly resemble real data (e.g. matching design-comp sample values) over arbitrary placeholders, so the UI reads correctly during review.
- Not be treated as a spec to preserve — once real queries are wired up, mock data is deleted, not kept as a fallback.
