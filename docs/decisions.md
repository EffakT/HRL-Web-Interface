# Decisions

Chronological record of architecture decisions, reversals, and incidents — the "why" behind choices in the other docs. Newest at the bottom of each section's history isn't tracked; entries are grouped by topic instead.

## Players leaderboard uses event-based Last Move, not calendar trend

**Decision (2026-07-13):** replace the unresolved Trend concept with **Last Move**. HRL can go months without a new lap, so daily/weekly/monthly snapshots would usually compare identical states and provide little value. A recent-activity proxy was also rejected: submitting a lap or setting a PB does not necessarily move rank, while an inactive player can be displaced by somebody else.

After a one-time neutral baseline, accepted laps will compare the real before/after global ranking and retain each player's last non-zero movement (`▲ n`, `▼ n`, or `NEW`) plus when it happened. Players displaced by somebody else's improvement receive a real downward move. Unchanged subsequent events do not erase the last meaningful move. The label deliberately says Last Move rather than Trend because one event is not a sustained trajectory.

Initial scope is the global Players leaderboard only. The implementation must be database-serialized and idempotent under concurrent/retried lap processing, bulk-read on the page, and safe if movement data is absent. Full implementation plan: [player-rank-movement.md](player-rank-movement.md).

## Fresh reinstall, not incremental upgrade

Old app: Laravel 6.2, PHP 7.2, both EOL. Decided against incremental upgrade — fresh install on Laravel 13.8 (planned as "12" originally), keeping the existing database. Migrations are additive/conditional against the real schema, not naive fresh `create` migrations.

## Full lap history, not personal-best-only

**Originally planned** (before the real schema was inspected): a `lap_records` table upserted on personal best (old value overwritten), plus a separate lightweight `lap_events` table for activity tracking, plus `lap_record_splits` keyed by `sector_number`.

**Reality after inspecting the schema**: none of those tables exist. The real `lap_times` table is already append-only full history (every lap ever submitted), and `lap_time_splits` keys by `checkpoint_id` with richer `start_time`/`end_time`/`duration` fields.

**Decision**: keep full history as the real schema already does. Personal bests and course records become derived `MIN(time)` queries, not stored rows. This also means `lap_times`' own timestamp doubles as the activity/recency signal — no separate `lap_events` table needed. See [database.md](database.md).

## Claim-code ownership system: deferred

`users_players`/`users_servers` (claim-code based ownership of player/server profiles) exist in the real schema and are a genuine old-app feature, not legacy cruft. Decision: preserve the tables as-is, but no new feature work is planned around them until explicitly revisited. Don't design new features assuming ownership/claiming is wired up.

## Server-scoped vs. global leaderboards

**Originally built**: a single flat `/maps/{mapId}` route, reached directly from the Servers list, showing one leaderboard.

**User clarification**: the intended hierarchy has two distinct concepts —
- A **nested, server-scoped leaderboard**: Servers → a server's own map list → that server+map's leaderboard.
- A **global leaderboard**: a top-level Maps list (like Players) → a map's leaderboard aggregated across all servers.

**Decision**: split into `Servers\ServerMapLeaderboard` (nested) and `Maps\MapLeaderboard` (global), each reached via its own route hierarchy (`/servers/{id}/maps/{id}` vs. `/maps/{id}`). This is a deliberate addition beyond the old site, which never had a per-server map list route (`/servers/{id}` 404s there) — new UX for the redesign, not a parity port. The two components share their ranked-table/modal logic via `HasLapDetailModal` + Blade partials rather than duplicating ~150 lines twice.

## Variable split/sector count (up to 14, not always 3)

Some maps have up to 14 splits/sectors, not the fixed 3 the initial design assumed.

**Decided**:
- **Podium card**: replaced the 3-box split-time grid with a compact multi-segment sparkline bar (width per segment proportional to split time, green/gold color-coded vs. the lap's own average) — scales to any split count without adding rows. Individual split times moved off the podium entirely; a hover tooltip shows them instead.
- **Lap Detail modal**: capped panel height (`max-h-[85vh]`), made the split-rows area independently scrollable with a sticky column header, keeping the lap-time header and final-delta/buttons footer pinned.
- Mock data updated from 3 to 14 entries specifically to exercise this, not just build-to-spec unverified.

## Incident: fatal error on every Livewire click (missing `cache` table)

`.env` has `CACHE_STORE=database` (works fine for `sessions`/`jobs` since those tables exist in the real imported schema), but the `cache`/`cache_locks` tables — a Laravel 11+ convention — never existed in the old app's schema. Every `wire:click` goes through Livewire's checksum/rate-limiter, which reads the cache store, so clicking any interactive element fatal-errored: `SQLSTATE[42S02]: Base table or view not found: 1146 Table 'redesign_hrl.cache' doesn't exist`.

**Fix**: ran only the cache-table migration (`php artisan migrate --path=database/migrations/0001_01_01_000001_create_cache_table.php`), not a full `php artisan migrate` — the stock `users`/`jobs` migrations in the same scaffold would conflict with the already-existing real tables of those names.

## Incident: Lap Detail modal transition didn't play

First implementation used a PHP `@if` to conditionally render the modal — this meant every open/close was an instant DOM add/remove with no way for Alpine to animate a transition. Switched to Alpine `x-show` driven by `$wire.selectedDriverIndex !== null`, keeping the element always in the DOM.

That alone wasn't enough — the modal still opened instantly with no transition. Root cause: the element had no stable `wire:key`, so Livewire's DOM morph was recreating the node on every click instead of patching it in place, meaning Alpine saw a "new" element already in its final state with nothing to animate from. Fix: added a static `wire:key="lap-detail-modal"` (not keyed to the selected index). See [livewire-guide.md](livewire-guide.md) for the general pattern.

## Incident: sparkline tooltips clipped on first/last segments

Tooltips were centered under each sparkline segment via `left-1/2 -translate-x-1/2`. On the first/last segments (near the podium card's edges), the tooltip got visually clipped. Root cause: the podium card uses `.hud-clip` (`clip-path`), and `clip-path` clips **all** descendant overflow, not just elements with literal `overflow: hidden` — including absolutely-positioned tooltips extending past the card's box.

First fix attempt: manually special-case `left-0`/`right-0` for first/last segments — worked, but doesn't generalize to other clipped containers.

**Final decision**: adopted Alpine's official Anchor plugin (`@alpinejs/anchor`, wraps Floating UI) for real dynamic positioning with automatic overflow avoidance, registered via the `alpine:init` event since Livewire bundles its own Alpine instance (no standalone `Alpine.start()` in this app). See [livewire-guide.md](livewire-guide.md) for the usage pattern — this is now the standard approach for any future tooltip/popover.

## Sticky header background: match, don't replicate

The Lap Detail modal's sticky split-rows column header needed a background matching the surrounding gradient panel. First attempt hardcoded a duplicate copy of the panel's gradient values directly on the sticky element — this is fragile (two places to update if the panel's colors ever change) and also didn't look right (a locally-reproduced gradient isn't the same as a color that's actually part of the same continuous background). **Decision**: added `--color-hud-modal-start`/`--color-hud-modal-end` as real theme tokens in `resources/css/app.css`, referenced by both the panel and the sticky header — then further reverted the sticky header specifically to a **flat** color (`bg-hud-modal-start`, no gradient) per explicit follow-up, since a two-stop gradient repeated on a thin sticky bar wasn't actually what "match the surrounding" meant. Lesson: when two elements need "the same" background, both should reference one shared token — never duplicate a hex/gradient definition.

## Player lap popup: from "simpler" to full split comparison

First pass deliberately built the player lap-detail popup **without** a split comparison table, reasoning that "vs a record holder" doesn't naturally apply to a player's own personal lap log. It used its own standalone `$selectedLapIndex`/`openLap()`/`closeLap()` rather than the shared trait.

**Reversed** on explicit request: "show the splits against the map records in this instance." Once the scope actually matched the leaderboard modal's, switched to reuse `HasLapDetailModal` (rather than build a third near-identical mock-comparison implementation) and added `recordHolder`/`recordTag`/`recordTime` fields to the mock lap data to drive a "vs MAP RECORD" reference line.

Lesson: don't force premature reuse when scope looks different — but don't resist reuse once scope actually converges, either.

## Custom breakpoint scale replaces Tailwind defaults

Decision: fully replace Tailwind's default breakpoint scale (`sm`/`md`/`lg`/`xl`/`2xl`) with a project-specific named scale (`mm` 375px, `ml` 500px, `tp` 735px, `tl` 1020px, `d` 1270px, `dlg` 1440px, `dxlg` 1800px, `dxxlg` 2160px), via `--breakpoint-*: initial;` + redefinition in `resources/css/app.css`'s `@theme` block.

At the time of this decision, 52 existing usages of `sm:`/`md:`/`lg:` were already spread across 13 view files (every list/leaderboard page's desktop-vs-mobile table/list toggle, the nav's mobile overlay breakpoint, a couple of grid-column reflows). All were migrated onto `tp:` (735px), since they all served the same conceptual "desktop vs. mobile" split despite being defined at three different old breakpoint values (640/768/1024px) — consolidating onto one named breakpoint was judged more consistent than preserving the old values' exact pixel thresholds. See [architecture.md](architecture.md#breakpoints) for the full scale and [coding-standards.md](coding-standards.md) for the going-forward rule.

**Follow-up rule** (same session): two arbitrary pixel breakpoints already existed on the Servers featured-card layout (`max-[550px]:w-full`, `max-[360px]:w-full`, added just before the named scale existed). Decision: **never use hardcoded pixel breakpoints going forward** — snap to the nearest named value instead (`max-[550px]:` → `max-ml:` [500px], `max-[360px]:` → `max-mm:` [375px]), using Tailwind's auto-generated `max-{name}:` variants. If a genuinely new fine-tuning point recurs, extend the named scale rather than reaching for another one-off pixel value.

## Test/analysis tooling set up (Pest, PHPStan/Larastan, Rector, Semgrep)

Decision: get static analysis and testing tooling in place proactively, before real backend work starts, rather than bolting it on afterward.

- **Pest**: added the first real automated test (`tests/Feature/RoutesTest.php`, dataset-driven over every route), directly replacing the manual curl-and-grep verification process used throughout the earlier build (see the git history of this doc for how much of that there was). Left the stock Laravel installer example tests in place rather than deleting them, per the "don't delete tests without approval" convention — they're harmless boilerplate, not something worth a permission round-trip over.
- **PHPStan + Larastan**: installed at level 5 deliberately, not higher — the codebase is still frontend-on-mock-data, and the mock Livewire components return untyped arrays by design (see [coding-standards.md](coding-standards.md)'s mock-data section). A stricter level right now would mostly flag that intentional pattern rather than real problems. Passes clean at level 5 as of setup. Raise the level as real Eloquent models replace the mock arrays.
- **Rector**: configured with Laravel 13 + PHP 8.3 rule sets, `HasLapDetailModal` explicitly excluded (`withSkip`) since it leans on Livewire-specific dynamic property conventions Rector might "fix" incorrectly. Dry-run proposes 16 files' worth of changes (mostly `declare(strict_types=1)` + PHP 8.3 `#[Override]` attributes) — **not applied**, since "set the tool up" and "apply every suggested rewrite" are different asks; applying is a separate, deliberate follow-up action.
- **Semgrep**: this is the one tool that **could not actually be installed** in this sandbox. Semgrep ships only via pip, Homebrew, or Docker — this environment has no pip module (`ensurepip` itself is missing from the Python install), no passwordless sudo (so no `apt install python3-pip` either), no Homebrew, and no Docker. Checked GitHub releases directly too — Semgrep doesn't publish standalone binary assets there, only via PyPI. Wrote `.semgrep.yml` with a small set of custom rules anyway (unescaped Blade output, raw-SQL interpolation, hardcoded-credential-like assignments, a Livewire `mount()` validation reminder) so it's ready to run the moment someone has access to one of those install paths (e.g. CI). **These custom rules are unvalidated** — never actually run against real Semgrep here, so treat them as a first draft, not a proven ruleset.

Composer scripts added: `analyse` (PHPStan), `rector-dry`/`rector-fix`, `lint` (Pint + PHPStan), `check` (lint + rector-dry + test) — see [testing.md](testing.md).

## Global Player Ranking spec — ambiguity resolutions

The initial feature spec (as given) had a genuine overlap/contradiction and several unstated details that would each produce a different (and each individually reasonable) algorithm if guessed silently. Rather than pick silently, these were confirmed explicitly before writing [global-ranking.md](global-ranking.md):

- **Band boundary**: spec said "11–25" and "25–50" (25th place in both). Resolved to **11–25 and 26–50** — no overlap.
- **Interpolation shape**: spec just said "gradually decreases." Resolved to **linear within each band**, but with band 11–25 steeper than band 26–50 (continuing the pattern already visible in the explicit 1st–10th table, which itself eases from a 5-point gap near the top to a 2-point gap by 10th) — not a single curve across the whole 11–50 range, and not exponential/other non-linear shape.
- **Global Score definition**: not stated at all in the original spec. Resolved to **sum of per-map points**, not average — average would work against the stated "encourage participation across multiple maps" goal (a player who only plays one map perfectly could otherwise outscore a player who plays ten maps well).
- **Per-map lap-time ties**: not addressed in the original spec, but the overall tie-break rules referenced "earliest achievement date," implying recency/order already mattered somewhere. Resolved to **earliest lap wins the higher per-map position** (not shared/standard-competition-ranking positions), extending that same principle down to individual map rankings for consistency.
- **Leaderboard scope** (global vs. nested): not asked as a clarifying question — inferred from the spec's own stated purpose ("overall measure... across all race maps") to mean the **global, all-servers leaderboard** per map, not the server-scoped nested one. Flagged in the doc as an assumption to confirm, not a fully-resolved decision like the four above.

## Most Active Server spec — resolving formula-vs-goals tensions

The Most Active Server spec (as given) had the formula literally contradict two of its own stated design goals — worth confirming explicitly rather than writing a spec doc that documents its own internal contradiction. Written up in [most-active-server.md](most-active-server.md); resolutions:

- **"Valid Laps" counted raw, or capped?** The formula as given (`Valid Laps × 1`) would let one player inflate the score without bound by grinding the same map repeatedly — directly contradicting the stated goal "prevent one player repeatedly racing from dominating the score." Resolved: **"Valid Laps" now means distinct (player, map) participations**, not raw lap submissions — repeat laps on an already-played map add nothing further. Mirrors the same "count participation, not volume" principle [global-ranking.md](global-ranking.md) already uses (best lap per map, not every lap).
- **All-time cumulative or rolling window?** Spec didn't say, but an all-time cumulative score can only ever grow, which directly contradicts the stated goal "allow inactive servers to fall off the list over time." Resolved: **all three base metrics (Unique Players, Valid Laps, Maps Played) are computed over a rolling 90-day window**, not all-time — this is what actually makes "falling off" possible. The Recency Bonus (7/30/90-day tiers) stacks on top of this windowed base for extra emphasis on very recent activity. Noted in the doc: since the base window and the top recency-bonus tier are both 90 days, that top tier (+20) ends up applying to nearly any server with a nonzero base score — a minor redundancy, not fixed, just flagged for whoever tunes these numbers later.
- **Lap validity**: the schema has no validity/anti-cheat flag at all. Rather than invent a criterion, this was explicitly **deferred as a future problem** (per direction) — a map-file checksum to detect modded maps is one idea floated, though it wouldn't catch script-based cheats (hog-jumps, portals, etc.). Until that exists, every recorded lap is treated as valid for this algorithm.

This spec also **supersedes** an earlier, much simpler placeholder "most active server" definition that had been sitting in [database.md](database.md) since the initial schema-reconciliation pass (ranked by raw `lap_times` row count in a rolling 30-minute window) — that placeholder predates this full spec and is now retired in favor of it.

## Server Single page spec — docs-only by design, and reusing Global Ranking

Explicitly confirmed as a docs/planning task before writing anything, since the request read ambiguously between "spec this" and "build this now" — the previous two turns (Global Player Ranking, Most Active Server) were both pure planning, but this one described concrete page sections for an *already-built* real page (`ServerShow`), which could plausibly have meant "add these now, mock-data-first, like the rest of the frontend." Confirmed docs-only before proceeding, consistent with the session's stated framing. Written up in [server-single.md](server-single.md).

**"Top Players" reuses the Global Player Ranking algorithm**, not a new one — the request's own phrasing ("Based on our Global player ranking algo, we can add the following") was read as a direct instruction to reuse the formula, just scoped to one server's nested leaderboard positions instead of the global leaderboard, summed only across maps played on that server. Named **Server Score** to distinguish it from Global Score (same formula, different scope) — see [glossary.md](glossary.md). Implementation should keep the points table/interpolation logic in one shared place so Global Score and Server Score can't silently drift apart from duplicated formula code.

Two items were deliberately **left open rather than guessed**, since both have meaningfully different implementations depending on the answer:
- **"Number of records set"** — could mean "maps where this server currently holds the course record" (cheap, current-state query) or "how many times has a record been set here historically, even if since beaten" (expensive, requires point-in-time replay of full lap history). Both are legitimate readings of the phrase; picking one silently would bake in a potentially-wrong and expensive-to-reverse assumption.
- **"Average lap time"** (for Top Players) — raw average across every attempt vs. average of per-map personal bests. These produce different numbers with different meaning (practice-inclusive pace vs. skill ceiling).

## Players List redesign — validates existing design, surfaces one new hard question

Continued the same docs-only pattern as the previous three feature-spec turns without re-confirming — by this point in the session the pattern was well-established (two pure-algorithm specs plus one real-page spec, all docs-only), so re-asking would have been redundant rather than careful.

The request's own stated principle — "we don't want to just show raw lap counts, as grinders will dominate, it doesn't reflect skill, it dilutes the leaderboard's meaning... prioritize score, records, consistency, and activity" — is a direct, independent validation of a design choice [global-ranking.md](global-ranking.md) already made (position-based points, capped per map, rather than raw lap volume). Worth noting as confirmation, not just a new requirement: the earlier design reasoning holds up against a fresh, differently-worded statement of the same concern.

Two smaller things resolved along the way:
- **Global Player Ranking's own open "UI surface" question is now resolved**: `/players` becomes the Global Leaderboard. That open item in [global-ranking.md](global-ranking.md) is marked done.
- **New evidence on the still-open "number of records set" question** (first raised in [server-single.md](server-single.md)): this page's own framing ("make the page feel alive, instead of static") points toward the historical-events reading, since the current-state reading reduces to a mostly-static number system-wide. Recorded as a hint in [roadmap.md](roadmap.md), not treated as a final decision — the underlying question (cheap-and-static vs. expensive-and-dynamic) is still open and still worth a real answer rather than inferring one from page-copy tone alone.

One genuinely new hard question surfaced: the **Trending indicator** is the first planned feature in this project that plausibly can't stay within the "fully derivable from current data, no stored history" philosophy every other feature so far has followed ([database.md](database.md), [global-ranking.md](global-ranking.md), [most-active-server.md](most-active-server.md) are all explicitly designed to avoid this). A literal "did this player's rank move" comparison needs *some* prior state to compare against. Documented two paths (periodic snapshots vs. a recent-activity proxy) without picking one — this is a real architectural fork, not a copy-detail, and deserves a deliberate answer when it's actually built.

## Player Single expansion — reused definitions over new ones wherever possible

Continued the same docs-only pattern. This spec was deliberately written to **reuse already-established definitions rather than mint new ones** wherever the request's terms mapped onto something already decided:
- "Num Records" → the same current-state "Records" definition from [players-list.md](players-list.md), not a new one.
- "Top 3 finishes" → literally [global-ranking.md](global-ranking.md)'s tie-break rule 2, now surfaced as a visible stat instead of a hidden tiebreaker input. Worth noting this kind of reuse as a sign the earlier specs are holding together as a coherent system, not just individually plausible documents.
- "Maps completed" → same count as [players-list.md](players-list.md)'s "Maps with PBs," different label.

One new **terminology clash** was caught and flagged rather than silently creating ambiguity: "Total valid laps" here means a **plain raw lap count**, whereas "Valid Laps" on the Most Active Server page ([most-active-server.md](most-active-server.md)) means "distinct (player, map) participations" — a deliberate redefinition made there specifically to prevent gaming that particular score. There's no score being protected on this page, so the raw-count reading is correct here, but it's the same English phrase meaning two different things across two docs — recorded explicitly in the glossary so nobody assumes the two are interchangeable.

"Best Performance" ("top 3 laps or biggest achievements") was left as an explicit open item rather than picking one — the spec's own wording offered two genuinely different features (a trivial top-N query vs. an editorial "achievements" concept needing its own definition of "biggest"), and picking silently would have hidden that second option needs real design work before it's buildable.

This page's expansion also differs from the previous three specs in one respect: it builds around **existing, already-working code** (the lap log + `HasLapDetailModal` popup), not a wholly greenfield feature — noted explicitly in the doc so new sections get wired into that existing interaction pattern rather than duplicating it.

## Homepage: block-selection mechanism confirmed, one long-open question effectively resolved

Two things needed a real decision before this spec could be written coherently, both asked explicitly rather than assumed:

- **Block selection isn't just "hide empty ones"** — since most of the 6 candidate highlight blocks will almost always have *some* data (there's basically always a most-active server, always live stats), a pure empty/non-empty filter wouldn't naturally narrow anything to 3. Confirmed: a **fixed priority order** (Records > Most Active Server > Fastest Improvements > New Content > Achievements > Live Stats), take the first 3 that have something to show — not a computed per-load "interestingness" score, which would have needed its own scoring formula per block type.
- **"Live Stats Snapshot" vs. "Quick Stats" overlap** — confirmed these are genuinely different, not redundant: Quick Stats is simple all-time totals (always shown, outside the 3-block system), Live Stats Snapshot is windowed/recency-focused (30d/90d/today) and competes for one of the 3 highlight slots like the other 5 candidates.

**The long-open "number of records set" question (first raised in [server-single.md](server-single.md), hinted at again in [players-list.md](players-list.md)) is now effectively forced toward the historical reading**, not just hinted at: the homepage's "Latest / Current Records" block needs to show discrete recent record-setting events (map, player, time, server, "1h ago") — something the current-state reading (which only knows "who holds the record right now," not "when they took it") has no way to produce. This doesn't mean every instance of "records" across every doc must use the historical definition (per-page context still matters — see [players-list.md](players-list.md)'s note that its own "Records" column deliberately stays current-state), but at least one real feature now structurally requires the historical derivation to exist.

**A related but distinct question got clarified along the way**: "largest rank jump" and "new Top 10 entry" (Fastest Improvements block) sound like they need the same kind of historical tracking as the still-unresolved Trending indicator ([players-list.md](players-list.md)), but they don't — both are derivable by comparing current state to "current state minus the player's most recent qualifying lap," a stateless re-computation, not a stored point-in-time snapshot. The Trending indicator's problem is specifically wanting comparison against an arbitrary *past* point (e.g. "7 days ago"), which one-event-back comparison can't answer. Worth keeping these conceptually separate — "needs a before/after comparison" doesn't automatically mean "needs stored history."

Also added a **Product Pillars** concept to [scope.md](scope.md) (Map=Performance, Server=Community, Player=Mastery+Consistency) — this originated as homepage framing but clearly applies app-wide, so it was placed in the project's scope doc rather than left buried in a page-specific spec where later work wouldn't think to look for it.

## Roadmap re-prioritized: frontend design before real algorithms/data

After writing four page specs ([server-single.md](server-single.md), [players-list.md](players-list.md), [player-single.md](player-single.md), [homepage.md](homepage.md)) that all reference not-yet-built backend features (Global Player Ranking, Most Active Server scoring, historical record-events), the natural next question was sequencing: build the real algorithms first, or the page UIs first?

**Decided: frontend first, static/mock data throughout**, matching the exact sequencing already used for every other page in this project (Servers, Maps, Players were all built mock-first — see the early Progress Log history). There's no principled reason for these four specs to break that pattern just because their mock numbers happen to be described by a written algorithm doc instead of invented ad hoc like the earlier pages' sample data was. [roadmap.md](roadmap.md)'s "Next up" section was restructured into two explicit phases (frontend design, then backend/real data) rather than one interleaved list, so this ordering is visible at a glance rather than something you'd have to infer from reading every line.

One addition beyond a pure reorder: **Homepage's mock data should include mock "current state" inputs** (fake recent records, improvements, new content) specifically so the block-*selection* logic (which 3 of 6 highlight types show) can actually be built and reviewed — not just the visual design of whichever one block someone arbitrarily chose to mock. A page whose main feature is "pick the 3 most relevant things" needs varied mock inputs to meaningfully exercise that feature at all.

## Server Single page additions built (Phase 1, item 1)

All four planned additions from [server-single.md](server-single.md) were added to `ServerShow`/`server-show.blade.php`, in spec order (Stats Card, Top 3 Fastest Laps, Top Players, Maps unchanged, All Laps), all on mock data:

- **Top 3 Fastest Laps**: raw times across all maps on the server (not per-map), rank-colored green/cyan/gold to match the existing podium convention elsewhere, map name shown prominently since times aren't otherwise comparable across different maps.
- **Top Players (Server Score)**: reuses the green/cyan/gold rank convention; "Average lap time" uses the doc's default assumption (raw average across all attempts, not average-of-per-map-bests) — also still not formally decided.
- **All Laps**: this is the first table in the project built with **real pagination**, not a static mock footer like the earlier "‹ PREV · 1 2 3 · NEXT ›" placeholder on the Map Leaderboard partial. `ServerShow` uses Livewire's `WithPagination` trait for page-state management and manually constructs an `Illuminate\Pagination\LengthAwarePaginator` by slicing a 47-row mock array — there's no Eloquent model yet to paginate against directly, but the pagination *mechanism* (page query string, prev/next wiring, "showing X–Y / Z") is real and will keep working unchanged once `$this->allLaps` is replaced with a real paginated query. Verified by rendering `/servers/1?page=4` and confirming it correctly shows rows 46–47 of 47.

Neither of the two "not decided" definitional questions (records-set reading, average-lap-time reading) were resolved by this work — building the UI didn't require resolving them, only picking *a* consistent mock convention, which was the whole point of doing this frontend-first. ## Eloquent models built (Phase 2, item 10), surfaced a real data constraint

Built one model per real table — `Server`, `Map`, `Player`, `LapTime`, `LapTimeSplit`, `PlayerClaim`, `ServerClaim` — plus factories, no migrations (all tables already exist). Followed the project's existing Laravel 13 convention (seen in the pre-existing `User` model) of PHP attributes (`#[Fillable]`) and a `casts()` method rather than `protected $fillable`/`protected $casts` properties.

`users_players`/`users_servers` (the claim-code tables) were modeled as first-class models (`PlayerClaim`/`ServerClaim`) rather than forced into `belongsToMany` pivots — both tables have their own `id`, `claim_code`, `claimed_at`, and `deleted_at` (soft deletes), which is richer than a plain pivot table and awkward to express well through Eloquent's pivot API. `servers_maps`/`players_servers`, by contrast, are genuinely plain two-FK pivots with nothing beyond timestamps, so those stayed as ordinary `belongsToMany` relationships with no dedicated model.

Verified every model and relationship against the **real** `redesign_hrl` data (not factories) via Tinker: 3 servers, 10 maps, 817 players, 1657 lap times, 429 splits, 1 player claim, 1 server claim — all relationships (`Server::maps`, `Map::lapTimes`, `LapTime::with(['server','map','player','splits'])`, `PlayerClaim::with(['user','player'])`, etc.) resolved correctly.

**Real finding while verifying**: `lap_times.created_at`/`updated_at` are `DATE` columns, not `DATETIME`/`TIMESTAMP` — confirmed both via `SHOW COLUMNS` and by observing that every real row's `created_at` reads as exactly midnight. This directly contradicts several already-written mock UI strings that assumed hour/minute-level recency ("38m ago" on Homepage's Latest Records mock, "12m ago" on Player Single's Last Active). `LapTime`'s `created_at`/`updated_at` were explicitly cast to `'date'` (not Eloquent's implicit `'datetime'`) so the model's own type signature is honest about this rather than silently implying more precision than the data actually has. Documented as a new open question in [roadmap.md](roadmap.md) and a full breakdown in [database.md](database.md).

## Lap timestamp precision fixed going forward

Immediately after the above finding, the user decided: **"We'll add support for time to the data for future records. This can just be the time we received the record, as it's close enough."** — i.e. don't invent a separate precise-submission-time field or wait on the webhook payload inspection; just widen the existing `created_at` column so future inserts get real timestamp precision, and treat "time the row was inserted" as an acceptable proxy for "time the lap happened."

Verified: all 1657 existing rows preserved unchanged (still reading as midnight — MySQL doesn't invent data), and a test insert (wrapped in a transaction, rolled back afterward) confirmed new rows now get a real `H:i:s` timestamp. Removed the `'date'` cast from `LapTime` back to Eloquent's default `datetime` handling, since the column can now actually hold that precision.

**What this does and doesn't fix**: going forward, any lap submitted after this migration (2026-07-05) has real time-of-day precision. Historical rows before that point are permanently day-only — there was never a time-of-day to recover. This means relative-time displays ("3h ago") will be genuinely accurate for new laps but could look misleadingly precise for old ones if a future implementation isn't careful to format historical (pre-migration) records at day-level rather than hour/minute-level. Flagged as a display-logic nuance for whoever builds the historical-record-breaking-events derivation (roadmap.md item 13), not a blocker for anything currently in progress.

## First page wired to real data (Server Single), and what that actually took

Started Phase 2 item 8 with `ServerShow` (Server Single) — Maps section, Stats Card, and All Laps now query `Server`/`Map`/`LapTime` directly instead of mock arrays. Top Players (Server Score) stays mock, since it depends on Global Player Ranking (item 11), not yet implemented.

**Discovered along the way, before any of the real-data wiring could even be tested**: the project's test suite runs against an in-memory SQLite database built entirely from `database/migrations` — and until now, none of those migrations created `servers`/`maps`/`players`/`lap_times`/etc., because those tables already exist in the real dev DB and were deliberately never migrated (per the project's "don't run `php artisan migrate` blindly" caution). That caution was correct for the *real* DB, but it meant the *test* DB had no schema for any of these tables at all — the moment `ServerShow` ran a real `Server::findOrFail()` query, `RoutesTest`'s `/servers/1` case failed with "no such table: servers," because Pest's `RefreshDatabase` (commented out project-wide, unused until now) had nothing to migrate.

**Fixed by writing `create_*_table` migrations for every real table** (`maps`, `servers`, `players`, `lap_times`, `lap_time_splits`, `servers_maps`, `players_servers`, `users_players`, `users_servers`), matching the real schema's exact column types — these are meant to run in fresh environments (CI, a new dev machine's test suite) only, never against the already-populated real dev/prod database. Ordered via explicit sequential timestamps (not `make:migration`'s default same-second timestamps, which alphabetized `lap_times` before `players` and would have broken the foreign keys). `RoutesTest.php` now uses `RefreshDatabase` and seeds a `Server::factory()->create(['id' => 1])` in `beforeEach()` so `/servers/1` has something real to resolve.

**Two more real-data landmines found while building this**:
1. **`servers_maps` and `players_servers` have massive duplicate rows** — one server had 207 `servers_maps` rows for only 9 distinct maps, and 201 `players_servers` rows for only 17 distinct players. Almost certainly the old app inserted a new pivot row every time it *saw* a pairing again rather than upserting. Fixed by adding `->distinct()` directly inside `Server::maps()`/`Server::players()`/`Map::servers()`/`Player::servers()`'s relation definitions, so any consumer gets correctly deduplicated results automatically. Caveat kept in code comments and [database.md](database.md): a bare `->distinct()` is NOT honored by the `.count()` aggregate shortcut (it silently returns the un-deduplicated count) — must use `->distinct('table.id')->count('table.id')` instead. `ServerShow`'s Stats Card sidesteps this entirely by counting distinct players via `lap_times` directly (`LapTime::where('server_id', ...)->distinct('player_id')->count('player_id')`) rather than trusting the pivot table's count, since that's both more semantically correct (a player only "played" if they have a lap) and avoids the pitfall.
2. **The real `users` table's `id` column is `int unsigned`**, not `bigint unsigned` — but this project's *stock* Laravel `users` migration (the framework default, already in the repo before this session) creates `id()` as `bigIncrements`. This is a real, pre-existing mismatch between the imported production schema and the framework scaffold, not something introduced now. Also the real `users` table has an `api_token` column the stock migration doesn't create. Not fixed — out of scope for this pass (nothing currently being wired touches `User`) — but `players.user_id`/`users_players.user_id`/`users_servers.user_id` were all deliberately modeled as plain `unsignedInteger` columns rather than `foreignId()->constrained()` specifically to avoid a cross-type foreign key error from this mismatch. Flagged here as a real open item, not silently worked around.

**Also implemented for real, not left as an open question**: the Stats Card's "records set" now uses the current-records-held reading (comparing each map's best time on this server against that map's global-best across all servers) — the doc's stated default assumption, which turned out to be simple enough to just build rather than leave mocked once real models existed.

Verified the whole page against real data (not just the test schema) via Tinker against `/servers/7`: real server name, real per-map lap counts/best times, real stats (86 laps, 17 players, 0 records held), and a real paginated All Laps query — not just "does it 200," actual correct values.

**PHPStan then caught two more real things**, not suppressed: (1) every relation method needed an explicit `@return BelongsToMany<Map, $this>`-style generic — without it, Larastan can't infer what a relation's collection actually contains and flags every property access on the result as "undefined property on `Model`." Added generics to every relation across all models. (2) The `selectRaw`/`groupBy` aggregate queries for per-map stats (`MIN(time) as best`) return rows that aren't real `LapTime` records — they have a `best` column the model doesn't actually have. Fixed by calling `->toBase()` before `->get()`/`->pluck()` on those two queries, which drops to a plain query builder returning `stdClass` rows instead of mis-typed Eloquent models — the honest representation of what those rows are, not a workaround. `composer analyse` passes clean at 0 errors with these fixes; output values unchanged, verified by re-running the same Tinker check.

## Split comparison refined: three real bugs, one UX gap, all fixed

After using `ServerMapLeaderboard`, the user reported four specific issues with the split comparison — all real, all fixed in `LapTimeSplit::compare()` and the two modal partials:

1. **"If the current lap has splits, but the #1 lap does not, show the splits but no delta."** Previously `compare()` returned an empty array (nothing shown) whenever *either* lap lacked splits — even when the selected lap itself had real data worth displaying. As it's half."** Correct diagnosis of a real bug: the delta bar is drawn from a center reference line (`left-1/2`/`right-1/2` CSS positioning) out to one edge, so it only ever occupies **half** the track's total width — but `barW` was being scaled up to a max of 100, meaning the "biggest delta in this comparison" row would render with `width: 100%` starting from the center line, overflowing 50% past the track's actual edge. **Placement**: `App\Models\GlobalRanking` — not a new `app/Services`/`app/Support` folder. This project's own coding standard requires approval before creating a new base folder, and there's already a precedent for exactly this kind of thing: `LapTimeSplit::compare()` is a static, heavily-documented pure-calculation helper that lives alongside the models it reads rather than in a service class. `GlobalRanking` follows the same pattern — it has no table and is never persisted, just a stateless calculator computed fresh on every call (`scores(?int $serverId = null)`), consistent with this project's "derive, don't cache" default.

**PHPStan/Collection covariance**: the first implementation returned `Illuminate\Support\Collection<int, array{...}>` with a full array-shape docblock. PHPStan kept reporting a return-type mismatch even when the dumped inferred type (via `\PHPStan\dumpType()`, used temporarily to debug this — not left in the code) printed *identically* to the declared docblock type. Rewrote to seed both fields directly from the first lap instead of via a null placeholder — same runtime result, but now provably typed correctly, and also removes a `foreach ($ranked as $index => &$player)` by-reference loop (a real footgun: a lingering reference to the last array element after the loop ends) in favor of `Collection::map()`, which returns fresh arrays.

## Global/Server Score: sum vs. average, made A/B-switchable via config

Reviewing real Server Score output surfaced a real, debatable case (server 7, "Halo Race Leaderboard" — its stored server name):

| Player | Maps raced | Per-map ranks | Score (sum) |
|---|---|---|---|
| `[Æ] kingas` | 6 | rank 1 on all 6 | 600 |
| `EffakT` | 8 | rank 2×5, rank 3×2, rank 4×1 (never #1) | **741** |

Under the spec's sum formula, EffakT — who has never once held a course record on this server — outranks kingas, who has a flawless record on every map they've raced. Verified this wasn't a bug (wrong tie-break, double-counted laps, etc.) by dumping the actual per-map breakdown via tinker: the numbers are exactly right per the documented formula (rank 2 = 95pts, rank 3 = 90pts, etc. — near-#1 finishes are worth almost as much as #1, and summing across 2 extra maps closes the gap and then some). This is [global-ranking.md](global-ranking.md)'s stated design goal working as intended ("playing more maps well can only help, never dilute") — but seeing it flip a genuine record-holder below a non-record-holder on real data made it worth being able to compare against an alternative, not just take the spec's word for it.

**Decided**: rather than pick one, made both live behind `config('ranking.global_score_variant')` (env `GLOBAL_RANKING_VARIANT`, default `sum` — no behavior change unless explicitly flipped):
- `sum` — original spec, as above.
- `average` — total points ÷ maps played, so breadth alone can't out-score consistency. Same example under `average`: kingas 600/6 = **100.0** (rank 1), EffakT 741/8 = **92.6** → rounds to 93 (drops to rank 3) — the ordering flips, confirmed against real data.

Implementation: `GlobalRanking::applyScoreVariant()` applies the configured variant to every player's total per-map points right before ranking/tie-break — so both the score shown *and* the rank order reflect whichever variant is active, not just a display-layer difference. Per-map Points/Map Rank values (used on `PlayerShow`'s Performance by Map, etc.) are unaffected either way — only the summed/averaged aggregate changes definition. This is a comparison tool, not a resolved decision — `sum` stays the default until there's a reason to prefer `average` long-term.

**Immediate follow-up — naive `average` had its own small-sample problem, also found on real data**: flipping to `average` and checking the *global* leaderboard (not server-scoped) surfaced a worse-looking case:

| Player | Maps raced | Records | Score (naive average) |
|---|---|---|---|
| `D Stroyer` | **1** | 1 (100% of maps raced) | **100.0** → rank 1 |
| `-db-styLe` | 9 | 6 (67% of maps raced) | 97.0 → rank 3 |

A player who's raced exactly one map and happened to get rank 1 on it scores a flat, trivial 100.0 — a naive average can't tell a 1-map sample from a 9-map sample. Confirmed correct math again (not a bug), but this is the textbook reason rating systems (IMDB's weighted rating, Bayesian averages generally) don't use raw averages once sample size gets small.

**Decided**: regularized `average` into a Bayesian/weighted average rather than adding a hard maps-played cutoff (the other option offered — a cutoff would just make low-sample players invisible instead of fairly ranked). `GlobalRanking::applyScoreVariant()` now computes, for the `average` variant only: each player's own raw average, the overall mean average across every ranked player, and blends them weighted by `mapsPlayed / (mapsPlayed + confidenceMaps)` — `confidenceMaps` (`config('ranking.average_confidence_maps')`, default 5) is "how many virtual maps of average performance" a low-sample player is assumed to have alongside their real ones. As `mapsPlayed` grows relative to `confidenceMaps`, the weight shifts almost entirely onto the player's own average.

Verified against real data, both scopes:
- Global: `D Stroyer` (1 map) drops from rank 1 (100.0) to **rank 27** (28); `-db-styLe` (9 maps, 6 records) rises to **rank 1** (67).
- Server 7 (the original example): `kingas` (6/6 records) still beats `EffakT` (0 records, 8 maps) under `average` — 93 vs. 89 — just with regularized rather than raw (100.0 vs. 92.6) values.
- `sum` is byte-for-byte unaffected — `applyScoreVariant()` short-circuits immediately unless `average` is active.

**Second immediate follow-up — `confidenceMaps=5` overcorrected, because this game only has 10 maps total**: reviewing the regularized global leaderboard, `-db-styLe` (real per-map average ≈96.8 across 9 maps, 6 outright records — about as strong a player as exists in this dataset) was landing at a *final* score of just **67**, visibly worse than their obvious real skill suggested. Investigated with real numbers:
- The population mean being blended toward was **11.12** — because **454 of 688 players (66%)** have raced **exactly 1 map**, most with a poor result. The "average player" this formula assumes doesn't exist here; the actual median is a one-off tester, not a committed racer.
- There are only **10 real maps** and the most anyone's raced is **9**. With `confidenceMaps=5`, even that maximally-engaged 9-map player only ever reaches `9/(9+5) = 64%` weight on their own average — the remaining 36% is a *permanent* pull toward that 11.12 floor, since nobody can ever play a 10th map to dilute it further. The formula's core assumption (a large sample ceiling relative to the confidence constant) doesn't hold when the confidence constant is comparable in size to the entire possible sample space.

**Decided**: lowered `average_confidence_maps` from 5 to **2** (both the config default and the inline fallback in `GlobalRanking::applyScoreVariant()`) rather than changing the shrinkage target — keeps the same well-understood mechanism, just calibrated to this project's actual (tiny) map count. At `confidenceMaps=2`, a 9-map player reaches `9/11 = 82%` weight on their own average, a 6-map player reaches `6/8 = 75%`, both much closer to "trust the player's real record" while still meaningfully damping true 1-map flukes.

Verified against real data, both scopes:
- Global: `-db-styLe` rises from 67 → **82** (rank 1, unchanged rank but much less compressed); `D Stroyer` (1 map) stays correctly de-prioritized at **rank 17** (43) — not overcorrected back into dominating, just no longer crushed to near-irrelevance either.
- Server 7: `kingas` (flawless 6/6) rises from 93 → **96**, `EffakT` stays at 91, `WAZZUP` (2/2 records) at 92 — ordering unchanged, values closer to real per-player averages.

No further open issue identified after this pass — if the map roster grows meaningfully beyond ~10 maps, `confidenceMaps` should be revisited (it can afford to be larger once players can realistically rack up more maps before hitting a ceiling).

**Fix**: each of the three sub-picks (biggest improvement, biggest rank jump, new Top 10 entry) now tracks which lap ids have already been used by an earlier sub-pick in the same call, and prefers the best candidate *not* already used — falling back to reusing it only if no distinct real candidate exists this week. Verified: excluding Nuevo001's lap surfaced a genuinely distinct real story that was sitting right there — Puddin's real Ice Fields lap, rank #17 → #11 (a jump that existed the whole time but lost out to Nuevo001's bigger one under the old "pick independently" logic).

**Separate discovery while investigating, not the actual cause of this bug**: while tracing which player was behind each bullet, found **8 different real `Player` rows all named exactly "Nuevo001"** (ids 24, 30, 66, 90, 166, 414, 428, 742) — confirmed genuinely distinct via 8 different `hash` values, not a duplicate-row bug. This is inherent to the old app's hash-based player identity (`Player::firstOrCreate(['hash' => sha256(...)])` — see this file's earlier webhook-inspection entry): the same in-game alias can belong to multiple real identities if a player's hash ever changes (reinstall, different machine/profile, etc.), and nothing in the schema disambiguates them. Didn't cause today's bug (both bullets traced to the same single `player_id`, confirmed above), but it's a real latent risk: if two *different* "Nuevo001" accounts both had a qualifying event in the same week, a highlight sentence naming both would be genuinely ambiguous to a reader with no way to tell them apart. Not fixed here — that's a player-identity/deduplication problem in the same territory as the already-deferred claim-code ownership system — but worth knowing about. Added to [roadmap.md](roadmap.md)'s open questions.

## Fastest Improvements was gameable — sandbag a bad first lap, then a mediocre one, to fake a huge "improvement"

Immediate follow-up, spotted by inspection rather than exploitation: "Nuevo001 improved Gephyrophobia by 81.37s (now #70)" *looked* impressive, but rank #70 earns **zero real points** under `GlobalRanking::pointsForRank()` (anything past rank 50 does). The delta was real, but the result behind it wasn't competitive at all — which is exactly the shape of the exploit: a brand-new player has nothing to beat on a map yet, so their *first-ever* lap trivially becomes their "PB" no matter how slow it is. Submit something deliberately terrible, then a merely-average second lap, and the "improvement" delta looks huge purely because the baseline was manufactured — the actual resulting rank can be mediocre and it still wins "biggest improvement of the week."

The same weakness applies identically to "largest rank jump": a sandbagged first lap puts you near the very bottom of a map's ranking, so *any* real second attempt produces a large-looking rank jump, regardless of whether the destination rank is actually good.

**Fix**: both sub-picks now require the *resulting* rank to actually earn points (`GlobalRanking::pointsForRank($rank) > 0`, i.e. rank ≤50) — reusing the ranking system's own existing cutoff rather than inventing a new threshold. This directly targets the exploit's actual weakness: sandbagging can inflate a *delta*, but it can't fake a *competitive result* — the player is still only as fast as their genuine second lap, and if that's not good enough to earn points, it's not highlight-worthy regardless of how big the improvement number looks.

**Not a complete anti-cheat fix, and not trying to be one** — a determined bad actor could pad multiple throwaway laps before a good one to defeat any single-prior-lap heuristic; this closes the casual/incidental version of the trick (which is what actually surfaced on real data), not a fully adversarial one. Real anti-cheat is already an explicitly parked, unsolved problem for this project (see [most-active-server.md](most-active-server.md)'s "Lap validity / anti-cheat mechanism" open item) — this fix doesn't attempt to solve that, just to stop a webhook quirk from accidentally producing a "look how much I improved" headline for a non-competitive result.

Verified against real data: with the gate applied, Nuevo001's rank-70 "improvement" no longer qualifies for either sub-item. The block now genuinely has only one real, competitive story this week — Puddin's Ice Fields lap (14.97s improvement, rank #17 → #11, both of which now earn real points) — and both remaining bullets describe that single genuine achievement from two angles. Left as-is rather than further suppressed: unlike the earlier fix (which closed a fake/gamed duplicate), this is a real accomplishment being described twice, which isn't dishonest, just two true facts about one good week.

**Small follow-up**: with only 2 of 6 candidate highlight blocks currently qualifying (real recent activity is sparse), the Latest Highlights grid — a fixed `tl:grid-cols-3` — left a dangling empty third column. The column count is now derived from `count($highlights)` (1 → full width, 2 → 50/50, 3 → the original thirds), computed inline in `home.blade.php` via a `match`. Confirmed the new `tl:grid-cols-2`/`tl:grid-cols-1` utility classes weren't already compiled anywhere else in the codebase, so ran `npm run build` and verified the class landed in the compiled CSS before checking the rendered page.

## Most Active Server implemented for real (roadmap item 12)

New `App\Models\MostActiveServer` — same non-Eloquent, computed-fresh-every-call precedent as `GlobalRanking` — implements [most-active-server.md](most-active-server.md) exactly: Activity Score (`Unique Players ×10 + Valid Laps ×1 + Maps Played ×20`, 90-day window, "Valid Laps" = distinct (player, map) participations not raw lap count), a recency bonus (+100/+50/+20 for activity within 7/30/90 days, highest only), and a 4-level tie-break (unique players → maps played → valid laps → most recent activity). Also computes the display-only 30d/90d unique-player pair Homepage's highlight wants, independent of the score's own 90-day window.

**No scheduled recalculation job was built.** The spec says "recalculated periodically," implying the author expected this might eventually need caching — but at real current scale (3 active servers total) a live query per page load is trivially fast, so building a job now would be speculative infrastructure with no measured need. Same "derive, don't cache until profiling says otherwise" default already applied to `GlobalRanking`. Revisit if the server roster grows enough for this to matter.

**Wired into two real consumers**: `ServerList`'s featured "MOST ACTIVE" card (replacing the old "laps in the last 30 days, tie-broken by recency" proxy that was explicitly documented as a stand-in for this exact item), and Homepage's Most Active Server highlight (top 3, podium-style, only servers with `totalScore > 0` — an inactive server showing up in a "most active" list would be worse than showing fewer than 3).

**PHPStan/Larastan quirk worth knowing about before writing another calculator like this**: `?->prop ?? default` on a genuinely nullable `Carbon` field falsely triggered `nullsafe.neverNull` ("using nullsafe access is unnecessary") even after confirming via `\PHPStan\dumpType()` that the variable's type really was `Carbon|null` at that exact line. Root cause narrowed down to the specific combination of nullsafe (`?->`) immediately followed by null-coalesce (`??`) on a `Carbon` property access — swapping to an explicit `$x !== null ? $x->prop : $default` ternary (semantically identical) made the false positive disappear immediately. Also tried and ruled out along the way: `Builder::max()` and `Builder::value()` both have known nullability blind spots in Larastan's stubs (they don't account for "zero rows matched" returning null) — `->orderByDesc(...)->first()` then `?->column` is the reliable pattern already used elsewhere in this codebase (e.g. `ServerList::buildRow()`) for "most recent row's timestamp, or null."

Verified against real data: "Halo Race Leaderboard" (server 7) scores 412 (312 activity + 100 recency bonus) from 11 unique players / 9 maps / 22 valid laps in the last 90 days — correctly ranks #1 over the two long-dormant demo servers (both score 0, last activity 4-5 years ago). Both `ServerList` and the Homepage highlight render this correctly end-to-end.

## Historical record-breaking events implemented for real (roadmap item 13)

The last genuinely-hard piece of Phase 2: unlike `GlobalRanking`/`MostActiveServer`, this can't be answered from current state (optionally minus one lap) — "when did a lap actually become the new fastest time on a map" requires replaying a map's entire lap history in order and watching the record change hands. New `App\Models\RecordHistory::events(?int $mapId = null)` does exactly that: for each map, walks every real lap chronologically (`created_at`, then `id` as the tie-break for historical rows sharing a day-precision timestamp — same convention as everywhere else), and records a new "event" whenever a lap is strictly faster than the fastest time before it. A map's very first-ever lap always counts (trivially becomes the record, `previousTimeRaw` is `null`). Ties don't count as breaking a record, matching this app's established "earliest lap wins a tie" rule.

Two derived helpers: `recent(int $limit, ?int $withinDays)` — most recent events overall, optionally windowed, for Homepage's "Latest / Current Records" highlight — and `firstRecordFor(int $playerId)` — a player's own earliest event, for Achievements' "first record" sub-item (previously skipped for lack of exactly this).

**Wired into three real consumers**:
- Homepage's "Latest / Current Records" highlight (`RecordHistory::recent(3, 7)`) — the last of the six highlight blocks to go real.
- Player Achievements' "first record for a player" sub-item — checks whether a lap from this week matches that player's `firstRecordFor()` result (built once per call via `collect(RecordHistory::events())->unique('playerId')`, which correctly returns each player's *earliest* event since `events()` is already chronologically ordered ascending and `Collection::unique()` keeps the first occurrence per key — no separate sort needed for this specific lookup).
- Players List's info-card "Total records set" — switched from `sum(firstPlaces)` (current-state: how many maps each player currently holds) to `count(RecordHistory::events())` (historical: how many times a record has ever changed hands). The per-player "Records" **table column** on the same page deliberately keeps the current-state reading — `players-list.md` already flagged these two could legitimately differ, and now that both are real and visible on the same page, they read fine, not confusing (19 vs. individual per-player counts).

**Same PHPStan `?->prop ?? default` quirk found in the `MostActiveServer` work reappeared** in two places here (`recent()`'s sort, `firstRecordFor()`'s sort) — same fix, explicit `!== null ? ... : ...` ternary instead. A third, subtler variant also showed up: `recent()` originally reassigned `$events` inside a conditional (`if ($withinDays !== null) { $events = $events->filter(...); }`), and PHPStan then treated `$event['setAt']` as *always* non-null in the `sortByDesc()` call after that block — even though the filter (and its narrowing) only actually runs when `$withinDays !== null`. Restructured to compute the filter predicate as a single unconditional expression (`$cutoff === null || ...`) instead of conditionally reassigning the collection — no more incorrect narrowing, and arguably cleaner code anyway.

Verified against real data: 19 total historical record-breaking events across all maps, oldest from 2020-04-15 (several maps' very-first-ever lap, `previousTimeRaw: null`, confirming that branch works), most recent from 5-6 years ago — correctly **zero** within the last 7 days, matching the already-established fact that this week's real activity (Puddin, Nuevo001) never actually reached rank 1 on anything. Confirmed the Homepage's "records" highlight correctly falls out of this round's fixed-priority selection (empty, not padded with anything fake) and the other three real blocks (`most-active-server`, `fastest-improvements`, `live-stats`) fill in instead. Players List now shows "19 RECORDS SET" for real.

## Webhook rebuild (roadmap item 14)

Rebuilt `ApiController::newTime` + `ProcessNewLap` (`-legacy`) as `Api\V1\LapSubmissionController::store` + `App\Jobs\ProcessNewLap` behind `POST /api/v1/laps`. Full old-vs-new behavior comparison lives in [database.md](database.md)'s "Webhook → job flow" section; this entry covers the implementation decisions and one real bug the port to a different SQLite version surfaced.

**No authentication — explicitly reconfirmed, not just ported silently.** The old webhook had zero auth of any kind (any IP could POST fake laps). Rather than carry that forward without comment, this was surfaced to the user directly before building anything: keep it open (matching the old app) or add a shared-secret header. **User's call: keep it open.** Documented here so this isn't mistaken for an oversight later — see [security.md](security.md)'s "Not yet addressed" section, still flagged as a known gap.

**`QueryServer` rebuilt with real key/value parsing, not fixed offsets.** The legacy implementation read `numplayers` from a hardcoded array index and player/score/ping/team fields from hardcoded slot offsets computed from it — fragile against any server response with a different key ordering or extra fields. The rebuild (`app/Helpers/QueryServer.php`, behind a new `GameServerQuery` interface bound in `AppServiceProvider` so tests can fake it without opening a real socket) parses the response into an actual associative array first, then reads fields by key. Modernized `socket_*` option calls (`SO_RCVTIMEO`/`SO_SNDTIMEO` instead of the legacy blocking-connect-with-manual-timeout loop) but kept the exact wire format (`\query`, 6 bytes, no trailing backslash — confirmed from the legacy `socket_send(..., "\\query", 6, ...)` call).

**A failed live query no longer discards the lap.** The old app aborted the entire submission if the UDP query failed — meaning a temporary game-server hiccup silently dropped real lap data with no record it ever happened. Decided this was a bug worth fixing, not a behavior worth preserving: a failed query now just logs a warning and proceeds with a placeholder server name (`"Unknown (ip:port)"` for a brand-new server, or the existing stored name left untouched for an already-known one) rather than losing the lap.

**Duplicate pivot rows fixed at the write site, not via a DB migration.** [database.md](database.md)'s "Duplicate pivot rows" section already identified the root cause (`insertOrIgnore` with no unique constraint to violate) and flagged fixing it as future work *if/when the pivot-writing logic is ported*. That time is now. Rather than add a unique constraint retroactively (the real imported tables already contain hundreds of duplicate rows per pairing — a dedupe-then-constrain migration against production-derived data is a bigger, riskier task on its own, left for a future session if ever needed), the new job calls `$server->players()->syncWithoutDetaching([...])` / `$server->maps()->syncWithoutDetaching([...])` — an application-level fix. `syncWithoutDetaching` checks for an existing pivot row before inserting, so no *new* duplicates get created going forward, without touching the real schema or the existing duplicate rows.

**Logs every attempt, not just personal-best improvements** — the behavioral change decided earlier (see [database.md](database.md)) is now implemented: `LapTime::create()` runs unconditionally, `isNewRecord` is computed first (via the pre-insert `MIN(time)`) purely to decide the response shape and whether to broadcast, not whether to write the row.

**Broadcasting wired for roadmap item 16, not yet consumed.** A new `App\Events\LeaderboardUpdated` (`ShouldBroadcast`, public `Channel`, fired only when `isNewRecord`) exists so the webhook and the future Reverb/Echo work (item 16, still not started) don't need to touch each other later — this event is the seam. With this dev environment's default `BROADCAST_CONNECTION=log`, submissions currently just write a log line; no frontend listens yet.

**Real bug found and fixed: a SQLite/PDO bug in this environment silently breaks parameterized aggregate comparisons.** The leaderboard-position calculation (how many other players already have a faster personal best) was first written the same way the legacy job did it — a `GROUP BY player_id` subquery wrapped in an outer query filtering on the aggregate alias via a bound parameter (`WHERE best_time < ?`). A test asserting the exact position number failed with an impossible-looking result (`3` where `2` was correct), traced all the way down to raw `PDO`: `WHERE <aggregate-alias> < ?` (and the equivalent `HAVING MIN(time) < ?`) returns **every row, completely ignoring the filter**, specifically when the comparison value is a *bound parameter* — the identical query with a literal value filters correctly. Confirmed via `sqlite_version()` (3.34.1, PHP 8.4's bundled `pdo_sqlite`) with a minimal reproduction outside Laravel entirely (raw `PDO::prepare`/`execute`), so this isn't a Laravel/Eloquent query-builder bug. **Fixed by not doing aggregate filtering in SQL at all**: `leaderboardPosition()` now pulls each player's per-map best time via a plain `GROUP BY` `pluck()` (small result set — real scale is at most a few hundred players per map, see [database.md](database.md)) and does the "how many are faster" comparison in PHP. This matches this app's existing precedent (`GlobalRanking`, `MostActiveServer` already compute rankings in PHP over plain arrays rather than complex SQL aggregation) and sidesteps the SQLite bug entirely — no dependency on which SQLite version a given environment happens to bundle. Worth remembering for any *other* aggregate-comparison SQL written against this test suite's SQLite backend.

Added `LapSubmissionTest.php` (9 tests) covering: server/player/map creation with a live-queried hostname, map-label alias+race-type derivation, the failed-query placeholder-name fallback, logging every attempt (not just PBs), the pivot-duplication fix, split storage, the `LeaderboardUpdated` broadcast firing only on genuine improvements, the leaderboard-position/gap calculation (the test that caught the SQLite bug above), and payload validation. A `GameServerQuery` fake (bound per-test via the container) stands in for the real UDP client — no real socket ever opens in tests. Suite: 70 → 79 tests, 179 → 208 assertions.

## Live server info (roadmap item 19)

Empirically confirmed the `QueryServer` UDP response's real fields (rather than guessing) by querying the actual production server (id 7) with the rebuilt `QueryServer` helper — it's genuinely live and reachable (~5ms response). Full field list and the SAPP-mod discovery in [database.md](database.md). This resolved the one remaining unknown from item 14's protocol recovery: whether "current map" was even queryable at all. It is — `mapname` matches `maps.name`'s format exactly, no translation needed.

New `App\Console\Commands\RefreshLiveServerInfo`, scheduled every minute, stores the result on new `servers` columns (`current_map_id`, `live_player_count`, `queried_at`, `query_successful`) rather than live-fetching per request — the same "scheduled job, not live query" reasoning already decided when this item was added (UDP has real latency/failure modes, unlike this app's other derive-fresh-in-PHP calculators).

**Real production DB migrated for this** (`add_live_query_fields_to_servers_table`, run scoped via `--path=` per this project's standing rule — never a blanket `migrate`). Used `unsignedInteger`/`->foreign()` for `current_map_id`, not Laravel's `foreignId()`/`constrained()` shorthand — `maps.id` is `increments()` (`INT UNSIGNED`), not `bigIncrements()`, matching every other FK to it in this schema (e.g. `lap_times.map_id`); `foreignId()` would have created a mismatched `BIGINT UNSIGNED` column.

**Design decisions for how live vs. proxy data interact** (the two things roadmap item 19 flagged as needed before implementation):
- **Freshness window**: 5 minutes — a generous margin over the 1-minute schedule so a missed run or two doesn't immediately flip a server back to "unknown."
- **Unreachable display**: neither pure "stale cached value" nor pure "explicit unknown" — split by field. A *fresh failed* query is treated as authoritative for "online" (it just directly confirmed unreachability — more trustworthy than the old recency proxy), but falls back to the lap-history proxy for "map" (a failed query has no live map to report). A *stale* query (or none yet) falls back to the old proxies entirely for both fields. A failed query never wipes previously-known `current_map_id`/`live_player_count` — only `queried_at`/`query_successful` update.
- **Never fabricate a `Map` row from a live query** — same principle as the webhook (item 14): if the live `mapname` doesn't match any existing `Map`, `current_map_id` just stays `null`. A `Map` row's existence should only ever be caused by an actual lap submission.

**Scope decision**: `live_player_count` is stored but not yet surfaced in `ServerList`'s UI. That page's existing `players`/`playersRaw` fields already mean something different — "distinct players who have ever played here" (all-time roster, used for the relative "load" bar) — not "currently connected right now." Introducing a second, differently-scoped player-count concept into that card was left as a follow-up display decision rather than folded into this task without a clear design for how the two coexist.

**A real, pre-existing PHPStan/Larastan gap found while wiring this up**: Larastan doesn't resolve a custom `casts(): array` entry into its Carbon type for non-standard (non `created_at`/`updated_at`/`deleted_at`) column names in this project's setup — confirmed via `\PHPStan\dumpType()` on both the new `queried_at` field *and* the pre-existing, untouched `notify_outage_last` field (both declared `'datetime'` in `casts()`), both resolving to `string|null` despite the cast working correctly at runtime (verified in tinker). Fixed with `@property` class-level docblock annotations on `Server` — the standard, sanctioned mechanism for declaring Eloquent magic-property types to static analysis tools (the same thing `php artisan ide-helper:models` generates), not a suppression of a real bug.

Added `RefreshLiveServerInfoTest.php` (4 tests: live data stored correctly, no `Map` fabrication, failed query doesn't wipe good data, archived servers skipped) and extended `ServerListTest.php` (+3 tests: fresh-live-data precedence, fresh-failed-query-is-authoritative-for-online, stale-live-data-falls-back-to-proxy). Suite: 79 → 86 tests, 208 → 229 assertions.

## Webhook auth reversal — HRL query verification (SEC-01, 2026-07-07)

A site audit (`SITE_AUDIT.md`) flagged the 2026-07-06 "keep the webhook open" decision above as critical: an unauthenticated caller can fabricate leaderboard data outright, not just abuse rate limits. The audit's own recommended mitigation (add TLS/HMAC via a local relay on each game-server host) was rejected as out of scope for this repo — the Lua/SAPP script that actually submits laps isn't part of this codebase, its HTTP capabilities are unknown, and building/deploying a relay on every game-server host is real infrastructure work beyond a Laravel change.

Instead, at the user's direction, `App\Helpers\LapSubmissionVerifier` cross-checks every submission against a **live UDP `\query` response from the same ip:port**, reusing the UDP query protocol already built for roadmap item 19 (`App\Helpers\QueryServer`) rather than inventing new transport. The Lua script publishes `hrl_enabled`/`hrl_protocol`/`hrl_token` via SAPP's existing `query_add` capability (no TLS/crypto needed); the same token is echoed in the HTTP submission body. Verification requires all of: a successful UDP query, `hrl_enabled=1`, a supported protocol version, a matching token, the UDP `mapname` matching the submitted map, and the submitted player name appearing in the UDP `player_0..N` list. See [security.md](security.md)'s "SEC-01 — HRL query verification" for the full field spec and rationale.

**Deliberately not full authentication** — a server operator controls their own query responses and could fabricate activity on their own server (no worse than what they could already do by actually playing), and an on-path attacker can still see/alter plaintext HTTP/UDP traffic. It does close the specific hole the audit flagged: an arbitrary internet caller who isn't running a live, matching game server can no longer inject data.

**`config('webhook.hrl_query.enforce')` defaults to `false`** — a genuinely important rollout decision, not an oversight: flipping straight to hard rejection would break every real game server whose Lua script predates this feature (none of them send `hrl_*` fields yet). Default behavior only logs a warning (`missing_hrl_marker`, etc.) and still records the lap; `enforce` should only flip to `true` once every active server's script is confirmed updated. This also means none of the pre-existing `LapSubmissionTest.php` fakes (none of which send `hrl_*` fields) needed to change — they exercise the log-only path by design.

Also added, independent of HRL verification: a short-window (`config('webhook.duplicate_window_seconds')`, default 10s) exact-duplicate-payload rejection via `Cache::add()`, and the `webhook` rate limiter now keys by ip **and** port rather than IP alone (distinct game servers behind one host IP no longer share a budget).

New tests: `tests/Feature/LapSubmissionVerifierTest.php` (verifier unit-of-work, all rejection reasons + the retry-once behavior), plus 4 new cases in `LapSubmissionTest.php` (log-only-by-default, enforced-rejection, enforced-acceptance, duplicate-rejection).

## SEC-01 follow-up hardening, same day (2026-07-07)

A second audit pass reviewed commit `0f2362a`'s implementation directly (findings recorded in `SITE_AUDIT.md`) and found five real gaps, all fixed before `enforce` should ever be considered safe to flip on. Full technical writeup lives in [security.md](security.md)'s "Follow-up hardening" section; this entry covers the reasoning behind each fix.

**Two rate limits instead of one.** The original `webhook` limiter keyed by `ip:port` alone — but `port` is caller-supplied and unchecked at the throttle layer (it's only cross-checked against a live UDP query *inside* the controller, after the rate limiter has already run), so an attacker could rotate it every request for an unlimited effective rate while still forcing a real UDP round-trip per request. `RateLimiter::for()` closures can return an array of `Limit`s that all apply together — added a coarser per-IP ceiling (`config('webhook.rate_limit.per_ip_per_minute')`, 600/min default) alongside the existing per-`ip:port` one (120/min), so port rotation no longer helps.

**Idempotency reworked from "reject duplicates" to "replay the original result."** The initial version's `Cache::add()` guard had two real problems: it reserved the dedupe key *before* verification/processing ran, so (a) a pre-commit failure (an exception mid-processing) left the key reserved for the rest of the window even though nothing was actually recorded — a legitimate retry of a never-completed request would fail with `duplicate_submission` for no reason — and (b) even a *successful* retry (e.g. the game server's own HTTP client timed out waiting for a response it never saw, and resent the identical lap) got a bare `409` instead of the response it actually wanted, with no way to safely retry at all. Fixed by giving the cache entry a real two-state lifecycle (`processing` → `done`, storing the actual response body + status code): a repeat while `done` now replays that exact response; an exception during processing calls `Cache::forget()` to release the reservation instead of leaving it stuck. Also accepts an optional client-supplied `submission_id` as the idempotency key (falls back to the previous content-hash approach when absent) — a real Lua-side retry can now say "this is the same attempt" explicitly rather than relying on every field happening to match byte-for-byte.

**Token rotation grace.** `LapSubmissionVerifier` now also accepts an `hrl_token_prev` UDP field, so a lap submission racing a Lua-side token rotation boundary isn't wrongly rejected by an already-rotated token. Deliberately did **not** implement a numeric token max-age check — that would need the Lua script to publish and the server to trust an issue timestamp, which is more surface than the current/previous fallback already covers; flagged as an open item to revisit only if the fallback proves insufficient in practice.

**Removed the duplicate UDP round-trip.** `LapSubmissionVerifier::verify()` was already fetching a live query response; `ProcessNewLap::resolveHostname()` was fetching a second, independent one for the exact same ip:port right after. `verify()` now returns its raw `response` array in its result, and the controller threads it into `ProcessNewLap`'s new `?array $liveQueryResponse` constructor parameter, which `resolveHostname()` prefers over querying again. Falls back to querying itself when null (verification disabled, or its own query failed) — unchanged from before in that case.

**Tightened player-slot matching.** `player_not_online` previously matched any UDP key merely starting with `player_` — changed to an exact `/^player_\d+$/` pattern so a hypothetical future query field like `player_count` can't accidentally satisfy "is this player online."

New tests: 6 more cases in `LapSubmissionTest.php` (idempotent replay by content-hash and by `submission_id`, a genuinely-concurrent in-flight duplicate still getting `409`, reservation release on a pre-commit failure, both rate-limit tiers), plus 2 more in `LapSubmissionVerifierTest.php` (token rotation grace, the tightened player-key regex). 117 → 125 tests, 286 → 308 assertions.

## SEC-01 second follow-up, same day (2026-07-07) — and a new standing rule

A third audit pass reviewed `eb28327` directly and found five more real gaps (recorded in `SITE_AUDIT.md`). Two of them — the exact rate-limit numbers, and how much idempotency infrastructure to build — were phrased by the audit in open-ended terms ("substantially lower," "consider separate allowances") rather than a single unambiguous fix. Rather than pick numbers/mechanisms solo (as had happened implicitly in the first two SEC-01 passes), these were surfaced to the user directly via AskUserQuestion before writing any code. **The user then gave explicit standing feedback**: ask first whenever an audit finding requires a decision it doesn't literally specify, not just implement a "reasonable" interpretation and let them review after the fact. Saved as a memory (`feedback_audit_scope_discipline`) so this applies to every future audit-remediation pass, not just this one.

**Durable idempotency (user's choice: simple column, not a claims table).** Offered two designs — a new `lap_submissions` claims table, or a unique column directly on `lap_times` — the user picked the simpler column. Added a nullable `submission_id` string column to `lap_times` plus a unique index on `(server_id, submission_id)` (a new migration, run scoped via `--path=` against the dev DB per the standing migration-safety rule). **Two-tier rate limiting (user's choice: build both, not just lower one number).** Given the choice between a single lower flat ceiling vs. a full tiered "verified sources get more" system, the user asked for the tiered system, with specific starting numbers: unverified 30/min per IP + 15/min per ip:port + 2/sec burst; verified (after a real, passing HRL query verification from that exact ip:port, cached 5 minutes) 300/min + 120/min + 10/sec burst. ## SEC-01 third follow-up, same day (2026-07-07) — the servers unique-constraint saga

A fourth audit pass verified the staging deployment directly (confirmed the migration ran, the index exists, no duplicate `(ip, port)` pairs in real data, enforcement still off) and found four more gaps. Two had one unambiguous fix each — revoke the verified rate-limit marker immediately on any failed verification (previously only expired naturally after 5 minutes, letting a source that stopped answering UDP keep the generous tier), and store a canonical content hash alongside a cached idempotent response so a reused `submission_id` with genuinely different lap content gets `409 idempotency_conflict` instead of silently replaying stale data (plus, per the audit's own forward-looking note, `submission_id` is now `required` once `webhook.hrl_query.enforce` is on).

**The `servers` unique-constraint fix took three real attempts, all worth recording** so the next person doesn't repeat them:

1. **First attempt: plain `unique(ip, port)`.** Rejected on review (by me, before even trying it) — `servers` uses `SoftDeletes`, and a plain unique index would permanently block that ip:port from ever being assigned to a genuinely new server once the old one is archived.
2. **Second attempt: `unique(ip, port, deleted_at)`.** Looked like the obvious fix for the SoftDeletes problem — include `deleted_at` so archived rows don't collide. Wrote it, ran it, and a test immediately proved it does nothing at all: SQL's `NULL != NULL` means a unique index never considers two `NULL`s equal, so since every active row has `deleted_at IS NULL`, the constraint silently never fires between active rows — the exact case it was supposed to protect. Caught by `Server::factory()->create()` twice with the same ip/port in a test and expecting (and not getting) a thrown exception.
3. **Third attempt: a generated column referencing the row's own `id`.** The idea: `0` for every active row (so active duplicates collide), the row's own always-unique `id` once soft-deleted (so archived rows never collide with each other or a new active row). Failed immediately against the real MySQL dev database with error 3109 — MySQL explicitly disallows a generated column referencing an `AUTO_INCREMENT` column. This surfaced a fact worth remembering: **this app's actual dev/prod database is MySQL**, not SQLite — SQLite is test-only (`phpunit.xml`'s `DB_CONNECTION=sqlite` override). Assuming "whatever works in the Pest suite works in production" would have shipped a broken migration.
4. **What shipped: `COALESCE(deleted_at, sentinel)`.** A generated `active_since` column collapsing every active row to the same fixed sentinel timestamp (so active duplicates collide) while soft-deleted rows keep their own real `deleted_at` (so archived rows don't collide with each other or a future reuse). Even the sentinel value took two tries: `'1970-01-01 00:00:00'` and then `'1970-01-01 00:00:01'` were both rejected by MySQL's `TIMESTAMP` column-type validity range (`00:00:00` is one second below the documented minimum; `00:00:01` still failed, apparently pushed out of range by the connection's session time zone) — settled on an arbitrary year-2000 date, safely inside the 1970–2038 range and nowhere near either edge.

Given this constraint's complexity was now visibly more than "just add a unique index," the user's earlier standing instruction (ask before deciding something the audit doesn't literally specify) applied again — the choice between accepting the soft-delete limitation outright, building the generated-column workaround, or skipping the DB constraint for now was put to the user directly rather than picked solo. **User's call: build the generated-column workaround properly.**

`ProcessNewLap::handle()` now retries the whole transaction once on a `UniqueConstraintViolationException` that its `violatesSubmissionIdUniqueness()` helper doesn't attribute to the `submission_id` constraint — by elimination, the only other unique index this transaction can hit is the new `servers` one, meaning a concurrent request already created the row; the retry's `firstOrCreate()` SELECT finds it. That helper also replaces the earlier version's generic SQLSTATE-code check (`in_array($e->getCode(), ['23000', '23505'])`) — broad enough to misreport an unrelated integrity error (e.g. a foreign-key violation) as a safe-to-replay duplicate — with Laravel's specific `UniqueConstraintViolationException` and its `$columns`/`$index` properties, which name the actual constraint that fired.

The true concurrent-request race (two simultaneous first-ever submissions actually racing inside `Server::firstOrCreate()`) isn't reproducible in a single-threaded Pest test — covered instead by a schema-level test proving the constraint rejects a duplicate insert and correctly permits reuse of a soft-deleted server's old ip:port, plus code review of the retry logic itself.

New tests in `LapSubmissionTest.php`: marker revocation on failed verification, the `servers` unique constraint (active-duplicate rejection + soft-deleted-reuse permission), `idempotency_conflict` on a changed-content resubmission, and `submission_id` required under enforcement. 129 → 133 tests, 321 → 331 assertions.

## SEC-01 fourth follow-up, same day (2026-07-07) — closing the cache/durability gap on payload-conflict detection

A fifth audit pass verified the deployed MySQL schema directly (the generated `active_since` column, the `servers_ip_port_active_since_unique` index, no duplicate active identities, archived-address reuse working) and found three more gaps, all narrower than the previous rounds — no open-ended design decisions this time, so nothing needed to go back to the user before implementing.

1. **The content-hash conflict check (third follow-up) only worked while the cache entry was alive.** Once the 5-minute cache entry for a `submission_id` expired, was evicted, or the app restarted, reusing that `submission_id` with different content hit the database's unique constraint directly, and `ProcessNewLap::replayDuplicateSubmission()` had no fingerprint to compare against — it just replayed the old lap as a success, silently losing the fact that this was a materially different attempt. Fixed by persisting the hash, not just caching it: a nullable `submission_hash` column on `lap_times` (`add_submission_hash_to_lap_times_table` migration), populated on every insert. `replayDuplicateSubmission()` now compares the incoming submission's fingerprint against the stored one and throws a new `LapSubmissionConflictException` on mismatch, which the controller catches and turns into the same `409 idempotency_conflict` the cache-based path already returns.

3. **`ProcessNewLap` assumed by elimination that any non-submission_id unique violation was the `servers` race.** True today (only two unique constraints exist), but silently wrong the moment a third one is added later — it would get retried instead of surfaced as a real bug. New tests: a durable (cache-flushed) version of both the conflict-detection and genuine-retry cases in `LapSubmissionTest.php`, a reflection-based test exercising both unique-violation classifier methods against constructed exceptions for all three index shapes, and a new `LapSubmissionHashTest.php` for the hash's field coverage and split-order normalization. 133 → 137 tests, 331 → 349 assertions.

## SEC-01 fifth follow-up, same day (2026-07-07) — canonicalizing the content hash's numeric inputs

A sixth audit pass confirmed the `submission_hash` column deployed correctly and found one real narrow gap plus a doc drift, both fixed directly (no open-ended decisions this round).

`LapSubmissionHash::compute()` hashed validated request fields as-is. Laravel's `numeric`/`integer` validation rules accept a numeric string (`"42.5"`, `"2"`) without ever coercing it to a native type — so a real request could reach the hash with `player_time` or a split's `checkpoint_id` as either a number or an equivalent string, and those two semantically identical submissions would hash differently, defeating the whole point of the fingerprint. ## SEC-02's real "Not verified" gap: `$request->secure()` was always true, root-caused and fixed (2026-07-08)

The user ran `curl -sSI http://redesign.hrl.effakt.info/` directly (checking `SITE_AUDIT.md`'s open SEC-02 verification note) and got a plain `200 OK` — no redirect to HTTPS, despite `RedirectIfNotSecure` being registered on the `web` middleware group and `APP_ENV=staging` satisfying its `environment('production', 'staging')` check. Root-caused with a temporary debug route (`Route::get('/__scheme-debug', ...)` dumping `request()->secure()`, `$_SERVER['HTTPS']`, `$_SERVER['SERVER_PORT']`, and the `X-Forwarded-Proto`/`X-Forwarded-Ssl` headers as JSON), curled over both `http://` and `https://` against the real live domain, then removed immediately once the data was captured — a one-off diagnostic, not a permanent addition, following the same pattern as the earlier standalone `pusher-js` verification scripts. ## The `trustProxies('*')` fix above shipped a real open redirect; corrected same day (2026-07-08)

The user re-tested the fix above directly against the live site and found a genuine vulnerability introduced by it: `curl -sSI http://redesign.hrl.effakt.info/maps -H 'X-Forwarded-Host: evil.example' -H 'X-Forwarded-Port: 444'` returned `Location: https://evil.example:444/maps`. Trusting `HEADER_X_FORWARDED_HOST`/`HEADER_X_FORWARDED_PORT` from any IP (`at: '*'`) meant Laravel built the redirect's target host/port from those headers — and since they're client-supplied on the original request, an attacker could set them to anything, turning a same-origin HTTP→HTTPS redirect into a full open redirect. Reproduced live before touching anything (`curl` above, confirmed with the vulnerable code still in place), then fixed.

The safety argument for `trustProxies('*')` in the earlier entry ("PHP-FPM only listens on a Unix socket reachable exclusively via this box's own nginx") was real but incomplete: it correctly rules out *bypassing the proxy entirely*, but says nothing about which headers the proxy itself *forwards as-is* versus *overwrites with its own accurate value*. `X-Forwarded-Host`/`X-Forwarded-Port` fall in the first category — the edge (Nginx Proxy Manager) passes through whatever the original client sent, since a legitimate client can validly claim different apparent host/port values in a genuinely multi-tenant reverse-proxy setup. `X-Forwarded-Proto`/`X-Forwarded-For` fall in the second category for this specific topology — the edge overwrites them with the connection's real scheme/source IP before forwarding, so a client-forged value never actually reaches this app (confirmed by the user's own spoof testing: forged `X-Forwarded-Proto`/`X-Forwarded-For` were silently overwritten by the edge, while forged `X-Forwarded-Host`/`X-Forwarded-Port` were not). The two categories need different trust decisions even though all four are nominally "forwarded headers behind the same fixed proxy" — treating the whole set as one trust decision was the actual mistake, not the `at: '*'` scope itself.

Fixed in `bootstrap/app.php` by dropping `HEADER_X_FORWARDED_HOST`/`HEADER_X_FORWARDED_PORT` from the trusted mask, keeping only `HEADER_X_FORWARDED_FOR | HEADER_X_FORWARDED_PROTO`. `FOR` is kept (not just `PROTO`) because `$request->ip()` is load-bearing elsewhere — `AppServiceProvider`'s webhook and read-API rate limiters and `App\Helpers\ResolveSubmittingIp`'s NAT-address remap both key off it, and since the game-server webhook traffic also passes through the same NPM edge, `$request->ip()` needs `X-Forwarded-For` trusted to resolve to the real submitting IP rather than the edge's own address.

Also removed `App\Http\Middleware\AddHstsHeader` entirely (deleted the class, unregistered from `bootstrap/app.php`) per the user's explicit instruction, citing "your plaintext-API policy" — HSTS is host-wide, not path-scoped, so advertising it on any HTTPS response would tell HSTS-aware browsers to force HTTPS for the *entire* hostname, including `/api/v1/*`, which must stay reachable over plain HTTP for legacy Halo game-server clients. This should have been the decision when SEC-02 first shipped (2026-07-07) — the original entry added HSTS "deliberately no `includeSubDomains`/`preload`" but didn't weigh the host-wide-vs-path-scoped distinction at all, which is the actual reason it doesn't belong on this specific host while the legacy API lives here. No test covered either the trust-header regression or the HSTS removal (no existing test exercises scheme/host/port resolution or asserts on `Strict-Transport-Security`) — both fixes verified live only, same as the original SEC-02 close-out and its immediate predecessor entry above.

Verified live after both corrections: the exact spoofed-header `curl` now redirects to the real domain instead of `evil.example`; the normal HTTP→HTTPS redirect and `/api/v1/*` plain-HTTP access are both unaffected; `curl -sSI https://redesign.hrl.effakt.info/` no longer includes `Strict-Transport-Security` in its response headers. Full suite unchanged: 120/142 passing, 305 assertions, same pre-existing 21 `LapSubmissionTest` failures; `vendor/bin/pint --dirty` and `vendor/bin/phpstan analyse` both clean.

## PERF-01 — Home page recomputation, measured and fixed (2026-07-08)

Measured before touching anything, matching this project's own "use actual evidence, not guesses" convention: `\DB::listen()` around a real `Home::mount()` call counted **132 SQL queries and ~2.3s** at current real scale (~1,668 laps, 823 players) — consistent with a live TTFB complaint of ~1.8–2.0s at what's still a genuinely small dataset. Read `Home.php`, `GlobalRanking.php` to find the actual cause rather than guessing: `fastestImprovements()` and `achievements()` each ran their own identical query for the same 7-day recent-laps window (a duplicate query, easy zero-risk win), and — the dominant cost by far — both looped over every recent lap calling `GlobalRanking::forPlayer()`/`GlobalRanking::mapRank()` with no `excludeLapId`, i.e. asking for that lap's player's *current* rank. `GlobalRanking::scores()` (which both `forPlayer()` and the global side of `mapRank()` ultimately run) has no way to answer "what's just this one player's rank" without computing the ranking for *every* player — it queries every lap in the whole table (not just the recent window), eager-loads `player`/`map`/`server` on all of them, and re-groups/re-ranks everything in PHP, every single call. With 13 recent laps in the window and several call sites hitting this per lap, that's the whole table's ranking recomputed from scratch a few dozen times over, to extract one player's row each time.

Fixed in `app/Livewire/Home.php` without touching `GlobalRanking` itself (a shared, heavily-used, already-tested calculator — changing its public contract to serve this one caller felt like a bigger, riskier change than the actual problem, which is Home calling it far more than it needs to, not `GlobalRanking` being slow per call):

1. `loadHighlights()` now queries the shared 7-day `$recentLaps` window exactly once and passes it into both `fastestImprovements()` and `achievements()`, instead of each querying it independently.
2. `loadHighlights()` also calls `GlobalRanking::scores()` exactly once (the "current", non-excluded ranking) and passes the result down as `$currentScoresByPlayer` (keyed by `playerId`). Every "current rank" lookup that used to call `GlobalRanking::forPlayer()`/`GlobalRanking::mapRank()` fresh now reads out of that one shared array instead — a new `Home::currentMapRank()` helper finds a player's per-map rank in their already-computed `perMap` breakdown rather than re-querying `mapRank()`.
3. The genuinely-per-lap `excludeLapId` calls (each lap's "before" state is a real, distinct computation — can't be shared across *different* laps) are still made fresh, since they can't be precomputed once. But `fastestImprovements()`'s `topEntryCandidates` and `achievements()`'s third loop both independently ask `GlobalRanking::forPlayer($playerId, excludeLapId: $lapId)` for the exact same (player, lap) pairs whenever a recent lap's player appears in both blocks' candidate lists (both dedupe the same shared `$recentLaps` by `player_id`) — added `Home::forPlayerBeforeLap()`, a per-request memoization keyed by `"{$playerId}:{$lapId}"`, shared across both methods via a private instance property (`$forPlayerBeforeLapCache`, reset fresh on every request since Livewire constructs a new component instance per request). Used `array_key_exists`, not `??=`, since a real `null` result (the player has no qualifying lap after exclusion) must still count as "already computed," not be treated as "not cached yet" and recomputed every time.

Verified the memoization itself works in isolation (reflection-invoked the private method twice with identical arguments, confirmed the query count didn't move on the second call) — but it happened to yield zero measurable savings on this specific fix's before/after full-page measurement, because the current 13-recent-lap dataset doesn't actually have a player appearing in both blocks' candidate lists yet. Kept anyway: it's correct, adds no complexity of concern, and is now in place for whenever that overlap does occur.

**Result**: 132 → 94 queries (**-29%**), ~2.3s → ~1.5s (**-35%**) for the same `Home::mount()` call at the same real data volume. Confirmed against the live site too: `curl -sS -o /dev/null -w "%{time_starttransfer}"` against `https://redesign.hrl.effakt.info/` measured ~1.5s, down from the audit's reported ~1.8–2.0s baseline. Full suite re-run: 120/142 passing (same pre-existing 21 `LapSubmissionTest` failures, unrelated), `HomeTest.php` (7 tests) unaffected; `vendor/bin/pint --dirty` and `vendor/bin/phpstan analyse` both clean.

**Deliberately not done in that pass**: the fuller recommendation this addressed also suggested caching/materializing the home aggregates entirely, not just deduplicating redundant computation within one request — a real architectural decision (cache store, TTL, invalidation) with genuine design freedom, so it wasn't picked unilaterally, consistent with this project's standing rule to ask before choosing a mechanism for anything phrased open-endedly.

## PERF-01's caching fix had two real bugs of its own; both caught before shipping, same day

The user reviewed the caching implementation above (not a live-traffic re-audit this time, a code review) and found two genuine defects plus one factual error in the listener's own documentation:

1. **Cache stampede**: `Cache::remember()` has no locking. On a cache miss (right after every invalidation), every concurrently-connected visitor would independently run the full ~1.5s/94-query rebuild at once — which is exactly the "every live lap triggers this for every visitor" problem the cache exists to solve, just moved to the moment immediately after invalidation instead of eliminated.
2. **Stale write after invalidation (the more serious one — silent data staleness, not just a perf regression)**: if a new lap arrives *while* another request's cache-miss rebuild is still in flight, the listener's `Cache::forget()` clears the key — but the in-flight rebuild, computed against the *old* data, finishes afterward and writes its now-stale result right back under that same key via `Cache::remember()`, undoing the invalidation. Visitors would then see stale highlights (missing the new lap) for up to the full 10-minute TTL, with no way to tell anything was wrong.
3. **Documentation bug**: the listener's docblock claimed it "runs inside `ProcessNewLap`'s own execution (already queued)" — checked `ProcessNewLap` directly and confirmed this is factually wrong. `ProcessNewLap` deliberately does **not** implement `ShouldQueue` and is called synchronously (`new ProcessNewLap(...)` then `->handle()` directly, never `::dispatch()`) — its own docblock already documents this as a deliberate fix for a past bug ("it declared `ShouldQueue` but always called `->handle()` synchronously, never `::dispatch()`, because the game server needs its leaderboard position back in the same [request]"). The listener's *conclusion* (don't make this listener `ShouldQueue` either) was still correct, just for the wrong stated reason — fixed by correcting the docblock to say what's actually true: `ProcessNewLap` runs synchronously in the webhook's own HTTP request, so this listener already runs inline there too, and queuing it separately would only delay invalidation for no benefit.

Recommended fix, implemented as suggested: a generation key plus a distributed lock with a cache recheck.

**Generation key** (`Home::GENERATION_KEY = 'home:highlights:gen'`, a monotonic counter): `Home::CACHE_KEY` is now a *prefix*, not a literal key — every read computes the actual data key as `CACHE_KEY:{generation}`, reading the generation *before* checking the cache. `InvalidateHomeHighlightsCache` increments the generation instead of forgetting the data key directly. This closes the stale-write race completely: a rebuild that started under generation 5 can only ever write to `home:highlights:5`; once a new lap bumps the generation to 6, every subsequent read asks for `home:highlights:6` instead, so the late write to `:5` lands on an abandoned key nobody reads again (harmless — it just expires via its own TTL) rather than overwriting the current one. A real, non-obvious gotcha caught while implementing this: `Cache::increment()` on the `database` cache store (this app's real driver, per `.env`) returns `false` and does nothing if the key was never set — it does **not** auto-initialize to 0 first, unlike some other drivers (confirmed by reading `Illuminate\Cache\DatabaseStore::increment()`/`::add()` directly, not assumed). Fixed by calling `Cache::add(Home::GENERATION_KEY, 0, null)` (atomic "set if missing," via `add()`'s `insertOrIgnore`, safe under concurrent callers) immediately before every `increment()` call.

**Distributed lock + cache recheck** (`Cache::lock()`, using this app's existing `cache_locks` table): on a cache miss, a request acquires a lock keyed to that specific generation's data key before rebuilding, rather than rebuilding unconditionally. Concurrent requests block for up to 5 seconds waiting for the lock; once acquired, the lock holder re-checks the cache first (double-checked locking) since whoever held the lock immediately before likely already populated it while this request was waiting — only a genuine cache-miss-after-acquiring-the-lock actually calls `computeHighlights()`. If the wait times out (`LockTimeoutException` — the holder is still running, or its process died mid-computation without releasing), the request computes directly rather than failing outright: slower than a cache hit, but never worse than not having caching at all.

Verified the lock-wait/fallback path directly, not just by reading the code: manually held the generation-0 lock in a separate `tinker` session, then called `Home::mount()` and confirmed it waited the full ~5-second `LOCK_WAIT_SECONDS` window before falling back to a direct (correct, fresh) computation — 6.2s elapsed, matching the 5s wait plus real computation time. Verified the stale-write-race fix with a new Pest test (`it('never serves a stale write made against an old generation after invalidation')`) that plants a stale write under the pre-invalidation generation directly (true multi-process concurrency isn't reproducible in a synchronous test process, but this exercises the actual invariant the generation key provides — a write against an old generation can never become what a current read sees) and confirms a fresh visitor still gets correct, current data. True concurrent-stampede behavior (multiple simultaneous requests actually sharing one rebuild) is verified by code review/architecture reasoning and the manual lock-holding test above, not an automated test — matching this project's existing precedent for the similar "can't fully exercise real concurrency/WebSocket transport in Pest" limitation noted elsewhere (see `docs/testing.md`).

**Result**: cold path now costs 101 queries/~1.35s (was 94/~1.5s — the small increase is the generation read plus lock acquire/release against `cache_locks`, both cheap); warm path is 2 queries/<1ms (was 1 — the extra query is the generation read itself). Live site TTFB unaffected, still ~37–48ms across repeated checks. Full suite: 145 tests (was 144, +1 new race test), 123 passing, 311 assertions — same pre-existing 21 `LapSubmissionTest` failures, unrelated; `vendor/bin/pint --dirty` and `vendor/bin/phpstan analyse` both clean.

## SEC-04 — checkpoint counts learned per map, forked on mismatch (2026-07-08)

Discussed with the user before building, since the finding's own recommendation ("cap split count to the protocol maximum, enforce sane numeric ranges") left real design freedom: what's the actual ceiling, and what happens when a submission doesn't match it? Checked real data first rather than guessing: every map's recorded `checkpoint_id`s form a stable, contiguous `1..N` set across every lap ever submitted for it (Bloodgulch always 1-5, Death Island always 1-14, confirmed by grouping `lap_time_splits` by map), and real submissions max out at 14 checkpoints — a map's checkpoint layout is a fixed, physical fact, learnable from real submissions rather than needing an arbitrary blanket number.

The user's design, given directly: maps are only ever added, never redesigned in place, so a checkpoint-count mismatch should be treated as a genuinely different map sharing the same underlying map file — forked into its own `{map_name}-splits-{count}` identity — rather than rejected or silently mixed into the original leaderboard. For a brand-new, never-before-seen checkpoint count, cap it at a sane ceiling (suggested: ~20) so an absurd claim (thousands of fake checkpoints) can't create an equally absurd map row, and make that ceiling config-driven so it's easy to raise later without a code change.

Implemented as two layers: (1) `config('webhook.max_checkpoints')` (default 20, `WEBHOOK_MAX_CHECKPOINTS` env-overridable) enforced as a flat `max:` rule on `splits` in `StoreLapTimeRequest` — a hard, protocol-wide ceiling independent of which map; (2) a new nullable `checkpoint_count` column on `maps` (migration `add_checkpoint_count_to_maps_table`, run against the real dev MySQL DB, not just a fresh-environment migration — this is a genuine schema addition, not a mirror of pre-existing structure), learned from the first split-bearing submission for a given `map_name` and enforced after that by a new `ProcessNewLap::resolveMap()`, replacing the previous plain `Map::firstOrCreate()` call. A submission with no splits (the common case — only ~4% of real laps have any, see database.md) never touches this at all, always resolving to the plain map.

On a mismatch, `resolveMap()` doesn't reject the lap or touch the original map's row — it resolves (or creates) a second `Map` identity named `{map_name}-splits-{count}`, with its own `label` (`"{original label} ({count} CP)"`) and its own `checkpoint_count` baseline, and the lap gets recorded against that instead. Worth noting explicitly since the user referenced `race_type` as the precedent for this naming style: `race_type` doesn't actually create a separate `Map` row today — `ProcessNewLap::mapLabel()` only changes the *display label* on the same `name`/id, so "Any Order" and normal-order laps on the same map currently share one leaderboard. A checkpoint-count fork is a real, distinct `Map` row, not just a label variant — a deliberate, stronger form of separation than `race_type` gets, since a different checkpoint layout is an actually different course, not a mode of the same one.

`Map` needed a `checkpoint_count` integer cast (`casts(): array`, matching `Server`'s existing pattern) — without it, the column comes back from MySQL as a string, and `resolveMap()`'s `===` comparison against a real PHP `count()` int would silently never match, turning every submission into a spurious fork. Caught and verified directly in `tinker` against the real dev DB before writing any test: confirmed the cast returns a real `int` and that `5 === 5` (int) held, not `"5" === 5` (which is `false` in PHP).

Added 5 new tests to `LapSubmissionTest.php`: learning the baseline from a first split-bearing submission, reusing the same map identity on a later matching submission, forking into a real second `Map` row on a mismatch (asserting both maps' `checkpoint_count`/`label` and which lap landed under which `map_id`), a splitless submission never establishing or disturbing a baseline, and the flat ceiling rejecting an over-limit submission with a `422`. 145 → 150 tests, 311 → 335 assertions — same pre-existing 21 `LapSubmissionTest` failures (a `submission_id`-required mismatch unrelated to this work, traced to the real `.env`'s `WEBHOOK_HRL_VERIFY_ENFORCE` now being `true` with no `.env.testing` to isolate tests from it) plus the same 1 pre-existing error, both unchanged; new tests explicitly set `config(['webhook.hrl_query.enforce' => false])` so they don't depend on that ambient value. `vendor/bin/pint --dirty` and `vendor/bin/phpstan analyse` both clean.

## SEC-04, numeric-range portion closed same day

The user asked for recommendations on the remaining part of the finding (`player_time`/split `duration`/`startTime`/`endTime` upper bounds) before moving on. Queried real data for each field rather than picking round numbers: `player_time` ranges 15.3s-1170.9s across every real lap; split `duration` ranges 1.17s-182.2s, and a direct query confirmed **zero real splits ever exceed their own lap's `player_time`** (`whereColumn('duration', '>', 'time')` on a join, count 0) — a genuine invariant, not a guess.

`startTime`/`endTime` turned out to be the one surprising result, and changed the recommendation: querying their real distribution showed two very different populations — small values that look lap-relative (0-3600ish) and a large fraction (266 of 469 real split rows) of large values (`993881.17`, `612271.29`, `247805.9`, ...) that look like an absolute server-clock/uptime counter instead, plus a literal `999999.99` value repeated in 34 rows (a sentinel, not a real timestamp). Grepped the codebase for where these two fields are actually consumed: nowhere in leaderboard or comparison logic — only `duration` feeds `LapTimeSplit::compare()`; `startTime`/`endTime` are stored and folded into `LapSubmissionHash` but otherwise inert. Recommended against a tight bound on these specifically, since it would reject real, already-accepted submissions for a field nothing functionally depends on — presented this finding to the user before implementing, since it reversed the default assumption ("bound everything") the original finding implied.

Implemented, agreed by the user (an explicit "yes" to the three-part recommendation):

1. `player_time` gets `lte:config('webhook.max_lap_time_seconds')` (default 3600s, env-overridable `WEBHOOK_MAX_LAP_TIME_SECONDS`) — 3x headroom over the real max.
2. `splits.*.duration` gets `lte:player_time` — Laravel's `lte` rule supports comparing against another field by name directly (confirmed via a real `Validator::make()` call in `tinker`, not assumed: submitting `duration: 50` against `player_time: 42.5` produced "The splits.0.duration field must be less than or equal to 42.5."), so no custom closure rule was needed. A relational bound instead of a flat number, self-tuning as `max_lap_time_seconds` changes, and already true of every real split on record.
3. `splits.*.startTime`/`endTime` get only a loose `gte:0`/`lte:99999999` overflow guard — wide enough to comfortably admit the observed `999999.99` sentinel and the large absolute-clock-style values, tight enough to reject a truly pathological value (e.g. `1e300`).

Added 6 more tests to `LapSubmissionTest.php`: `player_time` rejected just over the ceiling and accepted exactly at it, a split `duration` longer than its own lap's `player_time` rejected, the real large `startTime`/`endTime` values (mirroring the actual `993881.17`/`999999.99` observed in production data) still accepted, and negative/beyond-overflow-guard values rejected. 150 → 156 tests, 335 → 351 assertions — same pre-existing 21 failures/1 error, unchanged; `vendor/bin/pint --dirty` and `vendor/bin/phpstan analyse` both clean. ## Reversal, same day: `race_type` now gets its own `Map` identity (2026-07-08)

Immediately after the follow-up above shipped, the user reversed the earlier "`race_type` is label-only" decision directly: "yes, race_type should be its own map identity." ## TEST-01 — code review found the MySQL concurrency test could pass without proving anything; three real gaps closed in turn (2026-07-09)

A review of the map-variant-cap concurrency test (added earlier this session) found it was weaker than it looked, in three progressively deeper ways — each fixed, and the fix for the third one verified to actually discriminate, not just added ceremony.

**Gap 1 — no real synchronization barrier.** `proc_open()` guarantees both worker processes *start*, not that they reach `resolveMapVariant()`'s critical section at overlapping times; a lucky scheduler could let one finish entirely before the other even started, which wouldn't exercise the race at all. Fixed with a real barrier in `resolve_map_variant_worker.php`: each worker touches its own "ready" file once booted, then waits (up to 5s) for the other's before proceeding.

**Gap 2 — the barrier's own timeout silently degraded coverage.** If the other worker never showed up within 5s, the original code proceeded to call `resolveMapVariant()` anyway — meaning a slow boot could let the test pass without ever proving the two workers raced. Fixed: a timed-out worker now writes `ERROR:barrier_timeout` and exits without calling `resolveMapVariant()` at all, caught by the test's existing `->not->toStartWith('ERROR:')` assertion.

**Gap 3 — the deepest one, and the only one that actually proves the lock matters.** Even with both workers guaranteed to *start* together, MySQL/OS scheduling could still let one worker's entire count-then-insert finish before the other even read its count — the exact same passing outcome (one OK, one REJECTED, count === cap) whether or not `lockForUpdate()` does anything. Fixed by extracting the count-read step into its own overridable `protected countExistingMapVariants()` method on `ProcessNewLap`, and having the worker inject a real 300ms delay there via an anonymous subclass — without a genuine row lock serializing the two transactions, both workers would read the same under-cap count during that window and both insert. **Verified this actually discriminates**: temporarily removed `lockForUpdate()` and reran — failed exactly as expected (`2 is identical to 1`, both workers wrongly succeeded) — restored it, confirmed green again.

**Made the discriminating power permanent, not just manually checked once.** A one-off manual edit-and-revert proves the *current* code is right but doesn't protect against a future accidental removal of the lock being missed if the harness itself ever quietly lost its ability to detect it (a tuned-down delay, a Laravel/MySQL upgrade changing transaction timing). `acquireMapVariantLock()` became its own overridable method too, and the worker script now takes a `<lockMode: locked|lockless>` argument. A new permanent test — `'proves the harness itself would catch a missing lock (negative control...)'` — runs the exact same race with a deliberately no-op lock and asserts the race *does* manifest: deterministically 2 successes and `cap + 1` variants, given the fixed delay window. If this ever started passing with 1 success instead, the harness itself would have lost its discriminating power, independent of whether `ProcessNewLap`'s real lock is intact. 205 → 206 tests.

**Also fixed, narrower**: the test now `->skip()`s with a clear message (rather than erroring) when `TEST_DB_USERNAME`/`TEST_DB_PASSWORD` aren't set, so a fresh checkout/another machine doesn't fail the default suite — noted as *conditional* coverage, not always-on, since a green `composer test` doesn't by itself prove this specific test actually ran. `composer.json`'s `setup` script now also runs `npx playwright install chromium`, since `Browser` is in `phpunit.xml`'s default testsuites and nothing previously installed it automatically.

Since `render()` runs before Livewire's own `render` hook applies the attribute's params, a `render()`-time `layoutData()` call always overrides the attribute's static `'title'`/`'description'` on the same page, without needing to touch or strip the attribute at all. Wired into all five detail components (`PlayerShow`, `ServerShow`, `ServerPlayerShow`, `MapLeaderboard`, `ServerMapLeaderboard`); the three list pages and two static pages (`opt-in`, `contact`) only needed a static description added directly to their existing `#[Layout(...)]`/`<x-layout>` usage, since their titles were already fine.

## `schedule:work` stalled silently for ~23 hours — stuck mutex lock, fixed and hardened (2026-07-10)

User reported a real server (`114.23.254.181:2302`, id 7) showing offline on Servers List despite the game server actually being up ("running, but no players online"). Investigated as a live production question, not a code review — the real IP the user typed (`.81`) didn't exist in `servers` at all; the actual row was `.181`, one digit off.

**Root cause, found by reading the actual scheduler log, not guessing**: `storage/logs` had nothing, but `../../logs/redesign-hrl-schedule.log` (the supervisor-managed `schedule:work` process's own stdout) showed "No scheduled commands are ready to run" on every single tick, indefinitely — `App\Console\Commands\RefreshLiveServerInfo` (roadmap item 19) hadn't actually executed in ~23 hours, even though `php artisan schedule:list` correctly reported it as due and the process itself was alive. Confirmed by running the command manually: it succeeded instantly (server reachable, 0 players — exactly matching the user's own observation), proving the *job* was fine; the *scheduler* wasn't invoking it.

Traced to a genuinely stuck `withoutOverlapping()` mutex: Laravel's `CacheSchedulingMutex` stores a lock row in the `cache_locks` table (not the general `cache` table — a real gotcha, an initial grep of `cache` for anything resembling a mutex found nothing) keyed as `framework/schedule-{sha1(cron+command)}` plus a per-minute suffix. Found exactly one row there, with a still-future expiration. **Restarting the `schedule:work` process (`kill` + supervisor's `autorestart`) did not fix it** — confirmed live, waited a full tick after restart, `queried_at` never moved — because the lock lives in the database, not the process's memory, so a fresh process immediately collides with the same stuck row. Deleting that one row fixed it instantly: the very next minute's tick updated the affected server's `queried_at` on its own, confirmed live via a polling watch rather than assumed.

**Fixed two ways, both discussed with the user first** (a real design/hardening decision, not something to pick solo): (1) removed `->withoutOverlapping()` from the schedule entry entirely (`routes/console.php`) — this job is idempotent (each run just re-queries a handful of servers over UDP, ~2s each) and a genuinely overlapping run is harmless, so removing the mutex removes this whole failure class rather than trying to make the lock more robust; (2) added `App\Console\Commands\CheckSchedulerHealth` (`app:check-scheduler-health`), a deliberately independent watchdog — alerts via `Log::critical()` if no server's `queried_at` has updated within 10 minutes, registered via a **real OS crontab entry**, not `Schedule::`, since a check that itself depends on the same (possibly-stuck-again, or fully dead) scheduler would never fire when it's needed most. Deliberately alert-only, not self-healing: this user's account doesn't have `supervisorctl` access to restart `schedule:work` (confirmed live, `PermissionError`), and auto-deleting `cache_locks` rows on an unattended schedule is a sharper tool than a health check should wield without a human looking.

**A real, separate discovery while adding the crontab entry**: `docs/deployment.md` had documented (2026-07-08) that this app's user had no crontab access at all. Re-tested live rather than trusting that old note — `crontab -l`/`crontab <file>` both worked without any permission issue. The original finding was accurate when made; it just was never re-checked since. Doesn't change the earlier decision to keep the *main* per-minute job on `schedule:work` under supervisor (still the better fit for a long-running per-minute cadence), but means crontab is a real, available option for anything — like this watchdog — that specifically needs to run independently of it.

## 404 error message no longer leaks the internal Eloquent class name (2026-07-14)

The user pointed out the `/api-docs` page's own example error response — Laravel's real default `ModelNotFoundException` message, `"No query results for model [App\Models\Map] bloodgulch2"` — exposes app-internal namespace/class structure to a public API consumer for no benefit, and asked for it to just say `"No query results for map bloodgulch2"`.

**First attempt used the wrong hook, caught by the tests before shipping**: registered `$exceptions->render(function (ModelNotFoundException $e, Request $request) {...})` in `bootstrap/app.php`, expecting it to intercept the message before the default renders. It never fired — the existing 404 tests (strengthened with an exact-message assertion specifically to catch this) kept showing the unmodified default message. Read `Illuminate\Foundation\Exceptions\Handler::render()` to find out why: it calls `prepareException($e)` — which unconditionally rewrites every `ModelNotFoundException` into a `NotFoundHttpException` (carrying the original message forward) — *before* checking any registered `render()` callback, so a callback typed for `ModelNotFoundException` specifically can never match by the time callbacks are considered.

**Fixed with `$exceptions->map(ModelNotFoundException::class, ...)` instead** — `map()` runs earlier than `prepareException()`'s hardcoded rewrite, so returning an already-constructed `NotFoundHttpException` with the clean message here passes through unchanged (it's no longer a `ModelNotFoundException` by the time the automatic rewrite check runs). Message format: `"No query results for {$model} {$ids}"`, where `$model` is `Str::snake(class_basename($e->getModel()), ' ')` — not a flat `strtolower()`, which would turn `LapTime` into the run-together `laptime` instead of the more readable `lap time` (`Map` is unaffected either way, matching the user's own example exactly). Applies globally (not path-restricted to `/api/*`) since it's purely a message-content change, not a behavior change — a web-route 404 renders its own generic HTML error page regardless of the underlying exception message, so this can't affect anything there. Updated the `/api-docs` example and 3 existing `ApiTest.php`/`RoutesTest.php` assertions from a bare `assertNotFound()`/backslash-presence check to the exact real message.
