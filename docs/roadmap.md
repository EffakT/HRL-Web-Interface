## Open questions / not yet decided

- **Duplicate player display names, different real identities** — discovered 2026-07-06 while debugging a Homepage highlight: 8 different `Player` rows are all named exactly "Nuevo001" (confirmed via 8 distinct `hash` values — genuinely different accounts, not a data bug). Inherent to the old app's hash-based player identity — the same alias can end up on multiple accounts if a player's hash ever changes. No current UI disambiguates same-named players (e.g. a `#123` suffix, or surfacing the hash/first-seen date). Hasn't caused a visible bug yet, but is a real risk for any feature that names two different players in one sentence (Homepage highlights, potentially others). Related to the already-deferred claim-code ownership system — same "player identity" territory, not a quick fix. See [decisions.md](decisions.md).
- **Global/Server Score: sum vs. average** — real data surfaced a case where summing per-map points lets breadth of participation out-score a flawless record on fewer maps (see [decisions.md](decisions.md)). Both are now implemented and switchable via `config('ranking.global_score_variant')` / `GLOBAL_RANKING_VARIANT` (`sum` is the default). No decision yet on which one this project should settle on long-term — currently a live comparison tool, not a resolved question.

- ~~Should the rebuilt webhook log every lap attempt, or keep the old app's behavior of only inserting on a personal-best improvement?~~ — **decided 2026-07-06: log every attempt.** See [decisions.md](decisions.md) and [database.md](database.md)'s "Webhook → job flow" section.
