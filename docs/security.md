# Security

No formal security review has been done yet. Phase 2 real-data integration is underway, but write endpoints and authentication are not built. This doc tracks what's decided, what Laravel/Livewire give for free, and what's still open.

## Secrets & credentials

- DB credentials, API tokens, etc. live in `.env` — never commit it, never hand it to an AI agent or third party directly.
- The Git repository was initialized only after verifying `.gitignore` excludes `.env`, dependency directories, compiled frontend assets, and Laravel key files. Re-check the staged file set before the first commit and push.
- The configured GitHub repository is public. No push has occurred yet; the staged index must pass a secret-file/content review before the initial commit. GitHub access should be provided via an account SSH key or repository-scoped deploy key, never by placing a personal access token in project files or chat transcripts.
- A dedicated Ed25519 deploy key was generated outside the application Git root at `../../.ssh/hrl_web_interface_v2` (the hosting account's home directory), and this repository alone is configured to use it. Only the `.pub` half belongs in GitHub; the private half remains server-local and must never be committed or pasted elsewhere.
- **Open item**: confirm the dev DB user has read-only or otherwise scoped privileges, as originally intended. Not yet verified — see [roadmap.md](roadmap.md).
- Laravel Boost (AI tooling) should only ever be pointed at the dev DB copy, never prod credentials.

## Framework defaults already in place

- CSRF protection on all state-changing routes (Laravel default middleware).
- Blade `{{ }}` output escaping used throughout — no raw `{!! !!}` in the codebase currently.
- Livewire's own checksum/rate-limiting on component actions (this is what depends on the `cache` table — see [decisions.md](decisions.md) for the incident where a missing `cache` table caused every `wire:click` to fatal-error).

## Known issues to fix from the old app

- Old API requires a per-user token generated in profile; auth/token strategy for the new API isn't decided yet.

### SEC-01 — HRL query verification (added 2026-07-07)

The webhook can't use TLS/HMAC on the game-server side — the Lua/SAPP script that submits laps isn't part of this repo and isn't known to have that capability. Instead, `App\Helpers\LapSubmissionVerifier` (used by `LapSubmissionController::store()`) cross-checks every submission against a **live UDP `\query` response from the same ip:port the HTTP request came from**, reusing the protocol already built for roadmap item 19 (`App\Helpers\QueryServer`, `App\Models\Server`'s live-query fields). This binds the HTTP submission to a game server that is actually running an HRL-aware script, currently on the submitted map, with the submitting player online — without needing any cryptography on the Lua side.

The Lua script must publish these query_add fields (SAPP's existing `query_add <key> <value>` capability) alongside the standard ones (`hostname`, `mapname`, `player_0..N`, etc.):

```text
hrl_enabled = 1
hrl_protocol = 1
hrl_token = <short-lived random value, rotated periodically>
```

...and include the same `hrl_token` value as an `hrl_token` field in the HTTP submission body. `LapSubmissionVerifier::verify()` then requires, in order: a successful UDP query (one retry on failure), `hrl_enabled === '1'`, a supported `hrl_protocol` (config `webhook.hrl_query.supported_protocol`), a matching `hrl_token`, `mapname` matching the submitted `map_name`, and the submitted `player_name` appearing among the response's `player_0`/`player_1`/... values. Any failure returns a structured reason (`udp_timeout`, `missing_hrl_marker`, `protocol_unsupported`, `token_mismatch`, `map_mismatch`, `player_not_online`) that's logged.

**Rollout note — `enforce` defaults to `false`** (`config/webhook.php`, env `WEBHOOK_HRL_VERIFY_ENFORCE`): a failed verification is logged but the lap is still recorded. This is deliberate, not a bug — real game servers are still running Lua scripts written before this feature existed, and none of them publish `hrl_*` fields yet. Flipping `enforce` to `true` before every active server's script is updated would silently reject every legitimate submission. Turn it on only once the rollout is confirmed complete (watch the warning logs for `missing_hrl_marker` from known-good servers dropping to zero).

**Lua/SAPP client updated (2026-07-07)** — `hrl.lua` (the actual deployed script, not a reconstruction) now generates a 32-hex-char `hrl_token`, publishes it plus `hrl_enabled`/`hrl_protocol`/`hrl_token_prev` via SAPP's `query_add <key> <value>` console command (invoked from Lua via `execute_command`, since `query_add` itself has no direct Lua binding — confirmed against [SAPP's own docs](https://halo-sapp.readthedocs.io/en/latest/commands/query.html), which also confirms `query_add` overwrites an existing value on repeat calls, exactly what token rotation needs), rotating every 5 minutes. Every lap submission now also includes a fresh `submission_id` (SEC-01 audit follow-up's idempotency key) and the current `hrl_token`. This is still a rollout in progress, not a flip-the-switch moment — `enforce` should stay `false` until every *other* active server's script is confirmed updated too (this repo only controls the webhook side; each game-server host needs its own script deployed).

**Three real fixes from `SITE_AUDIT.md`'s dedicated Lua review (2026-07-07, see [decisions.md](decisions.md) for full detail)**: rotation now compares `os.time()` (wall clock) instead of `$ticks` (which SAPP resets on every map change, previously able to stall rotation indefinitely); `OnScriptUnload()` now clears the published `hrl_*` fields via `query_del` so a stopped script can't keep claiming HRL is active; and `LapSubmissionVerifier`'s player-online check now converts the UDP response's raw Windows-1252 `player_N` bytes to UTF-8 before comparing against the already-UTF-8 submitted name — without this, any non-ASCII player name (real ones exist in this community, e.g. `"GåþøFêîk¬£Q"`) would have been silently rejected as `player_not_online`.

**What this doesn't protect against**: a server operator controls their own query responses and could fabricate activity *on their own server* (they already could, by running a real client and legitimately racing). An on-path attacker between this app and a legitimate server could still observe or alter plaintext HTTP/UDP traffic — this is verification of "a live HRL-aware game server is actually there," not encryption or non-repudiation. The audit's alternative (a local TLS-terminating relay per game-server host, HMAC/mTLS) would close that gap but requires real infrastructure work outside this repo on every game-server host; not pursued here.

A short-window idempotency guard (`Cache::add`, `config('webhook.duplicate_window_seconds')`, default 10s) runs independently of HRL verification — see "Follow-up hardening" below for how this evolved from a plain duplicate-rejecting guard into a real idempotent-replay mechanism.

#### Follow-up hardening (2026-07-07, same day)

A second review of commit `0f2362a`'s implementation (recorded directly in `SITE_AUDIT.md`) found five real gaps before `enforce` could ever safely flip to `true`. All five are fixed in the same day's follow-up commit — see [decisions.md](decisions.md) for the full writeup:

1. **Rate-limit bypass via port rotation** — the webhook limiter's only key was `ip:port`, and `port` is caller-supplied and unverified at the rate-limit layer, so an attacker could rotate it on every request for a fresh 120/min allowance each time (while still forcing a real UDP query attempt per request). Fixed: `RateLimiter::for('webhook', ...)` now returns **two** limits that both apply — the existing per-`ip:port` limit, plus a coarser per-IP ceiling (`config('webhook.rate_limit.per_ip_per_minute')`, default 600/min) that port rotation can't evade.
2. **No token rotation grace or replay protection beyond exact-value dedup** — `LapSubmissionVerifier` now also accepts an optional `hrl_token_prev` query field (the Lua script's previous token, if it rotates them), so a submission racing a rotation boundary isn't wrongly rejected. Real replay protection is handled by point 3's idempotency key, not by the token itself — see the class docblock for why the token isn't meant to be a durable secret.
3. **Retry poisoning — a legitimate retry got an error, not its original result.** The original `Cache::add()` guard reserved the dedupe key *before* verification/processing; if a legitimate retry arrived (e.g. the client never saw its own successful HTTP response), it got a bare `409 duplicate_submission` instead of the original success payload, and a pre-commit failure left the reservation stuck for the rest of the window even though nothing was recorded. Fixed: the cache now tracks a real state machine (`processing` → `done`, storing the actual response body/status) — a repeat of the same key while `done` replays the original response verbatim; a `Throwable` during processing releases (`Cache::forget`) the reservation instead of leaving it stuck. Also accepts an optional client-generated `submission_id` field as the idempotency key when present, falling back to the previous content-hash approach otherwise.
4. **Duplicate UDP query on every accepted submission** — `LapSubmissionVerifier::verify()` already performs a live UDP query; `ProcessNewLap::resolveHostname()` was doing a second, independent one for the exact same ip:port. Fixed: `verify()` now returns the raw `response` array, and `LapSubmissionController` passes it into `ProcessNewLap`'s new `$liveQueryResponse` constructor parameter — `resolveHostname()` reuses it instead of querying again. (When verification is disabled, or its own query failed, `$liveQueryResponse` is `null` and `ProcessNewLap` falls back to its own query, unchanged from before.)
5. **Loose player-field matching** — `player_not_online` matched any query key merely *starting with* `player_`, which a future query extension (e.g. a hypothetical `player_count`) could accidentally satisfy. Fixed: matches only `/^player_\d+$/`.

**Still open, deliberately not built**: a numeric token *maximum age* check. Doing that meaningfully needs the Lua script to publish an issue timestamp the server can compare against its own clock, which is more surface on the Lua side than the current/previous-token grace already implemented — revisit only if the current/previous fallback proves insufficient in practice.

#### Second follow-up hardening (2026-07-07, same day, commit `eb28327` reviewed again)

A third audit pass reviewed `eb28327` and found five more real gaps — two of which involved genuine open-ended design decisions (exact rate-limit numbers, how much idempotency infrastructure to build), so those were confirmed with the user directly before implementing, rather than decided solo. Full technical detail below; reasoning in [decisions.md](decisions.md).

1. **Cross-server `submission_id` collision (High)** — when a client sends its own `submission_id`, the idempotency cache key was JUST that value, with no ip/port namespace. Two different game servers generating similar counters (e.g. both starting from 1) could collide and one would receive the other's cached response. Fixed: the idempotency key is now always `lap-submission:{ip}:{port}:{submissionKey}`, whether `submissionKey` is a real `submission_id` or the content-hash fallback. Also added a `min:8` validation rule on `submission_id` as a minimum-entropy floor.
2. **Idempotency wasn't durable (High)** — the cache is an optimization, not a guarantee; if it's ever lost (app restart, cache eviction, a very late retry past the retention window) nothing stopped a genuine duplicate lap from being recorded. **User's call, given two schema options**: a simple unique column on `lap_times` itself, rather than a separate claims table. Added a nullable `submission_id` string column + a unique index on `(server_id, submission_id)` (multiple NULLs allowed, so laps without one are unaffected) via a real migration. `ProcessNewLap::handle()` now catches the resulting `QueryException` on a genuine collision and replays a best-effort response built from the already-recorded lap's current state (not a fabricated success) — see `replayDuplicateSubmission()`.
3. **UDP exhaustion still practical at the initial 600/min per-IP ceiling (High)** — the audit calculated that forcing two 1-2s UDP timeouts per request could still occupy ~40 PHP workers from one source even under the reduced port-rotation-proof ceiling. **User's call, given two options**: build a two-tier system, not just lower the flat number. A source starts in a strict "unverified" tier (30/min per IP, 15/min per ip:port, burst 2/sec) and only earns a much more generous "verified" tier (300/min, 120/min, burst 10/sec) after a request from that exact ip:port has actually passed HRL query verification — cached at `LapSubmissionVerifier::verifiedMarkerKey()` for 5 minutes. Verification still runs on every single request regardless of tier; the marker only ever changes how much traffic is allowed. The coarse per-IP ceiling applies at both tiers specifically so running many ports can't bypass it even once verified. Also reduced `hrl_query.timeout_seconds` from 2 to 1. **Not implemented**: nginx/PHP-FPM concurrency/timeout limits — that's deployment/infra configuration outside this Laravel repo, flagged as a prerequisite rather than something this codebase can enforce.
4. **10-second reservation/result window was too short for one purpose and too long for another (Medium)** — a single shared TTL couldn't simultaneously (a) outlast a slow in-flight request without letting concurrent duplicates through, and (b) outlast how long a real client might wait before retrying. Split into `processing_reservation_seconds` (30s — generous margin over the worst-case UDP+DB round trip) and `result_retention_seconds` (300s — "several minutes," per the audit).
5. **Duplicate UDP query on the failure path (Medium, a residual case of the first follow-up's point 4)** — when verification's own query totally failed (`udp_timeout`), `ProcessNewLap` would still attempt its own separate query against the same already-unresponsive ip:port. Fixed by giving `$liveQueryResponse` a real tri-state: `null` (verification didn't run, try normally), `false` (verification tried and got nothing, don't retry), or the actual response array.

New tests: `LapSubmissionVerifierTest.php` gained cases for `verifiedMarkerKey()` and the `false` response state; `LapSubmissionTest.php` gained cases for cross-server non-collision, the durable DB constraint surviving a cache flush, and the tiered rate limiter's upgrade behavior. 125 → 129 tests, 308 → 321 assertions.

#### Third follow-up hardening (2026-07-07, same day, commit `a9605d1` reviewed again)

A fourth audit pass reviewed `a9605d1`, confirmed the migration/index/data-integrity state directly against the staging deployment, and found four more real gaps.

1. **Verified-tier rate-limit marker wasn't revoked on failure (High)** — a source could pass verification once, then stop answering UDP entirely, and keep the generous "verified" rate-limit tier for up to the marker's full 5-minute TTL while forcing timeout work on every request. Fixed: `LapSubmissionController::process()` now calls `Cache::forget()` on the verified marker immediately whenever verification fails, for any reason — the current failing request may still consume whatever allowance was in effect when it started, but every request after it immediately drops back to the strict "unverified" tier.
2. **`servers` had no unique `(ip, port)` constraint (High)** — durable lap idempotency (see the second follow-up) is scoped by `server_id`, but `Server::firstOrCreate()` itself still wasn't atomic: two concurrent first-ever submissions for a brand-new ip:port could both pass the SELECT before either INSERT committed, creating two distinct `Server` rows for the same real game server. This one took three attempts to get right, all recorded in the `add_unique_ip_port_index_to_servers_table` migration's docblock: a plain `unique(ip, port)` would permanently block that ip:port from ever being reused after the server is archived (soft-deleted); `unique(ip, port, deleted_at)` directly doesn't work because SQL's `NULL != NULL` means the constraint silently never fires between active rows at all; and a generated column referencing the row's own `id` for the archived branch is rejected outright by MySQL (error 1364/3109, generated columns can't reference an `AUTO_INCREMENT` column) — this app's real database is MySQL, not just the SQLite used for tests. The version that shipped: a generated `active_since` column, `COALESCE(deleted_at, '2000-01-01 00:00:00')` — every active row collapses to the same sentinel so active duplicates still collide correctly, while a soft-deleted row keeps its own real `deleted_at` timestamp so archived rows (and a later legitimate reuse of that ip:port) don't collide with anything. `ProcessNewLap::handle()` now retries the whole transaction once on a `UniqueConstraintViolationException` that isn't a `submission_id` collision — the concurrent request has already created the row by the time of the retry, so `firstOrCreate()`'s SELECT finds it.
3. **Duplicate-detection was too broad (Medium)** — the previous version checked a generic SQLSTATE code (`23000`), which covers *any* integrity-constraint violation on some drivers, including a foreign-key failure — a real bug elsewhere could have been misreported as a safe-to-replay duplicate and answered with a false `200`. Fixed: catches Laravel's specific `UniqueConstraintViolationException` (not the generic `QueryException`), and uses its `$columns`/`$index` properties to confirm WHICH unique constraint fired — `submission_id`-related columns/index name means a lap duplicate; anything else is treated as the `servers` race in point 2 and retried instead.
4. **A reused `submission_id` with different lap content silently replayed the old response (Medium)** — nothing distinguished a genuine retry (identical content) from a bug or a colliding ID (different content) once a `submission_id` had already succeeded once. Fixed: the cache entry now also stores a canonical content hash alongside the response; a repeat of the same `submission_id` with a *different* hash returns `409 idempotency_conflict` instead of replaying stale data. Also, per the audit's forward-looking recommendation, `submission_id` is now `required` (not just `nullable`) whenever `webhook.hrl_query.enforce` is on — at that point every real submission is already expected to come from an HRL-aware script, so requiring its idempotency key too costs nothing extra.

New tests: `LapSubmissionTest.php` gained cases for the marker-revocation behavior, the `servers` unique constraint (including that a soft-deleted server's old ip:port stays reusable), the `idempotency_conflict` response, and `submission_id` being required under enforcement. The true concurrent-request race in point 2 isn't independently reproducible in a single-threaded Pest test — covered by the schema-level constraint test plus code review instead. 129 → 133 tests, 321 → 331 assertions.

#### Fourth follow-up hardening (2026-07-07, same day, commit `c529742` reviewed again)

A fifth audit pass reviewed `c529742`, inspected the deployed MySQL schema directly (confirmed the generated `active_since` column, the `servers_ip_port_active_since_unique` index, no duplicate active identities, and that an archived ip:port stays reusable), and found three more real gaps — all Medium/Low, none blocking rollout by themselves.

1. **Durable payload-conflict detection was cache-only (Medium)** — the content hash used to catch a reused `submission_id` with changed content (third follow-up, point 4) only existed in the 5-minute idempotency cache entry. After that cache entry expired, was evicted, or the app restarted, reusing the same `(server_id, submission_id)` with different content hit the database's unique constraint instead, and `ProcessNewLap::replayDuplicateSubmission()` had no way to tell the new payload apart from a genuine retry — it just replayed the old lap as a success. Fixed: added a nullable `submission_hash` column to `lap_times` (`add_submission_hash_to_lap_times_table` migration), populated on every insert. `replayDuplicateSubmission()` now compares the incoming request's fingerprint against the stored one and throws a new `App\Exceptions\LapSubmissionConflictException` on a mismatch, which `LapSubmissionController` catches and turns into the same `409 idempotency_conflict` response the cache-based check already returns — so the two layers now agree even once the cache copy is gone. Rows recorded before this column existed have `submission_hash = null`, treated as "nothing to compare against" rather than a mismatch.
2. **The canonical content hash omitted material fields (Medium)** — it covered player hash, map name, lap time, token, and splits, but not `race_type` or `player_name`; two submissions that genuinely differed only in those fields would have hashed identically. The splits array also hashed positionally, so two payloads with the same splits in a different order would incorrectly appear to differ. Fixed: extracted the hash into a shared `App\Helpers\LapSubmissionHash::compute()`, used by both `LapSubmissionController` (cache key/conflict check) and `ProcessNewLap` (durable `submission_hash`) so the two layers can never disagree on what counts as "the same lap." It now includes `race_type` and `player_name`, and normalizes `splits` by sorting on `checkpoint_id` before hashing so equivalent payloads submitted in a different split order still match.
3. **Server-identity-race classification was implicit (Low)** — after ruling out the `submission_id` index, `ProcessNewLap` assumed any other `UniqueConstraintViolationException` on this transaction was the `servers` active-identity race and retried unconditionally. Correct today, but silently wrong if a future migration adds another unique constraint this transaction can hit. Fixed: added `violatesServerIdentityUniqueness()`, checking the exception's `$columns`/`$index` for the `active_since` column / `servers_ip_port_active_since_unique` index specifically; anything else now rethrows instead of being retried.

New tests: `LapSubmissionTest.php` gained cases for a content mismatch surviving a `Cache::flush()` (durable path), a genuine retry still replaying correctly after the same flush, and a reflection-based test exercising `violatesSubmissionIdUniqueness()`/`violatesServerIdentityUniqueness()` against constructed exceptions for all three index shapes (submission_id, server identity, and an unrelated/future index). A new `LapSubmissionHashTest.php` covers the hash including `race_type`/`player_name` and normalizing split order. 133 → 137 tests, 331 → 349 assertions.

#### Fifth follow-up hardening (2026-07-07, same day, commit `aeef9e4` reviewed again)

A sixth audit pass confirmed the deployed `submission_hash` column and found one remaining narrow correctness gap, plus a doc-comment drift:

1. **`LapSubmissionHash` didn't canonicalize numeric types before hashing (Low)** — Laravel's `numeric`/`integer` validation rules accept a numeric string without coercing it, so a request that validates cleanly could reach the hash helper with e.g. `player_time` as `42.5` or `"42.5"`, or a split's `checkpoint_id` as `2` or `"2"` — two semantically identical submissions that would hash differently. Fixed: `compute()` now casts `race_type`/checkpoint IDs to `int` and `player_time`/split durations/timestamps to `float` before hashing.
2. **Equal-checkpoint splits stayed payload-order-dependent (Low, related)** — nothing rejected two splits claiming the same `checkpoint_id`, so sorting by that key alone doesn't fully canonicalize such a payload. Fixed at the source: `StoreLapTimeRequest` now applies Laravel's `distinct` rule to `splits.*.checkpoint_id`, rejecting the payload outright (`422`) rather than trying to define an ordering for an already-ambiguous submission.
3. **Migration doc-comment drift** — the `add_submission_hash_to_lap_times_table` migration's docblock said laps submitted without a `submission_id` have no fingerprint; the actual code stores `submission_hash` on every new lap regardless. Corrected — only laps recorded *before this column existed* have none.

New tests: `LapSubmissionHashTest.php` gained a case proving native-numeric and numeric-string variants of the same submission hash identically; `LapSubmissionTest.php` gained a case for the duplicate-checkpoint `422`. 137 → 139 tests, 349 → 352 assertions.

Operationally, HRL enforcement remains off during the Lua rollout, so SEC-01 is not closed yet. Nginx/PHP-FPM concurrency protection also remains a deployment prerequisite outside this repo.

**Deployment note (2026-07-07):** the deployed PHP-FPM pool is confirmed configured for **30 workers**. Recorded here as the concrete concurrency ceiling this repo's rate limits sit behind — Laravel's `RateLimiter` caps request *rate* per key (IP/ip:port), not the number of PHP workers occupied at once across all sources, so this number is the actual backstop against the UDP-exhaustion risk described above. Not yet cross-checked against the current tiered rate-limit numbers for headroom.

- **Input validation** for real Eloquent-backed forms — none exist yet since everything is still mock data. The lap-submission webhook (2026-07-06) is the one exception so far — validated via `StoreLapTimeRequest`.

#### NAT internal-IP rewrite restored (2026-07-07)

`ApiController.php-legacy` hardcoded a rewrite of three specific internal addresses on the same private subnet to a known public IP in `newTime()`/`claimPlayer()` — a fixup for the old hosting site's UniFi router occasionally resolving the submitting game server's IP as one of the router's own internal addresses instead of the real public one. This wasn't carried into the rebuild and was noticed missing when the user's own loopback started showing an internal IP again.

Ported as a config-driven map (`config('webhook.internal_ip_map')`) rather than a hardcoded snippet, via a new `App\Helpers\ResolveSubmittingIp::resolve()`. Applied in two places, not one — both must resolve the same IP or they disagree about identity:

- `LapSubmissionController::store()`, before `$ip` is used for the idempotency key, verification, or storage.
- The `webhook` rate limiter in `AppServiceProvider::boot()`, which runs as route middleware *before* the controller — without the same rewrite there, the "verified" tier's cache marker (keyed by the controller's resolved IP) would never match a lookup keyed by the raw one, silently stranding every NAT'd source in the strict "unverified" tier forever.

New test: `LapSubmissionTest.php`'s NAT-rewrite case. 139 → 141 tests, 352 → 355 assertions.

## SEC-02 — HTTPS redirect + HSTS, API stays on plain HTTP (2026-07-07)

An audit finding flagged HTTP being served without a redirect to HTTPS or an HSTS header. `POST /api/v1/laps` and the read endpoints under `/api/v1/*` are called by legacy Halo game-server clients (older Wine/non-browser HTTP stacks) that can't do TLS, so a blanket site-wide redirect isn't viable — and there's no separate API subdomain to exclude instead; it's the same host, path-based.

Fixed by scoping at the middleware-group level in `bootstrap/app.php`, not by host/path matching: `App\Http\Middleware\RedirectIfNotSecure` (HTTP → HTTPS, 301, only in `production`/`staging`) is registered on the `web` group only. `routes/api.php` runs under the `api` group and is untouched — `/api/v1/*` keeps working over plain HTTP. `SESSION_SECURE_COOKIE=true` is now set in `.env`, safe since web traffic is guaranteed HTTPS once the redirect is live. **HSTS was deliberately never added** (an earlier version of this fix added it and then removed it — see the 2026-07-08 entries below): it's host-wide, not path-scoped, so it would tell any HSTS-aware browser to force HTTPS for the *entire* hostname including `/api/v1/*`, which must stay reachable over plain HTTP. Reconsider once the legacy API moves to its own hostname or its clients can do TLS.

**Resolved (2026-07-08, first pass)**: the open verification item above turned out to be a real gap, confirmed with a temporary debug route dumping `$_SERVER['HTTPS']`/`SERVER_PORT`/`X-Forwarded-Proto` (added, curled over both schemes, then removed immediately). FastPanel's nginx unconditionally sets `HTTPS=on` and `SERVER_PORT=443` for this vhost regardless of the real connection — a genuine plain-HTTP request on port 80 still reported as "secure" to PHP-FPM, so `RedirectIfNotSecure` never fired. The real reverse-proxy layer (Nginx Proxy Manager, see [deployment.md](deployment.md)) does send an accurate `X-Forwarded-Proto` header; Laravel just wasn't configured to trust it (`bootstrap/app.php` had no `trustProxies()` call). Fixed by adding `$middleware->trustProxies(at: '*', headers: ...)`, at the time trusting `X-Forwarded-Proto`/`-For`/`-Host`/`-Port` from any IP.

**Corrected (2026-07-08, second pass, real vulnerability)**: the first pass's blanket header trust was wrong — trusting `X-Forwarded-Host`/`X-Forwarded-Port` turned the redirect into a real, live, confirmed **open redirect**. A request with `X-Forwarded-Host: evil.example` and `X-Forwarded-Port: 444` produced `Location: https://evil.example:444/...`, since Laravel's redirect builds its target host/port from the (now-trusted) forwarded headers rather than the actual connection. Found and reported directly by the user re-testing against the live site. Fixed by dropping `HEADER_X_FORWARDED_HOST`/`HEADER_X_FORWARDED_PORT` from the trusted set — only `PROTO` (needed for the redirect scheme fix) and `FOR` (this app's real client IP, used by the webhook rate limiter and NAT-remap logic; confirmed safe to trust because the edge already overwrites any client-forged `X-Forwarded-For`/`-Proto` before they reach this app — the same thing that makes `PROTO` safe to trust here doesn't extend to `HOST`/`PORT`, which the edge does *not* overwrite). Also removed the `AddHstsHeader` middleware entirely (deleted the class) — it should never have been added under the stated plaintext-API compatibility policy in the first place. Verified live: the spoofed-header request now redirects to the real domain, the HTTPS response no longer sends `Strict-Transport-Security`, the normal HTTP→HTTPS redirect and `/api/v1/*` plain-HTTP access are both unaffected. Full incident detail in [decisions.md](decisions.md).

## SEC-04 — Webhook resource-exhaustion inputs (2026-07-08)

An audit finding flagged that `StoreLapTimeRequest` validated `splits` as an array with no maximum length, and that `player_time`/split `duration`/`checkpoint_id`/`startTime`/`endTime` had no upper bounds — combined with the endpoint being unauthenticated (and, at the time, `WEBHOOK_HRL_VERIFY_ENFORCE` not actually rejecting anything), a valid-shaped request could insert an unbounded number of child rows in one transaction.

**Checkpoint-count portion resolved.** Checked real data first: every map's recorded checkpoint IDs form a stable, contiguous `1..N` set across every lap ever submitted for it (confirmed by querying `lap_time_splits` grouped by map — Bloodgulch always 1-5, Death Island always 1-14, etc.), and only ~4% of real laps carry any splits at all (see [database.md](database.md)'s sparse-splits note) — so a map's checkpoint layout is a fixed, learnable fact, not something that needs an arbitrary global-only ceiling.

Two layers, agreed with the user before building (the exact ceiling number and the "fork instead of reject" design were both real judgment calls, not derivable from the code alone):

1. **A hard, protocol-wide ceiling** (`config('webhook.max_checkpoints')`, default 20, env-overridable) enforced in `StoreLapTimeRequest` — real maps top out at 14, so 20 leaves headroom while still bounding the raw resource-exhaustion risk regardless of which map a submission claims.
2. **A per-map learned baseline**, below that ceiling: a new nullable `checkpoint_count` column on `maps`, set from the first split-bearing submission for that map and enforced after that (`App\Jobs\ProcessNewLap::resolveMap()`). A submission with no splits (the common case) never establishes or checks this — it always uses the plain map.

A checkpoint-count **mismatch doesn't reject the lap or corrupt the original map's leaderboard** — maps are only ever added, never redesigned in place (confirmed with the user), so a different count for the same underlying map file means a genuinely different course sharing that file. It's forked into its own `{map_name}-splits-{count}` map identity (its own row, own baseline, own leaderboard) instead. This is a stronger separation than `race_type` gets today, worth noting: a different `race_type` only changes the display *label* on the same `Map` row (same `name`, same id) — laps from "Any Order" and normal-order races on the same map currently share one leaderboard. A checkpoint mismatch, by contrast, gets a real distinct `Map` row, since it represents an actually different route, not just a mode variant of the same one.

Added `LapSubmissionTest.php` coverage: learning the baseline, reusing it on a match, forking on a mismatch (checking both maps' rows, labels, and which lap landed where), a splitless submission never touching the baseline, and the ceiling rejecting an over-limit submission with a `422`.

**Numeric-range portion resolved, same day.** Checked real data before picking any thresholds: `player_time` ranges 15.3s-1170.9s (~19.5 min) across every real lap; split `duration` ranges 1.17s-182.2s and — checked directly — **zero real splits ever exceed their own lap's total `player_time`**. `startTime`/`endTime` turned out to be the surprising one: real data is inconsistent across Lua script versions in the wild — some rows use small, lap-relative-looking values, but a large fraction (266 of 469 real split rows) use large absolute-server-clock-like values, including a literal `999999.99` sentinel in 34 rows. Checked where these two fields are actually used: nowhere in leaderboard/comparison logic (only `duration` feeds `LapTimeSplit::compare()`) — they're stored and folded into the idempotency hash, otherwise inert.

Three changes in `StoreLapTimeRequest`/`config/webhook.php`, informed by that data rather than guessed:

1. `player_time` gets an upper bound of `config('webhook.max_lap_time_seconds')` (default 3600s/1hr, env-overridable) — 3x headroom over the real max.
2. `splits.*.duration` gets `lte:player_time` — a relational bound, not a flat number, and one already true of every real split on record, so it's a correctness constraint rather than an arbitrary cap.
3. `splits.*.startTime`/`endTime` deliberately do **not** get a tight bound — a real bound would reject real, already-accepted submissions (the absolute-clock-style values) for a field nothing depends on functionally. They get a loose `gte:0`/`lte:99999999` overflow guard only, wide enough to comfortably admit the observed `999999.99` sentinel.

Added 6 more `LapSubmissionTest.php` cases: `player_time` rejected over the ceiling and accepted exactly at it, a split `duration` longer than its lap's `player_time` rejected, the real large `startTime`/`endTime` values still accepted, and negative/beyond-overflow-guard values rejected. Body-size capping at the proxy remains outside this repo's reach, same FastPanel/nginx access limitation noted elsewhere in this doc — that's the one piece of the original finding still open.

**Review follow-up, closed same day.** A second review of the above found `splits.*.duration` had no lower bound at all (a negative duration passed validation cleanly) and that `checkpoint_id`'s `distinct` rule only proved N *distinct* values, not the map's real contiguous `1..N` layout — `[-7, 40, 999]` satisfied the exact same "3 checkpoints" check as a genuine map. It also found the checkpoint-count baseline's `if ($map->checkpoint_count === null) { $map->update(...); }` was a plain read-then-write, not concurrency-safe, and that `maps.name` had no unique constraint backing any of `resolveMap()`'s `firstOrCreate()` calls.

Fixed:

1. `splits.*.duration` gets `gt:0` alongside its existing `lte:player_time`.
2. `StoreLapTimeRequest::withValidator()` sorts all submitted `checkpoint_id`s and rejects the request unless they equal `range(1, N)` exactly — checked after casting to int (validated input can carry a numeric field as either a native type or a numeric string, same as `App\Helpers\LapSubmissionHash::compute()` already accounts for).
3. Baseline establishment in `ProcessNewLap::resolveMap()` is now a conditional `UPDATE ... WHERE checkpoint_count IS NULL` — only the first concurrent request's write can succeed; a losing request re-reads whatever count won and falls through to the same mismatch handling any later submission gets. Mirrors the same CAS pattern used for `servers`' identity race (see SEC-01's `add_unique_ip_port_index_to_servers_table` migration).
4. A real `unique` index on `maps.name` (`add_unique_index_to_maps_name` migration) — `ProcessNewLap::handle()`'s existing `UniqueConstraintViolationException` retry (SEC-01) now also recognizes this index (`violatesMapNameUniqueness()`) and retries the whole transaction, the same pattern as the `servers` identity race.
5. **Prerequisite data fix**: the real dev/prod database had one pre-existing duplicate `maps.name` (`bloodgulch`, ids 1 and 10) that would have blocked the unique constraint outright. Traced id 10 to `ProcessNewLap.php-legacy`'s race-type-suffixed label handling — confirmed dead (0 `lap_times`, only 2 stale `servers_maps` pivot rows) — and merged via a new `merge_duplicate_map_names` migration (generic over any future duplicate found, keeps the row with the most `lap_times`, refuses to auto-merge if more than one candidate actually has lap data). **Real, user-confirmed decision**: `race_type` deliberately does NOT get its own `Map` row today (see `docs/decisions.md`) — logged as a real future roadmap item (not actioned here) rather than reversed under this fix, since `race_type` was never persisted per-lap (only folded into a label string and a one-way hash), so historical laps could never be retroactively split by it even if the model changed. See [roadmap.md](roadmap.md).
6. `config('webhook.max_map_variants_per_name')` (default 3) caps how many `{map_name}-splits-{count}` forks one base map name can accumulate — `ProcessNewLap::resolveMapVariant()` throws a new `TooManyMapVariantsException` beyond the cap (`LapSubmissionController` returns `422 checkpoint_layout_mismatch`), rather than forking indefinitely. Reusing an already-forked variant doesn't count against the cap again.
7. Backfilled `checkpoint_count` for all 9 real maps (`backfill_map_checkpoint_counts` migration) from their own historical `lap_time_splits`, closing the gap where every real map's baseline was still `null` until its next split-bearing submission. Each map's historical checkpoint IDs were confirmed to already form a clean `1..N` set before backfilling (a map that didn't would be left `null` and learned fresh instead of guessed).

Added 12 more `LapSubmissionTest.php` cases covering all of the above (zero/negative duration, non-contiguous and out-of-order-but-valid checkpoint IDs, the CAS update proof, the `maps.name` DB-level constraint and its violation-classification helper, the variant cap and its reuse exemption). 156 → 168 tests, same pre-existing 21 failures/1 error unchanged.

**Reversal, same day: `race_type` now gets its own `Map` identity.** Immediately after the above shipped, the user reversed the "label-only" design this doc previously described as settled: `race_type` (0=normal, 1="Any Order", 2="Rally") now forks into its own `Map` row via a new `raceTypeMapName()` helper (name suffixes `-anyorder`/`-rally`, kept deliberately separate from the human-readable label suffixes so the two schemes — and `resolveMapVariant()`'s own `-splits-{count}` pattern — can never collide), rather than only changing the display label on one shared row. Index 0 (normal races) is a no-op suffix, so existing real map rows keep their exact current `name` — no data migration needed. Composes cleanly with the checkpoint-count fork (e.g. `bloodgulch-anyorder-splits-6`), since `resolveMap()` folds `race_type` into the base name before any of that logic runs. `MapList`/`MapLeaderboard`/`ServerMapLeaderboard` needed no changes — they already work generically per `Map` row, the same path the checkpoint-count fork already exercised; only `ProcessNewLap` changed. One accepted, pre-existing-pattern limitation: `RefreshLiveServerInfo`'s live "current map" detection can only ever resolve to the race_type-0 row, since the UDP query protocol has no `race_type` field — the same limitation the checkpoint-count fork's variants already have. 168 → 173 tests. Full detail in [decisions.md](decisions.md).

**Real Any Order rejection investigated (2026-07-09, TEST-01 audit follow-up) — check confirmed correct, no bug**: a real Any Order (`race_type` 1) lap was rejected live with checkpoint IDs `[1, 4, 5]`. First instinct — that Any Order/Rally races don't need every checkpoint, so the contiguity check was wrong — was itself wrong, caught before shipping: "any order" means the *player* can complete checkpoints out of course sequence, not that the map's physical checkpoint set changes, and the check already tolerates out-of-order submission via `sort()`. Confirmed against real data: `bloodgulch`'s established `checkpoint_count` is `5`, but the rejected payload only had 3 splits — the game server's Lua script is under-reporting splits in Any Order mode, a producer-side bug the backend correctly caught rather than silently accepted. No code change; `LapSubmissionTest.php` keeps the real anonymised payload as a permanent regression test asserting the (correct) rejection, plus a complementary test proving a genuinely complete-but-shuffled Any Order submission is accepted. Separately clarified: Rally (`race_type` 2) isn't currently usable at all due to an unrelated known Lua bug ("last lap doesn't count") — a real Rally golden fixture is out of scope until that's fixed game-side. Full detail in [decisions.md](decisions.md).

**Root cause confirmed by reading the real Lua script, same day**: `hrl.lua`'s `NormalizeCheckpointId()` decodes the game's raw checkpoint value via `log2(raw_id + 1)`, correct only for a contiguous run of low bits (the normal-race pattern) — Any Order mode sets non-contiguous bits, which this formula silently mis-decodes rather than erroring on. Almost certainly the actual mechanism that dropped checkpoints 2/3 from the real rejected payload. Not fixed here (`hrl.lua` is outside this repo, not requested this session) — flagged for the user's own follow-up. Full detail in [decisions.md](decisions.md).

**Concurrent map-variant cap: a real race condition found and fixed (2026-07-09, TEST-01 audit follow-up)**. The `resolveMapVariant()` cap check (`config('webhook.max_map_variants_per_name')`) was a plain `count()` then `firstOrCreate()`, with no locking — a real two-process MySQL concurrency test (SQLite can't provide meaningful evidence for this; initially ran directly against the real `redesign_hrl` DB with throwaway, cleaned-up data since no separate test DB existed yet, now runs against a dedicated `redesign_hrl_test` database — see testing.md) reproduced the map's variant count actually exceeding its configured cap: two concurrent requests proposing two different new checkpoint counts both read the same under-cap count, both passed the check, both inserted. Fixed the same way the checkpoint-count baseline already handles its own race (lock/serialize on whatever concurrent writers actually contend on) — `DB::transaction()` with `lockForUpdate()` on the *base* map row before counting existing variants, so a concurrent request for the same base map name blocks until the first transaction resolves. Re-ran the concurrency test 8 consecutive times post-fix with the cap holding every time. New `tests/Feature/MapVariantCapConcurrencyTest.php` + `tests/Concurrency/resolve_map_variant_worker.php`. Full detail in [decisions.md](decisions.md).

## SEC-05 — response hardening headers (2026-07-08)

An audit finding flagged that responses were missing CSP, `X-Content-Type-Options`, clickjacking protection, `Referrer-Policy`, and `Permissions-Policy`, and disclosed the exact PHP version via `X-Powered-By`.

New `App\Http\Middleware\AddSecurityHeaders`, registered on both the `web` and `api` groups (every header here is a no-op for a JSON response — CSP/frame-ancestors only affect how a *document* renders, not a raw fetched response, so there's no reason to exempt the read-only API). Adds `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, a locked-down `Permissions-Policy` (explicit `()` opt-out of every browser feature this app doesn't use — camera, microphone, geolocation, etc.), and suppresses `X-Powered-By` via `header_remove()` (`expose_php` itself is a php.ini/php-fpm-pool setting outside this repo's reach, same FastPanel-managed-config limitation as elsewhere in this doc).

**CSP shipped as `Content-Security-Policy-Report-Only`, not enforced, and tested against the real live site before any of this was trusted** — inspected the real rendered homepage first (no inline `<script>`, no `style=` attributes, no external CDN dependencies, everything same-origin) and drove headless Chrome (`google-chrome --headless=new --dump-dom`, capturing console output) against the live domain with the report-only header in place. That surfaced exactly one real violation: `'unsafe-eval'` needed by `script-src`, traced to Alpine.js's default expression evaluator (`x-data`/`x-show`/`x-on` compile via `new Function()` internally) — everything else (styles, fonts, images, the `wss://` Reverb connection) passed clean on the first try.

Presented two ways to close that gap rather than picking unilaterally (weakening `script-src` vs. changing how Alpine evaluates expressions is a real tradeoff): add `'unsafe-eval'` to `script-src` (simple, weakens what CSP protects against), or make Alpine not need it at all. **User chose the latter.** Turned out to need no dependency change at all: Livewire ships its own official CSP-safe bundle (`vendor/livewire/livewire/dist/livewire.csp.min.js`, built on a restricted, non-`eval`-based Alpine evaluator) behind a first-party `csp_safe` config toggle (`vendor/livewire/livewire/config/livewire.php`) — this app has no published `config/livewire.php` of its own, so it's overridden with a single `config(['livewire.csp_safe' => true])` call in `AppServiceProvider::register()` rather than publishing and maintaining a full copy of Livewire's config.

Re-verified rather than assumed: confirmed the live site was actually serving the CSP build (byte-for-byte match against `livewire.csp.min.js`, not the default `livewire.min.js`, despite both being served from the same URL), then re-ran the same headless-Chrome check — zero CSP violations, zero JS errors, full real content rendered — on both the homepage and a real Server Single page (the one page using the more complex `$wire.selectedPlayerIndex !== null` expression and the `@alpinejs/anchor` plugin's `x-anchor.top.offset.6="$el.parentElement"` directive, the two most complex expressions used anywhere in this app's Blade views — grepped for every `x-data`/`x-show`/`x-on`/`x-anchor` usage site-wide first to confirm there were only three distinct patterns to verify, all simple property access/comparisons).

Added `SecurityHeadersTest.php`: hardening headers present on both `web` and `api` responses, `X-Powered-By` absent, and the CSP's `connect-src` correctly scoped to this app's own domain (both `https://` and `wss://`) rather than a hardcoded string. **Still report-only, not enforced** — flip `Content-Security-Policy-Report-Only` to `Content-Security-Policy` once satisfied real traffic (not just this manual check) shows no further violations.

## SEC-06 — Reverb origin/resource policy (2026-07-08)

An audit finding flagged Reverb's `allowed_origins => ['*']`, no `max_connections`, and disabled `rate_limiting` — any site could open connections and consume broadcast capacity.

The recommendation mixed one concrete fix with two open judgment calls (a specific connection ceiling; edge-level per-IP limiting in Nginx Proxy Manager, infrastructure outside this repo) — surfaced to the user via `AskUserQuestion` rather than picked solo, per the standing ask-first rule. Result: `config/reverb.php`'s app entry now sets `allowed_origins` to `['redesign.hrl.effakt.info']` (installed Reverb v1.10.2 matches origin against the parsed hostname only, not a scheme-bearing URL — confirmed before writing this), `max_connections` to a generous round `500` as a backstop pending real traffic data (not a measured figure), and enables the already-configured `rate_limiting` (60 attempts/60s, `terminate_on_limit` true). All three are `env()`-overridable. The NPM edge-level per-IP limit stays explicitly out of scope — outside this repo and this environment's access. New `ReverbConfigTest.php` asserts the three values. Required restarting the supervisor-managed Reverb process (config changes don't hot-reload) — no `sudo`/supervisor socket access from this environment, so the user restarted it directly.

**Verified live against the restarted process**, over the real Pusher WebSocket protocol (raw sockets, not just an HTTP-upgrade check — Reverb accepts the `101` upgrade for *any* origin and only rejects at the Pusher protocol layer afterward, so a plain `curl`/upgrade-only check would have looked like a false negative): a connection from the allowed origin (`https://redesign.hrl.effakt.info`) receives `pusher:connection_established`; a connection from a disallowed origin (`https://evil.example`) receives `pusher:error` code `4009` ("Origin not allowed"); and sending 61 messages on one connection succeeds 60 times then the 61st is rejected and the connection is terminated, matching `max_attempts: 60`/`terminate_on_limit: true`. `max_connections` (500) was confirmed loaded via `config:show` post-restart but not exhausted with 500 real connections — it shares the same stock Reverb enforcement code path as the two checks above.

## SEC-07 — Least privilege (2026-07-08, partial)

An audit finding flagged two least-privilege gaps: `.env` mode `0644` (readable by every local account that can traverse the project path), and the application's DB user holding `ALL PRIVILEGES` on `redesign_hrl.*` when runtime only needs CRUD.

`.env` fixed directly — `chmod 0600`, owned solely by the app's own user/group, so no other account needs read access. Confirmed the app still boots and reads config correctly afterward.

**DB privilege reduction: explicitly deferred, not done.** The current `redesign_hrl` DB user has no `GRANT OPTION`, so it can't reduce its own grant — closing this needs a MySQL admin connection, which this environment doesn't have and shouldn't go looking for. Discussed creating a separate least-privilege runtime user (`SELECT`/`INSERT`/`UPDATE`/`DELETE` only — the app's `SESSION_DRIVER`/`QUEUE_CONNECTION`/`CACHE_STORE` are all `database`, so runtime never needs DDL; migrations would keep using the existing admin account) — user chose to skip this for now. `redesign_hrl` keeps `ALL PRIVILEGES` until revisited.

## players.hash / players identity — race condition found and fixed (2026-07-08)

While gathering evidence for PERF-02 (see [performance.md](performance.md)), a real, unmitigated bug surfaced: `players.hash` had no unique index, and `Player::firstOrCreate(['hash' => sha256(...)], ...)` (`ProcessNewLap.php`, every webhook submission) is select-then-insert, not atomic — two concurrent first-ever submissions could both pass the `SELECT` before either `INSERT` commits, creating two rows for what should be one identity. This is the identical race class already found and fixed twice in this codebase — `servers` (`ip`,`port`,`active_since` unique index + `violatesServerIdentityUniqueness()` retry) and `maps` (`name` unique index + `violatesMapNameUniqueness()` retry).

**Initial read was wrong, corrected by the user**: querying the live dev DB directly found 820 `players` rows for only 246 distinct `hash` values (up to 137 rows sharing a single hash), which first looked like either a huge pre-existing duplication problem needing a risky manual-merge decision, or a bug. **The user confirmed neither**: the game client stopped manufacturing one distinct hash per player at some point, so `hash` alone is no longer a reliable per-player identity — unrelated real players legitimately share a hash. The actual identity key is **`(hash, name)` together**, confirmed against the real data before writing any migration: zero existing rows share both columns, so a unique constraint could be added directly with no merge/dedup decision needed.

**Fixed**: `players` now has `unique(hash, name)` (migration `add_unique_index_to_players_name_hash`, replacing the plain non-unique `hash`-only index PERF-02 had just added — ordered `(hash, name)` so its leftmost prefix still serves the hot hash-only lookup path). `ProcessNewLap::recordLap()`'s `Player::firstOrCreate()` now matches on both `hash` and `name` together, and a new `violatesPlayerIdentityUniqueness()` joins `violatesServerIdentityUniqueness()`/`violatesMapNameUniqueness()` in `handle()`'s retry catch block, following the exact same pattern as `servers`/`maps`. This also confirmed the pre-existing "TAIIDOSH" case in `docs/database.md` (same name, different hash) was never in tension with this fix — different hash always means a different identity regardless of name, matching how it already behaved.

**A second real bug caught while testing this fix**: `violatesMapNameUniqueness()`'s SQLite fallback (`in_array('name', $e->columns)` — SQLite never populates `$index`, only `$columns`) started false-matching a genuine `players` `(hash, name)` violation, since that composite index also has a column literally named `name`. Tightened to an exact column-set match (`$e->columns === ['name']`) so the two constraints can't be confused regardless of which DB backend populates the exception. New tests in `LapSubmissionTest.php` cover both the DB-level constraint and the classifier's precision.

## Guidance going forward

Follow the Laravel Boost-provided `laravel-best-practices` skill's security section when writing real backend code: `$fillable`/`$guarded` on every model, authorize every action via policies/gates, no raw SQL with user input, `throttle` middleware on auth/API routes, validate file uploads by MIME/extension/size, `encrypted` cast for sensitive DB fields.

## Tooling (added 2026-07-05)

Static analysis is now in place to catch some of this automatically — see [testing.md](testing.md) for full detail:

- **PHPStan (Larastan)** (`composer analyse`) — level 7, catches type errors and some Laravel-specific misuse. Not a security scanner per se, but type safety reduces a class of bugs that become security issues (e.g. passing the wrong shape of data into a query). Level 8 is intentionally deferred because it surfaces real nullable-relation gaps that need careful code changes, not a baseline.
- **Semgrep** (`.semgrep.yml`) — local CLI is available and the custom rules are validated with positive/negative fixtures (see [testing.md](testing.md)). Custom rules cover unescaped Blade output, raw-SQL string interpolation, hardcoded-credential-like assignments, and a reminder to validate/authorize Livewire route params once they're real lookups. Hosted Semgrep also runs against the connected GitHub repository; still confirm whether the hosted policy loads the repo-local custom rules or needs them imported.
- **Rector** (`composer rector-dry`) — code-quality/modernization, not security-focused, but keeping the codebase on current Laravel/PHP idioms reduces the odds of copy-pasting a since-deprecated insecure pattern.

None of this replaces a real security review before launch — it's baseline hygiene tooling, run early specifically so gaps don't compound as real backend code gets written.
