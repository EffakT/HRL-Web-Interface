### SEC-01 тАФ HRL query verification (added 2026-07-07)

The webhook can't use TLS/HMAC on the game-server side тАФ the Lua/SAPP script that submits laps isn't part of this repo and isn't known to have that capability. Instead, `App\Helpers\LapSubmissionVerifier` (used by `LapSubmissionController::store()`) cross-checks every submission against a **live UDP `\query` response from the same ip:port the HTTP request came from**, reusing the protocol already built for roadmap item 19 (`App\Helpers\QueryServer`, `App\Models\Server`'s live-query fields). This binds the HTTP submission to a game server that is actually running an HRL-aware script, currently on the submitted map, with the submitting player online тАФ without needing any cryptography on the Lua side.

The Lua script must publish these query_add fields (SAPP's existing `query_add <key> <value>` capability) alongside the standard ones (`hostname`, `mapname`, `player_0..N`, etc.):

```text
hrl_enabled = 1
hrl_protocol = 1
hrl_token = <short-lived random value, rotated periodically>
```

...and include the same `hrl_token` value as an `hrl_token` field in the HTTP submission body. `LapSubmissionVerifier::verify()` then requires, in order: a successful UDP query (one retry on failure), `hrl_enabled === '1'`, a supported `hrl_protocol` (config `webhook.hrl_query.supported_protocol`), a matching `hrl_token`, `mapname` matching the submitted `map_name`, and the submitted `player_name` appearing among the response's `player_0`/`player_1`/... values. Any failure returns a structured reason (`udp_timeout`, `missing_hrl_marker`, `protocol_unsupported`, `token_mismatch`, `map_mismatch`, `player_not_online`) that's logged.

**Rollout note тАФ `enforce` defaults to `false`** (`config/webhook.php`, env `WEBHOOK_HRL_VERIFY_ENFORCE`): a failed verification is logged but the lap is still recorded. This is deliberate, not a bug тАФ real game servers are still running Lua scripts written before this feature existed, and none of them publish `hrl_*` fields yet. Flipping `enforce` to `true` before every active server's script is updated would silently reject every legitimate submission. Turn it on only once the rollout is confirmed complete (watch the warning logs for `missing_hrl_marker` from known-good servers dropping to zero).

**Lua/SAPP client updated (2026-07-07)** тАФ `hrl.lua` (the actual deployed script, not a reconstruction) now generates a 32-hex-char `hrl_token`, publishes it plus `hrl_enabled`/`hrl_protocol`/`hrl_token_prev` via SAPP's `query_add <key> <value>` console command (invoked from Lua via `execute_command`, since `query_add` itself has no direct Lua binding тАФ confirmed against [SAPP's own docs](https://halo-sapp.readthedocs.io/en/latest/commands/query.html), which also confirms `query_add` overwrites an existing value on repeat calls, exactly what token rotation needs), rotating every 5 minutes (`OnTick`, not tied to any game event). Every lap submission now also includes a fresh `submission_id` (SEC-01 audit follow-up's idempotency key) and the current `hrl_token`. This is still a rollout in progress, not a flip-the-switch moment тАФ `enforce` should stay `false` until every *other* active server's script is confirmed updated too (this repo only controls the webhook side; each game-server host needs its own script deployed).

**What this doesn't protect against**: a server operator controls their own query responses and could fabricate activity *on their own server* (they already could, by running a real client and legitimately racing). An on-path attacker between this app and a legitimate server could still observe or alter plaintext HTTP/UDP traffic тАФ this is verification of "a live HRL-aware game server is actually there," not encryption or non-repudiation. The audit's alternative (a local TLS-terminating relay per game-server host, HMAC/mTLS) would close that gap but requires real infrastructure work outside this repo on every game-server host; not pursued here.

A short-window idempotency guard (`Cache::add`, `config('webhook.duplicate_window_seconds')`, default 10s) runs independently of HRL verification тАФ see "Follow-up hardening" below for how this evolved from a plain duplicate-rejecting guard into a real idempotent-replay mechanism.

#### Follow-up hardening (2026-07-07, same day)

A second review of commit `0f2362a`'s implementation (recorded directly in `SITE_AUDIT.md`) found five real gaps before `enforce` could ever safely flip to `true`. All five are fixed in the same day's follow-up commit тАФ see [decisions.md](decisions.md) for the full writeup:

1. **Rate-limit bypass via port rotation** тАФ the webhook limiter's only key was `ip:port`, and `port` is caller-supplied and unverified at the rate-limit layer, so an attacker could rotate it on every request for a fresh 120/min allowance each time (while still forcing a real UDP query attempt per request). Fixed: `RateLimiter::for('webhook', ...)` now returns **two** limits that both apply тАФ the existing per-`ip:port` limit, plus a coarser per-IP ceiling (`config('webhook.rate_limit.per_ip_per_minute')`, default 600/min) that port rotation can't evade.
2. **No token rotation grace or replay protection beyond exact-value dedup** тАФ `LapSubmissionVerifier` now also accepts an optional `hrl_token_prev` query field (the Lua script's previous token, if it rotates them), so a submission racing a rotation boundary isn't wrongly rejected. Real replay protection is handled by point 3's idempotency key, not by the token itself тАФ see the class docblock for why the token isn't meant to be a durable secret.
3. **Retry poisoning тАФ a legitimate retry got an error, not its original result.** The original `Cache::add()` guard reserved the dedupe key *before* verification/processing; if a legitimate retry arrived (e.g. the client never saw its own successful HTTP response), it got a bare `409 duplicate_submission` instead of the original success payload, and a pre-commit failure left the reservation stuck for the rest of the window even though nothing was recorded. Fixed: the cache now tracks a real state machine (`processing` тЖТ `done`, storing the actual response body/status) тАФ a repeat of the same key while `done` replays the original response verbatim; a `Throwable` during processing releases (`Cache::forget`) the reservation instead of leaving it stuck. Also accepts an optional client-generated `submission_id` field as the idempotency key when present, falling back to the previous content-hash approach otherwise.
4. **Duplicate UDP query on every accepted submission** тАФ `LapSubmissionVerifier::verify()` already performs a live UDP query; `ProcessNewLap::resolveHostname()` was doing a second, independent one for the exact same ip:port. Fixed: `verify()` now returns the raw `response` array, and `LapSubmissionController` passes it into `ProcessNewLap`'s new `$liveQueryResponse` constructor parameter тАФ `resolveHostname()` reuses it instead of querying again. (When verification is disabled, or its own query failed, `$liveQueryResponse` is `null` and `ProcessNewLap` falls back to its own query, unchanged from before.)
5. **Loose player-field matching** тАФ `player_not_online` matched any query key merely *starting with* `player_`, which a future query extension (e.g. a hypothetical `player_count`) could accidentally satisfy. Fixed: matches only `/^player_\d+$/`.

**Still open, deliberately not built**: a numeric token *maximum age* check. Doing that meaningfully needs the Lua script to publish an issue timestamp the server can compare against its own clock, which is more surface on the Lua side than the current/previous-token grace already implemented тАФ revisit only if the current/previous fallback proves insufficient in practice.

#### Second follow-up hardening (2026-07-07, same day, commit `eb28327` reviewed again)

A third audit pass reviewed `eb28327` and found five more real gaps тАФ two of which involved genuine open-ended design decisions (exact rate-limit numbers, how much idempotency infrastructure to build), so those were confirmed with the user directly before implementing, rather than decided solo. Full technical detail below; reasoning in [decisions.md](decisions.md).

1. **Cross-server `submission_id` collision (High)** тАФ when a client sends its own `submission_id`, the idempotency cache key was JUST that value, with no ip/port namespace. Two different game servers generating similar counters (e.g. both starting from 1) could collide and one would receive the other's cached response. Fixed: the idempotency key is now always `lap-submission:{ip}:{port}:{submissionKey}`, whether `submissionKey` is a real `submission_id` or the content-hash fallback. Also added a `min:8` validation rule on `submission_id` as a minimum-entropy floor.
2. **Idempotency wasn't durable (High)** тАФ the cache is an optimization, not a guarantee; if it's ever lost (app restart, cache eviction, a very late retry past the retention window) nothing stopped a genuine duplicate lap from being recorded. **User's call, given two schema options**: a simple unique column on `lap_times` itself, rather than a separate claims table. Added a nullable `submission_id` string column + a unique index on `(server_id, submission_id)` (multiple NULLs allowed, so laps without one are unaffected) via a real migration. `ProcessNewLap::handle()` now catches the resulting `QueryException` on a genuine collision and replays a best-effort response built from the already-recorded lap's current state (not a fabricated success) тАФ see `replayDuplicateSubmission()`.
3. **UDP exhaustion still practical at the initial 600/min per-IP ceiling (High)** тАФ the audit calculated that forcing two 1-2s UDP timeouts per request could still occupy ~40 PHP workers from one source even under the reduced port-rotation-proof ceiling. **User's call, given two options**: build a two-tier system, not just lower the flat number. A source starts in a strict "unverified" tier (30/min per IP, 15/min per ip:port, burst 2/sec) and only earns a much more generous "verified" tier (300/min, 120/min, burst 10/sec) after a request from that exact ip:port has actually passed HRL query verification тАФ cached at `LapSubmissionVerifier::verifiedMarkerKey()` for 5 minutes. Verification still runs on every single request regardless of tier; the marker only ever changes how much traffic is allowed. The coarse per-IP ceiling applies at both tiers specifically so running many ports can't bypass it even once verified. Also reduced `hrl_query.timeout_seconds` from 2 to 1. **Not implemented**: nginx/PHP-FPM concurrency/timeout limits тАФ that's deployment/infra configuration outside this Laravel repo, flagged as a prerequisite rather than something this codebase can enforce.
4. **10-second reservation/result window was too short for one purpose and too long for another (Medium)** тАФ a single shared TTL couldn't simultaneously (a) outlast a slow in-flight request without letting concurrent duplicates through, and (b) outlast how long a real client might wait before retrying. Split into `processing_reservation_seconds` (30s тАФ generous margin over the worst-case UDP+DB round trip) and `result_retention_seconds` (300s тАФ "several minutes," per the audit).
5. **Duplicate UDP query on the failure path (Medium, a residual case of the first follow-up's point 4)** тАФ when verification's own query totally failed (`udp_timeout`), `ProcessNewLap` would still attempt its own separate query against the same already-unresponsive ip:port. Fixed by giving `$liveQueryResponse` a real tri-state: `null` (verification didn't run, try normally), `false` (verification tried and got nothing, don't retry), or the actual response array.

New tests: `LapSubmissionVerifierTest.php` gained cases for `verifiedMarkerKey()` and the `false` response state; `LapSubmissionTest.php` gained cases for cross-server non-collision, the durable DB constraint surviving a cache flush, and the tiered rate limiter's upgrade behavior. 125 тЖТ 129 tests, 308 тЖТ 321 assertions.

#### Third follow-up hardening (2026-07-07, same day, commit `a9605d1` reviewed again)

A fourth audit pass reviewed `a9605d1`, confirmed the migration/index/data-integrity state directly against the staging deployment, and found four more real gaps.

1. **Verified-tier rate-limit marker wasn't revoked on failure (High)** тАФ a source could pass verification once, then stop answering UDP entirely, and keep the generous "verified" rate-limit tier for up to the marker's full 5-minute TTL while forcing timeout work on every request. Fixed: `LapSubmissionController::process()` now calls `Cache::forget()` on the verified marker immediately whenever verification fails, for any reason тАФ the current failing request may still consume whatever allowance was in effect when it started, but every request after it immediately drops back to the strict "unverified" tier.
2. **`servers` had no unique `(ip, port)` constraint (High)** тАФ durable lap idempotency (see the second follow-up) is scoped by `server_id`, but `Server::firstOrCreate()` itself still wasn't atomic: two concurrent first-ever submissions for a brand-new ip:port could both pass the SELECT before either INSERT committed, creating two distinct `Server` rows for the same real game server. This one took three attempts to get right, all recorded in the `add_unique_ip_port_index_to_servers_table` migration's docblock: a plain `unique(ip, port)` would permanently block that ip:port from ever being reused after the server is archived (soft-deleted); `unique(ip, port, deleted_at)` directly doesn't work because SQL's `NULL != NULL` means the constraint silently never fires between active rows at all; and a generated column referencing the row's own `id` for the archived branch is rejected outright by MySQL (error 1364/3109, generated columns can't reference an `AUTO_INCREMENT` column) тАФ this app's real database is MySQL, not just the SQLite used for tests. The version that shipped: a generated `active_since` column, `COALESCE(deleted_at, '2000-01-01 00:00:00')` тАФ every active row collapses to the same sentinel so active duplicates still collide correctly, while a soft-deleted row keeps its own real `deleted_at` timestamp so archived rows (and a later legitimate reuse of that ip:port) don't collide with anything. `ProcessNewLap::handle()` now retries the whole transaction once on a `UniqueConstraintViolationException` that isn't a `submission_id` collision тАФ the concurrent request has already created the row by the time of the retry, so `firstOrCreate()`'s SELECT finds it.
3. **Duplicate-detection was too broad (Medium)** тАФ the previous version checked a generic SQLSTATE code (`23000`), which covers *any* integrity-constraint violation on some drivers, including a foreign-key failure тАФ a real bug elsewhere could have been misreported as a safe-to-replay duplicate and answered with a false `200`. Fixed: catches Laravel's specific `UniqueConstraintViolationException` (not the generic `QueryException`), and uses its `$columns`/`$index` properties to confirm WHICH unique constraint fired тАФ `submission_id`-related columns/index name means a lap duplicate; anything else is treated as the `servers` race in point 2 and retried instead.
4. **A reused `submission_id` with different lap content silently replayed the old response (Medium)** тАФ nothing distinguished a genuine retry (identical content) from a bug or a colliding ID (different content) once a `submission_id` had already succeeded once. Fixed: the cache entry now also stores a canonical content hash alongside the response; a repeat of the same `submission_id` with a *different* hash returns `409 idempotency_conflict` instead of replaying stale data. Also, per the audit's forward-looking recommendation, `submission_id` is now `required` (not just `nullable`) whenever `webhook.hrl_query.enforce` is on тАФ at that point every real submission is already expected to come from an HRL-aware script, so requiring its idempotency key too costs nothing extra.

New tests: `LapSubmissionTest.php` gained cases for the marker-revocation behavior, the `servers` unique constraint (including that a soft-deleted server's old ip:port stays reusable), the `idempotency_conflict` response, and `submission_id` being required under enforcement. The true concurrent-request race in point 2 isn't independently reproducible in a single-threaded Pest test тАФ covered by the schema-level constraint test plus code review instead. 129 тЖТ 133 tests, 321 тЖТ 331 assertions.

#### Fourth follow-up hardening (2026-07-07, same day, commit `c529742` reviewed again)

A fifth audit pass reviewed `c529742`, inspected the deployed MySQL schema directly (confirmed the generated `active_since` column, the `servers_ip_port_active_since_unique` index, no duplicate active identities, and that an archived ip:port stays reusable), and found three more real gaps тАФ all Medium/Low, none blocking rollout by themselves.

1. **Durable payload-conflict detection was cache-only (Medium)** тАФ the content hash used to catch a reused `submission_id` with changed content (third follow-up, point 4) only existed in the 5-minute idempotency cache entry. After that cache entry expired, was evicted, or the app restarted, reusing the same `(server_id, submission_id)` with different content hit the database's unique constraint instead, and `ProcessNewLap::replayDuplicateSubmission()` had no way to tell the new payload apart from a genuine retry тАФ it just replayed the old lap as a success. Fixed: added a nullable `submission_hash` column to `lap_times` (`add_submission_hash_to_lap_times_table` migration), populated on every insert. `replayDuplicateSubmission()` now compares the incoming request's fingerprint against the stored one and throws a new `App\Exceptions\LapSubmissionConflictException` on a mismatch, which `LapSubmissionController` catches and turns into the same `409 idempotency_conflict` response the cache-based check already returns тАФ so the two layers now agree even once the cache copy is gone. Rows recorded before this column existed have `submission_hash = null`, treated as "nothing to compare against" rather than a mismatch.
2. **The canonical content hash omitted material fields (Medium)** тАФ it covered player hash, map name, lap time, token, and splits, but not `race_type` or `player_name`; two submissions that genuinely differed only in those fields would have hashed identically. The splits array also hashed positionally, so two payloads with the same splits in a different order would incorrectly appear to differ. Fixed: extracted the hash into a shared `App\Helpers\LapSubmissionHash::compute()`, used by both `LapSubmissionController` (cache key/conflict check) and `ProcessNewLap` (durable `submission_hash`) so the two layers can never disagree on what counts as "the same lap." It now includes `race_type` and `player_name`, and normalizes `splits` by sorting on `checkpoint_id` before hashing so equivalent payloads submitted in a different split order still match.
3. **Server-identity-race classification was implicit (Low)** тАФ after ruling out the `submission_id` index, `ProcessNewLap` assumed any other `UniqueConstraintViolationException` on this transaction was the `servers` active-identity race and retried unconditionally. Correct today, but silently wrong if a future migration adds another unique constraint this transaction can hit. Fixed: added `violatesServerIdentityUniqueness()`, checking the exception's `$columns`/`$index` for the `active_since` column / `servers_ip_port_active_since_unique` index specifically; anything else now rethrows instead of being retried.

New tests: `LapSubmissionTest.php` gained cases for a content mismatch surviving a `Cache::flush()` (durable path), a genuine retry still replaying correctly after the same flush, and a reflection-based test exercising `violatesSubmissionIdUniqueness()`/`violatesServerIdentityUniqueness()` against constructed exceptions for all three index shapes (submission_id, server identity, and an unrelated/future index). A new `LapSubmissionHashTest.php` covers the hash including `race_type`/`player_name` and normalizing split order. 133 тЖТ 137 tests, 331 тЖТ 349 assertions.

#### Fifth follow-up hardening (2026-07-07, same day, commit `aeef9e4` reviewed again)

A sixth audit pass confirmed the deployed `submission_hash` column and found one remaining narrow correctness gap, plus a doc-comment drift:

1. **`LapSubmissionHash` didn't canonicalize numeric types before hashing (Low)** тАФ Laravel's `numeric`/`integer` validation rules accept a numeric string without coercing it, so a request that validates cleanly could reach the hash helper with e.g. `player_time` as `42.5` or `"42.5"`, or a split's `checkpoint_id` as `2` or `"2"` тАФ two semantically identical submissions that would hash differently. Fixed: `compute()` now casts `race_type`/checkpoint IDs to `int` and `player_time`/split durations/timestamps to `float` before hashing.
2. **Equal-checkpoint splits stayed payload-order-dependent (Low, related)** тАФ nothing rejected two splits claiming the same `checkpoint_id`, so sorting by that key alone doesn't fully canonicalize such a payload. Fixed at the source: `StoreLapTimeRequest` now applies Laravel's `distinct` rule to `splits.*.checkpoint_id`, rejecting the payload outright (`422`) rather than trying to define an ordering for an already-ambiguous submission.
3. **Migration doc-comment drift** тАФ the `add_submission_hash_to_lap_times_table` migration's docblock said laps submitted without a `submission_id` have no fingerprint; the actual code stores `submission_hash` on every new lap regardless. Corrected тАФ only laps recorded *before this column existed* have none.

New tests: `LapSubmissionHashTest.php` gained a case proving native-numeric and numeric-string variants of the same submission hash identically; `LapSubmissionTest.php` gained a case for the duplicate-checkpoint `422`. 137 тЖТ 139 tests, 349 тЖТ 352 assertions.

Operationally, HRL enforcement remains off during the Lua rollout, so SEC-01 is not closed yet. Nginx/PHP-FPM concurrency protection also remains a deployment prerequisite outside this repo.

**Deployment note (2026-07-07):** the deployed PHP-FPM pool is confirmed configured for **30 workers**. Recorded here as the concrete concurrency ceiling this repo's rate limits sit behind тАФ Laravel's `RateLimiter` caps request *rate* per key (IP/ip:port), not the number of PHP workers occupied at once across all sources, so this number is the actual backstop against the UDP-exhaustion risk described above. Not yet cross-checked against the current tiered rate-limit numbers for headroom.

- **Input validation** for real Eloquent-backed forms тАФ none exist yet since everything is still mock data. The lap-submission webhook (2026-07-06) is the one exception so far тАФ validated via `StoreLapTimeRequest`.

## Guidance going forward

Follow the Laravel Boost-provided `laravel-best-practices` skill's security section when writing real backend code: `$fillable`/`$guarded` on every model, authorize every action via policies/gates, no raw SQL with user input, `throttle` middleware on auth/API routes, validate file uploads by MIME/extension/size, `encrypted` cast for sensitive DB fields.

## Tooling (added 2026-07-05)

Static analysis is now in place to catch some of this automatically тАФ see [testing.md](testing.md) for full detail:

- **PHPStan (Larastan)** (`composer analyse`) тАФ level 5, catches type errors and some Laravel-specific misuse. Not a security scanner per se, but type safety reduces a class of bugs that become security issues (e.g. passing the wrong shape of data into a query).
- **Semgrep** (`.semgrep.yml`) тАФ **configured but not installed** in this environment (no pip/Homebrew/Docker available here тАФ see [testing.md](testing.md) for why). Custom rules cover unescaped Blade output, raw-SQL string interpolation, hardcoded-credential-like assignments, and a reminder to validate/authorize Livewire route params once they're real lookups. Run `semgrep --config=.semgrep.yml --config=p/php --config=p/security-audit .` wherever Semgrep can actually be installed (CI, or a local machine with pip/brew) тАФ this hasn't been run for real yet, so treat the custom rules as unvalidated until someone does.
- **Rector** (`composer rector-dry`) тАФ code-quality/modernization, not security-focused, but keeping the codebase on current Laravel/PHP idioms reduces the odds of copy-pasting a since-deprecated insecure pattern.

None of this replaces a real security review before launch тАФ it's baseline hygiene tooling, run early specifically so gaps don't compound as real backend code gets written.
