`LeaderboardModalTest.php` is the first real coverage: a homepage load with `assertNoJavascriptErrors()`, a real click-driven open/close of the podium's lap-detail modal (desktop and mobile viewports), also asserting no console errors either time. Deliberately does **not** test Escape-to-close or a focus trap — neither exists yet (the podium cards are plain `<div wire:click>`, not real buttons or links, and the modal has no `role="dialog"`/`aria-modal`/focus management at all) — that's the A11Y-01 finding, a separate open item, not something to fake passing coverage for here by testing behavior that doesn't exist.

Two real API gotchas hit while writing this, worth knowing before adding more browser tests:
- **Ambiguous text locators fail loudly, not silently** — `click('text=Player Name')` throws a strict-mode-violation error if that text appears more than once on the page (e.g. once in the podium card, once in a table row), rather than clicking the first match. Prefer a specific selector — an attribute selector on the element's own `wire:click` value (e.g. `[wire\:click="openLap(0)"]`) worked well here and doesn't depend on unique visible text.
- **Mobile emulation is `visit($url)->on()->mobile()`, not a `->onMobile()` method on the page itself** — `On` is an intermediate object returned by `->on()` that proxies through to `PendingAwaitablePage`'s methods, including `mobile()`/`desktop()`/browser-specific variants.

**Sandbox process-management gotcha, not a real bug**: running a browser test through a shell pipe (`php artisan test ... | tail`) after the outer command was `timeout`'d or interrupted can leave Playwright's own Node-based `run-server` child process alive holding the pipe's write end open — `tail` (and anything downstream) then hangs forever waiting for EOF that never comes, even though the actual test already finished and printed its result into the pipe. Looks exactly like a hung test. Fix: don't pipe browser test output through another command; run it directly (or via a backgrounded task you poll for completion) and, if something does look stuck, check for and kill stray `playwright run-server`/`chrome` processes rather than assuming the test itself is broken.

**A real gap this exact split caused (found 2026-07-08, see decisions.md — three separate bugs, all missed)**: neither of the two verification methods above actually exercises how Livewire's `#[On('echo:...')]` attribute subscribes to Echo or how it invokes the listener method. Calling the listener method directly (the Pest side) skips Echo entirely, and the manual `pusher-js` script bypassed Echo/Livewire on purpose, so none of this caught: (1) all 9 real listeners missing the leading dot Echo requires for a custom `broadcastAs()` name; (2) all 9 using `echo-public:`, which isn't a real Livewire 4 channel-type prefix at all (only `echo:`/`echo-private:`/`echo-presence:`/`echo-notification:` are); (3) three of those listeners (`ServerShow::loadServerData()`, `PlayerShow::loadProfile()`, `ServerPlayerShow::loadProfile()`) being typed to receive real Eloquent models, which would `TypeError` against the actual array payload Livewire's Echo bridge dispatches. All three shipped and passed the full suite simultaneously. Still not covered by an automated test; would need either a Pest 4 browser test driving a real page against a real Reverb instance, or a lighter unit test asserting each `#[On(...)]` attribute string against each event's real `broadcastAs()` value and each listener method's parameter list against the real dispatch shape.

**A second, different kind of gap found the same day, once the three bugs above stopped masking it**: two components were listening for the *wrong event entirely* — a mistake this test suite's structure can't catch even in principle, since "call the listener method and assert it re-fetched" only proves the method itself works, not that it's subscribed to an event that actually fires when it should. `MapLeaderboard`/`ServerMapLeaderboard` listened only for `LeaderboardUpdated` (PB-only), so their total-lap counts never updated after a non-PB attempt — the existing tests (`it('re-fetches the leaderboard when its live-update listener fires', ...)`) always constructed the *triggering* scenario as a new PB, so they never exercised the non-PB case that was actually broken. Similarly, `RefreshLiveServerInfo` never broadcast anything at all — untestable as a broadcasting gap by definition, since there was no event to assert on `Event::fake()`, only its absence. Both fixed (see decisions.md); the new `RefreshLiveServerInfoTest` case asserts `ServerStatusRefreshed` now dispatches. The general lesson: a "does the listener method work" test needs a companion "does something actually invoke this listener for every case it should" check — for now that's manual/code-review, not automated.

### PHPStan + Larastan — `composer analyse`

Config: `phpstan.neon`, level 8 (raised 5 → 6 → 7 on 2026-07-09, then 7 → 8 the same day in a deliberate second pass — see below). Fixed as a (mostly) pure type-annotation pass: Livewire component property/method array shapes (`@var`/`@param`/`@return`), `render(): View` return types, `LengthAwarePaginator<TKey, TValue>` generics, `collect()` template-resolution fixes, and `list<>`-shape corrections — no runtime behavior changed, confirmed by the full suite staying green throughout (204 tests).

**Level 8, done in batches (2026-07-09)**: first attempt surfaced 46 errors, almost all `property.nonObject`/`method.nonObject` on nullable Eloquent relations (e.g. `$lap->player->name`). Rather than null-guard 46 call sites individually, the errors split into two genuinely different categories, fixed differently:

- **~35 errors were a single false positive, not 35 real bugs**: every `LapTime::player()`/`map()`/`server()` access was flagged, but `lap_times.player_id`/`map_id`/`server_id` are NOT NULL FK columns with the default RESTRICT delete behavior (verified against the actual migrations) — a persisted `LapTime`'s related rows can never actually be missing. Larastan types every `belongsTo` as nullable regardless of DB constraints, with no way to see this invariant from the relation method alone. Fixed once, at the source: `@property-read Server $server` / `@property-read Map $map` / `@property-read Player $player` on `LapTime`'s class docblock declares the real, DB-enforced shape — resolved ~35 of the 46 errors in one change, no call-site edits needed. One side effect: this made a redundant `$lastLap->map !== null` check in `ServerList.php` provably always-true (now genuinely dead code, not silencing anything) — removed.
- **The remaining ~11 were real, individually-judged fixes**: `lap_times.created_at`/`updated_at` *are* genuinely DB-nullable (a `->nullable()->change()` migration widened them for historical imports) — `Home.php`/`MapLeaderboard.php`/`ServerMapLeaderboard.php`/`ServerShow.php`'s date formatting now use the same `?->diffForHumans() ?? '—'` / ternary pattern already established elsewhere in this codebase for that exact case (some of these specific call sites are additionally provably safe by their own query's `WHERE created_at >= ...` filter, documented inline, but the defensive form costs nothing and matches convention). `Request::ip()` is nullable in Symfony's stack (no `REMOTE_ADDR`/trusted proxy resolvable) — doesn't happen in this app's real deployment, but `LapSubmissionController`/`AppServiceProvider`'s rate limiter now fall back to `''` rather than crash, which fails verification/rate-limiting cleanly instead. `PlayerList.php`'s `avg()` — nullable on an empty collection — now uses `?? 0` directly rather than a same-emptiness-but-different-variable guard PHPStan couldn't follow. `PlayerShow.php`'s `$group->first()->server->name` (Collection::first() is nullable; the group is provably non-empty by construction, but PHPStan can't see that) rebuilt as a keyed `pluck()` lookup instead, avoiding `first()`'s nullable return entirely. `PlayerProfile.php`'s `mapWithKeys()`-built lookup (untyped value, so every read was nullable) rebuilt as a plain `foreach`-built array, which PHPStan types precisely as `array<int, int>` from `$laps`'s own known shape — no cast needed. Two spots in `Home.php` needed an explicit `$current === null` guard added to an existing condition, since the real non-null guarantee was previously expressed only indirectly (through a *different* variable, `$newRank`, derived from the same source) — PHPStan can't follow that kind of cross-variable narrowing, and there's no honest way to express it except checking the actual variable being accessed.

204/204 tests, Pint, and PHPStan level 8 all clean after. See [decisions.md](decisions.md) for the fuller narrative.

Two real gotchas worth remembering:
- A `LengthAwarePaginator` built via `->through()` needs the *exact same* declared array-shape generic on both the mapping closure's own return type and the paginator's `@return` — an equivalent-but-textually-distinct inline shape on each side fails PHPStan's invariance check even when they describe the same data. Fix: extract the row-mapping closure into a named method with its own `@return` and reference that same type on the paginator's generic (see `ServerShow::formatLatestLapRow()`/`laps()`).
- **A "just satisfy the list<> type" fix can silently break real behavior** — the level 6→7 pass wrapped every highlight candidate's `data` in `array_values()` to make `Home::computeHighlights()`'s return type check out, correct for the row-list candidates but wrong for `live-stats` (a flat associative record the Blade partial reads by string key) — `array_values()` unconditionally discards string keys. Caught before it shipped by tracing the Blade partial's actual usage, not just the PHPStan error. General lesson: when a mechanical type-fixing pass reaches for a code change (not just an annotation) to satisfy PHPStan, verify it against how the value is actually *consumed*, not just what PHPStan wants to see returned.

**Every model relationship method needs a `@return` PHPDoc generic** (e.g. `/** @return BelongsToMany<Map, $this> */`), not just the native `BelongsToMany`/`HasMany`/`BelongsTo` return type — without it, Larastan can't resolve what a relation's collection actually contains, and flags every property access on the result as "undefined property on `Illuminate\Database\Eloquent\Model`." All relations in `app/Models/` follow this convention; keep it going for any new relation.

**Raw aggregate queries (`selectRaw`/`groupBy` for things like per-map `MIN(time)`) should drop to `->toBase()`** before `->get()`/`->pluck()` rather than staying on the Eloquent builder. Those rows aren't real model instances (they're aggregates with columns the model doesn't actually have, e.g. a `best` alias), and Larastan correctly flags accessing them as "undefined property" on the model — `->toBase()` returns plain `stdClass` rows instead, which is both what's actually happening and what stops PHPStan from complaining. See `ServerShow::mount()` for the pattern. Passes clean at 0 errors as of the first real-data page (`ServerShow`).

### Semgrep — **local CLI + custom rules validated (2026-07-09); hosted GitHub scan also active**

`.semgrep.yml` at the repo root has custom rules for this codebase (unescaped Blade output, raw-SQL string interpolation, hardcoded-credential-like assignments, a reminder to validate/authorize Livewire `mount()` route params once they're real lookups). The Semgrep CLI (1.136.0, via `pipx`) turned out to already be installed in this environment — the earlier "can't install from scratch" note was about a from-scratch install (no pip module, no sudo), not this. Ran the real ruleset against the real codebase for the first time and found (then fixed) **two of the four custom rules were genuinely broken**:

- `hardcoded-credential-like-string` matched almost any string-literal assignment (`$foo = "bar";` included) — its `metavariable-regex` was scoped to a metavariable (`$PASSWORD`) that only one of three `pattern-either` branches actually bound, so the name-matching constraint silently never applied to the other two branches. Fixed by using one shared metavariable across the whole pattern.
- `blade-unescaped-output` never matched *anything*, ever — `{!! $X !!}`'s `$X` is a single-token metavariable in Semgrep's generic/spacegrep mode and can't span the multi-token run between `{!!`/`!!}`. Needed the ellipsis form (`$...X`).

### Real end-to-end Echo/Reverb browser delivery test (2026-07-09)

`tests/Browser/EchoReverbDeliveryTest.php` — a real Pest browser visits the real live homepage, a real disposable lap submission is triggered directly through `ProcessNewLap` (bypassing the public webhook's SEC-01 HRL verification) against the real `redesign_hrl` database, and the real supervisor-managed queue worker + Reverb push `LapSubmitted` out for real; the browser watches its Quick Stats "LAPS" count increment via Livewire's own `#[On('echo:...')]` reactive re-render, no page reload. This closes the one gap nothing else in the suite could: calling a listener method directly skips Echo entirely, and the earlier manual `pusher-js` script (see decisions.md) bypassed Echo/Livewire on purpose.

**Deliberately touches real production infrastructure** (explicit user decision, 2026-07-09, over two safer-but-heavier alternatives — an isolated Reverb/queue/DB stack, or not building this at all): briefly broadcasts one real, obviously-fake event to any real visitor's open browser tab, and writes to (then cleans up from) the real live database. Gated behind `ALLOW_LIVE_BROADCAST_TEST=true` — unlike every other test here, there's no "missing creds" signal to gate on by default (this DB's credentials are the app's own primary connection, always present), so an explicit opt-in is the only way to keep it out of an unaware default `composer test`/CI run.

**Four real, non-obvious gaps found building this, each worth remembering for any future test that spawns a child PHP process against real infra**:
1. **Pest's browser plugin never navigates externally, even given an absolute URL** — `LaravelHttpServer` (its default driver, confirmed by reading the source) intercepts every `visit()` and dispatches the request through the *test process's own* in-process Laravel kernel, rewriting even `https://` URLs to its own local `127.0.0.1:<port>` socket. This is fine for rendering (and the browser's own WebSocket connection is unaffected — the already-built JS bundle has the real Reverb host/key baked in from `npm run build` regardless of what serves the HTML), but it means the *test process's own* config (DB connection, cache store, etc.) is what actually serves both the initial page and Livewire's subsequent AJAX re-render — phpunit.xml's sqlite/array-cache overrides had to be reverted to the real `mysql`/`database` values for this one test only (restored in `finally`).
2. **Reverb's own origin allowlist (SEC-06) rejects Pest's local origin** — confirmed via `pusher:error` code `4009 "Origin not allowed"` on a raw WebSocket probe. `config/reverb.php`'s `allowed_origins` now also includes `'127.0.0.1'` (matches by hostname only, not port) — a permanent, deliberate widening (user's explicit choice over restarting live Reverb per test run, or abandoning the test), assessed as low real risk since exploiting it needs an attacker already controlling something on the connecting client's own localhost. Applying it needed a real Reverb restart (this environment's own user owns the process, no sudo/supervisor-socket needed — killed it directly, supervisor auto-restarted it with the new config).
3. **Broadcasts aren't replayed to late subscribers** — triggering the lap before Echo finished subscribing to the `activity` channel would silently miss the event even though Echo reports "connected" moments later. Fixed by polling `Echo.connector.pusher.channel('activity').subscribed === true` before triggering.
4. **The two deepest, easiest to miss**: the child process that triggers the lap (`tests/Browser/support/trigger_real_lap_broadcast.php`, launched via `proc_open()`) inherits the *parent test process's* `getenv()` — which already has phpunit.xml's `BROADCAST_CONNECTION=null`, `QUEUE_CONNECTION=sync`, and `CACHE_STORE=array` baked in. Overriding only `DB_CONNECTION`/`DB_DATABASE` (as the MySQL concurrency test does) silently broadcasts nothing at all — the DB writes succeed, so nothing *looks* wrong, but the worker log shows no new activity. Needed explicit `BROADCAST_CONNECTION=reverb`/`QUEUE_CONNECTION=database` overrides too. Separately, Home's highlights (PERF-01) are cached and invalidated via a generation key bumped in whatever cache store the invalidating process uses — even with broadcasting fixed, the *page-serving* process still needed its own `config(['cache.default' => 'database'])` override, or Echo/Livewire's listener would genuinely fire but keep serving a stale pre-lap snapshot from its own isolated `array` cache. Diagnosed by isolating each layer in turn: a raw `WebSocket` probe (ruled out network/origin), a direct `Echo.channel(...).listen(...)` binding bypassing Livewire entirely (ruled out message delivery once distinguished from Livewire wiring), then checking the worker log's actual timestamp against each attempt (caught the broadcast-silently-disabled bug) before finding the cache-store gap.

Full narrative in [decisions.md](decisions.md).

### Mutation testing (2026-07-09)

`pestphp/pest-plugin-mutate` (already a transitive dependency, native to Pest 4) rather than Infection — the first attempt used `infection/infection` directly, but it defaults to invoking raw PHPUnit and this project runs on Pest's own bootstrap (`vendor/bin/pest`), which errors immediately ("Please run [./vendor/bin/pest] instead"). Infection was removed; `--mutate --class=X` is Pest's own native mutation runner and needs no adapter.

**Real environment constraint hit immediately**: this box has only 6GB RAM, and running `--mutate` across a whole namespace at once (`App\Helpers`, ~6 classes, each mutant re-run spawning a fresh Laravel-booting Pest process) crashed the box outright, twice — once via a naive Infection `--filter` full-namespace run, once via Pest's own `--mutate --class="App\Helpers"`. One class at a time is stable; anything broader is not, on this box. Scope every future mutation run to a single class (or a small, related handful) rather than a namespace.

Real results, one class at a time:
- `ResolveSubmittingIp` — 1 mutation, **100%** killed.
- `LapSubmissionHash` — 24 mutations, initially **70.83%** (7 escaped). Real, genuine gap: nothing asserted that changing a split's `duration`/`startTime`/`endTime`, or omitting `hrl_token`, actually changes the canonical content fingerprint — a security-relevant function (SEC-01's idempotency/duplicate-detection) that could have silently ignored those fields with every existing test still green. Fixed with a new test case; re-ran and confirmed **95.83%** (23/24). The one remaining untested mutant (`if ($encoded === false)` → `=== true`) is the function's own documented, genuinely-unreachable invariant guard (`json_encode()` can't fail on this scalar-only input) — an accepted gap, not a real one.
- `GameServerQuery` — 0 mutations (it's an interface, no logic to mutate; correct, not a bug).
- `QueryServer`/`LapSubmissionVerifier` — not yet run; `QueryServer`'s real-socket tests (`tests/Unit/QueryServerTest.php`, forked loopback UDP server) are the likely next crash risk given this box's ceiling. Pick up one at a time, watching `free -h` between runs, per the constraint above.

### Dedicated test database (2026-07-09)

A disposable `redesign_hrl_test` MySQL database now exists for the real concurrency test (creds in the gitignored `.env.testing`, `TEST_DB_DATABASE`/`TEST_DB_USERNAME`/`TEST_DB_PASSWORD`). `MapVariantCapConcurrencyTest.php`'s `mysqlDirect()` and its `proc_open()` child env now point here instead of the real `redesign_hrl` database used when this test was first written (see the security.md/decisions.md entries on the map-variant-cap race for why this test needs a real MySQL connection at all). The test also refuses to run if `TEST_DB_DATABASE` is empty or points at `redesign_hrl`, so the default suite cannot silently mutate the live/staging DB. If the focused test returns MySQL `1044`, grant the configured `TEST_DB_USERNAME` access to `redesign_hrl_test` and run the app migrations there before treating this coverage as green. One real gotcha hit setting this up: Laravel loads `.env.testing` **instead of** `.env` when `APP_ENV=testing`, not merged with it — creating a `.env.testing` containing only the three `TEST_DB_*` lines silently dropped `APP_KEY` and everything else for every test until the file was filled out with the full set of app config values plus the `TEST_DB_*` additions.

**Code review follow-up (2026-07-09), two real gaps closed**: (1) `proc_open()` alone only guarantees both worker processes *start*, not that they actually overlap at `resolveMapVariant()`'s count-then-write critical section — app-boot latency varies per process, so one could realistically finish and commit before the other even starts, which wouldn't exercise the race at all. Fixed with a real synchronization barrier: each worker touches its own "ready" file once booted, then waits (up to 5s) for the other's ready file before proceeding — see `resolve_map_variant_worker.php`'s docblock. (2) The assertions only checked `successes <= 1` and `finalVariantCount <= cap`, which a regression that wrongly rejects *both* requests would still pass. Tightened to exactly one `OK`, exactly one `REJECTED`, and `finalVariantCount === cap` — the actual invariant a correctly-locked cap must produce under a genuine race. (3) The test now `->skip()`s with a clear message (rather than erroring) when `TEST_DB_USERNAME`/`TEST_DB_PASSWORD` aren't set — a fresh checkout/another machine won't have the gitignored `.env.testing` creds, and `composer test` runs the full default suite including this one. **This is conditional coverage, not always-on** — `composer test` can be fully green on a machine without `.env.testing` set up without ever having exercised the race regression this test guards against. Acceptable and intentional (the alternative was either erroring on every fresh checkout or risking mutating a real DB), but worth remembering when treating a green `composer test` run as proof this specific regression hasn't reappeared — check this test actually *ran* (not skipped), not just that the suite passed.

**Second code review follow-up, same day**: the 5s barrier timeout (above) originally proceeded to call `resolveMapVariant()` anyway if the other worker never showed up — meaning a slow boot or a genuinely stuck other worker could let the test pass without ever proving the two workers actually raced, silently reintroducing the exact non-overlap gap the barrier exists to close. Fixed: a timed-out worker now writes `ERROR:barrier_timeout` and exits without calling `resolveMapVariant()` at all — caught by the test's existing `->not->toStartWith('ERROR:')` assertion, which now fails loudly instead of passing on unproven "coverage."

**Third code review follow-up, same day — the deepest of the three, and the one that actually proves the lock matters**: even with both workers guaranteed to *start* `resolveMapVariant()` together, MySQL/OS scheduling could still let one worker's entire count-then-insert finish before the other even reads its count — producing the exact same passing outcome (one OK, one REJECTED, count === cap) whether or not `lockForUpdate()` does anything at all. `ProcessNewLap::resolveMapVariant()`'s count-read step is now its own overridable `protected countExistingMapVariants()` method purely so this test can inject a real delay there via an anonymous subclass in the worker script (300ms sleep between reading the count and returning it) — without a genuine row lock serializing the two transactions, both workers would read the same under-cap count during that window and both would insert. **Verified this actually discriminates, not just added ceremony**: temporarily removed `lockForUpdate()` and reran — the test failed exactly as expected (`Failed asserting that 2 is identical to 1` — both workers succeeded), then restored the lock and confirmed green again. This is the difference between "a test that happens to pass" and "a test that would fail if the fix regressed."

**Fourth follow-up, same day — made that manual verification permanent**: a one-off manual edit-and-revert proves the *current* code is right, but doesn't protect against a *future* accidental removal of the lock being missed if the test harness itself ever quietly lost its discriminating power (e.g. the 300ms delay got tuned down, or a Laravel/MySQL upgrade changed transaction timing). `acquireMapVariantLock()` is now its own overridable method too (same pattern as `countExistingMapVariants()`), and the worker script takes a `<lockMode: locked|lockless>` argument selecting between two subclasses — `lockless` no-ops the lock entirely. A new permanent test, `'proves the harness itself would catch a missing lock (negative control...)'`, runs the exact same race with `lockless` and asserts the race *does* manifest — deterministically `2` successes and `cap + 1` variants, given the fixed delay window, not a flaky "might race." If this negative control ever started passing with `1` success instead, that would mean the harness lost its ability to detect the real regression, independent of whether `ProcessNewLap`'s actual lock is intact — a check on the test's own validity, not just the production code. 205 → 206 tests.

## Conventions

Follow the Boost-provided `pest-testing` skill and `laravel-best-practices`:
- `LazilyRefreshDatabase` over `RefreshDatabase` for speed.
- Factories for all model creation in tests — build factories/states as models are created, not after.
- `assertModelExists()` / `assertSuccessful()` over raw `assertDatabaseHas()` / `assertStatus(200)`.
- Use fakes (`Event::fake()`, etc.) **after** factory setup, not before.
- Dataset pattern (see `RoutesTest.php`) for repetitive per-route/per-input tests instead of one test method per case.
